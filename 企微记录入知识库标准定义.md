# 企微记录入知识库标准定义

更新时间：2026-05-18

## 1. 总目标

企微历史记录入知识库的目标不是保存聊天记录，而是抽取可检索、可复用、可审核、可追溯、不会误导销售回复的知识单元。

正式处理顺序固定为：

```text
原始消息 -> 第 2 步确认切片号 slice_id -> 第 3 步按 slice_id 预处理字段 -> 人工复核 -> 发布到正式知识库
```

第 3 步不得新增、删除、拆分、合并或改写 `slice_id`。若发现切片边界错误，必须先回到第 2 步修正切片号，再重新预处理。

## 2. 当前切片定义

切片是企微历史记录进入知识库预处理的最小单位，对应 `wecom_raw_import.slice_id`。

### 2.1 红/绿对话切片

红/绿底色表示同一可视轮次内同时包含客户和销售消息，属于标准对话切片。

规则：

1. 同一红/绿切片内多行共享同一个 `slice_id`。
2. 客户方向多行合并为 `slice_question`。
3. 销售方向多行合并为 `slice_answer`。
4. 同一 `slice_id` 内标题、问、答、正文、质量分、处理结论、商业阶段、商业场景、命中依据和使用边界必须一致。

### 2.2 黄色客户单行切片

黄色底色表示只有客户消息，不是对话。

规则：

1. 黄色一行就是一个切片/跳过单元。
2. 连续黄色行不得合并成一个切片。
3. 黄色没有两种颜色交替区分。
4. 常见结论为 `不入库_无答`、`不入库_简单确认`、`不入库_无上下文价值`。

### 2.3 紫色销售单行切片

紫色底色表示只有销售消息，不是对话。

规则：

1. 紫色一行就是一个切片/候选单元。
2. 连续紫色行不得合并成一个“紫色块”整体处理。
3. 紫色没有两种颜色交替区分。
4. 紫色不是自动不入库。每一行可按复用价值判断为 `入库_切片`、`待人工确认` 或不入库。
5. 紫色销售主动触达切片的 `slice_question` 写触发场景/适用条件，`slice_answer` 写该行销售原话。

## 3. 字段标准

| 字段 | 标准 |
|---|---|
| `slice_id` | 切片唯一编号；红/绿可多行共享；黄/紫一行一个 |
| `slice_title` | 知识标题，表达触发场景、销售动作或可复用价值，不用“切片/模板”等泛化词凑标题 |
| `slice_question` | 客户问/触发场景；红/绿取客户原文合并，紫色写适用条件，黄色取客户单行原文 |
| `slice_answer` | 销售答/参考回复；红/绿取销售原文合并，紫色取销售单行原文，黄色通常为空 |
| `normalized_content` | 拟入库正文预览；只做必要脱敏和最低限度标准化，不改写原意 |
| `slice_quality_score` | 0-100，评价该切片作为知识库证据的价值 |
| `slice_process_result` | 切片级处理结论 |
| `business_stage` | 来自 `llm_business_scenario_standards.category` |
| `business_scenario_code` | 来自 `llm_business_scenario_standards.scene_value` |
| `business_scenario_name` | 来自 `llm_business_scenario_standards.scenario` |
| `scenario_match_reason` | 命中依据，必须写具体语义信号 |
| `usage_boundary` | 使用边界，说明何时可用、何时不可直接套用 |
| `skip_reason` | 不入库原因；入库或待确认时为空 |
| `publish_target` | 入库或待确认时为正式知识库候选 |

原始 `content` 列严禁修改。

## 4. 处理结论

| 结论 | 含义 |
|---|---|
| `入库_切片` | 可作为正式知识候选，仍需人工复核发布 |
| `待人工确认` | 有复用可能，但素材、边界、个案性或风险需人工确认 |
| `不入库_无答` | 只有客户内容，缺销售有效承接 |
| `不入库_简单确认` | 好的、收到、OK、谢谢等无新信息 |
| `不入库_无上下文价值` | 检索出来不能指导销售回复 |
| `不入库_强个案` | 只适用于具体联系人、电话、价格、供应商路径等，容易误导 |
| `不入库_乱码` | 未知消息类型、乱码或无法识别内容 |

`不入库_无问` 不应用来排除紫色销售消息。紫色应按“一行一个销售主动触达候选”判断。

## 5. 质量分

| 分数 | 标准 |
|---|---|
| 90-100 | 高度通用，可直接作为强规则或标准话术 |
| 80-89 | 高价值，问答清晰，复用场景明确 |
| 70-79 | 有入库价值，但需限定适用边界 |
| 60-69 | 弱参考，适合候选池或人工确认 |
| 40-59 | 个案化或上下文依赖强，默认不直接发布 |
| 0-39 | 无信息量、简单确认、乱码、无问答价值 |

质量分按切片评估，同一 `slice_id` 内多行必须一致。

## 6. 商业场景标准

商业阶段和商业场景只能从 `llm_business_scenario_standards` 启用项中选择。

当前标准包括：

| 编码 | 商业阶段 | 商业场景 |
|---|---|---|
| `lead_activation` | 基础跟进 | 新客激活 |
| `greetings` | 基础跟进 | 节气/开工问候 |
| `handover` | 基础跟进 | 人员离职/交接 |
| `re_activation` | 基础跟进 | 老客唤醒 |
| `lost_order_review` | 基础跟进 | 丢单复盘 |
| `case_sharing` | 价值呈现 | 案例分享 |
| `service_intro` | 价值呈现 | 业务深度介绍 |
| `meeting` | 价值呈现 | 线下约见/面谈 |
| `spec_alignment` | 价值呈现 | 产品/服务细节对标 |
| `budget_cut` | 异议处理 | 预算收紧 |
| `existing_supplier` | 异议处理 | 已有稳定供应商 |
| `pricing_bidding` | 异议处理 | 价格竞争/比价 |
| `internalization` | 异议处理 | 内部消化/自己做 |
| `mapping` | 深度摸排 | 决策链/关系网摸排 |
| `channel_investigation` | 深度摸排 | 渠道/合作伙伴调查 |
| `internal_support` | 深度摸排 | 多方协同与内部游说支持 |
| `relationship_maintenance` | 客情经营 | 社交应酬/客情深度维护 |
| `empathy` | 客情经营 | 行业共鸣/吐苦水 |
| `emergency_response` | 客情经营 | 紧急响应/确定性交付 |
| `referral` | 客情经营 | 转介绍请求与存量裂变 |
| `quotation_confirmation` | 商务转化 | 方案确认与正式报价 |
| `complaints` | 售后合规 | 售后纠偏/客诉处理 |

商业场景不能反向决定切片边界。同一个商业场景可以对应多个独立切片。

## 7. 标准化规则

1. 原始 `content` 不修改。
2. `slice_question` 和 `slice_answer` 优先保留原文，只做必要合并和脱敏。
3. 邮箱、手机号、直线电话、微信号、个人姓名、具体联系人等应脱敏。
4. 我方公司名、通用业务名称、非敏感服务能力可保留。
5. 未知消息类型写 `不入库_乱码`。
6. 图片、视频、文件若只有占位符且无摘要，通常 `待人工确认` 或 `不入库_乱码`；若文本中已有视频号/案例摘要，可作为案例分享候选。
7. 不能把销售原话润色成新的标准话术后写入 `slice_answer`。

## 8. 验证要求

每批处理后必须检查：

1. 无 `slice_id` 的行不能进入 `入库_切片` 或 `待人工确认`。
2. 黄/紫单向消息必须一行一个 `slice_id`。
3. 红/绿对话切片内字段必须一致。
4. 同一 `slice_id` 内 `slice_title`、`slice_question`、`slice_answer`、`normalized_content`、`slice_process_result`、`business_scenario_code` 不得出现不一致。
5. 最后一行如果属于红/绿多行切片，必须延伸处理到该切片结束；如果最后一行是黄/紫单行切片，则该行本身就是完整切片。
6. 分页展示不能重新按当前页首行推断底色；已锁定切片后，页面底色必须优先使用数据库 `slice_id`、`slice_row_start`、`slice_row_end`。跨页切片在上一页末尾和下一页开头必须保持同一红/绿对话底色。
7. 跨页示例：如果第 `1900-1902` 行共享同一个 `slice_id`，且包含客户与销售，则三行是同一个红/绿对话切片；不能因为 `1900` 在上一页末尾、`1901-1902` 在下一页开头，就把 `1900` 显示或处理成黄色客户单向切片。

## 9. 价值复核要求

字段齐全不等于符合入库要求。每个切片还必须判断字段是否有价值、是否正确、是否会误导检索。

复核规则：

1. `入库_切片` 只能给能直接指导类似场景销售回复、且检索出来不容易误导的内容。
2. 只有简短问候、纯关注视频号、单句“多关照”、单句“看一下”、单句“提提建议”等弱触达，不直接入库；通常为 `不入库_无上下文价值` 或 `待人工确认`。
3. 案例视频、视频号、图片、附件类内容如果只有引导语、缺少可独立检索的素材摘要，不能直接发布；一般为 `待人工确认`。
4. 客户行业、平台、联系人、部门、供应商、邮箱、电话等信息，如果没有可复用销售回复，不能因为字段可填就入库；可进入 CRM/画像或作为候选。
5. 客户已有供应商、预算、内部消化、邮件未收到、方案/报价确认、物料规格确认等场景，只有当销售回应具备可复用处理动作时，才可 `入库_切片`。
6. `待人工确认` 不是失败状态，而是保守状态：用于素材缺失、边界不清、强个案或复用价值需要人工判断的切片。
7. 每批处理完成后应抽样核对页面展示字段与数据库值，防止旧后端、浏览器缓存或不同数据库连接导致页面仍显示旧结果。

## 10. 每批完整校检字段

后续每 `100` 行左右为一批推进。每批必须逐切片检查右侧“入知识库详细信息”的完整字段，不得只检查字段是否非空。

每个切片至少校检以下字段：

| 字段/展示项 | 校检要求 |
|---|---|
| 知识标题 | 必须具体描述触发场景和销售动作，不能是泛化标题 |
| 知识分类 | 应与内容类型匹配；不能把弱触达或客户画像硬归为正式知识 |
| 服务 | 必须来自明确信息；不能由行业、国家、客户背景臆测 |
| 商业场景 | 必须来自 `llm_business_scenario_standards`，且与主要触发语义一致 |
| 知识切片 | 必须能作为完整可审阅单元；红/绿含问答，黄/紫为单行触发/原文 |
| 参考问/触发场景 | 红/绿取客户原文；紫色写适用条件；不得伪造客户问题 |
| 参考回复 | 必须保留销售原话或最低限度脱敏后的原话，不得润色成新话术 |
| 使用边界 | 必须说明适用条件和不可直接套用的风险 |
| 场景编码 | 必须与商业场景一致 |
| 命中依据 | 必须写具体语义信号，不能只复述分类名 |
| 质量分 | 必须体现检索复用价值；弱触达、纯寒暄、素材缺失不能高分 |
| 处理结论 | 必须区分 `入库_切片`、`待人工确认` 和各类不入库原因 |

校检完成标准：

1. 同一 `slice_id` 内上述字段一致。
2. `入库_切片` 和 `待人工确认` 必须具备标题、触发场景/参考问、参考回复或单行销售原文、商业阶段、商业场景、场景编码、命中依据、使用边界、质量分。
3. 不入库切片必须有明确 `skip_reason` 和低质量分。
4. 页面最后一行不能截断红/绿多行切片；若最后一行是黄/紫单行，则该行即完整切片。
5. 用户指定批次终点如果落在红/绿多行切片中，实际处理范围必须顺延到该 `slice_id` 的最后一行，并在历史记录中说明实际结束行；不得为了凑整数行数截断切片。
6. 黄/紫单行切片不仅 `slice_id` 不能合并，相应的 `slice_row_start`、`slice_row_end`、`normalized_content`、`slice_question`、`slice_answer` 也必须只来自本行；不得把相邻多个黄/紫单行消息写成同一个字段明细。

重处理要求：

1. 重新校检时不得沿用旧 `slice_title`、旧 `business_scenario_code`、旧 `scenario_match_reason`、旧 `usage_boundary` 作为判断依据。
2. 判断输入只能是原始 `content`、`role`、消息时间、对话双方、当前已锁定的 `slice_id` 边界，以及本标准定义。
3. 旧字段只能用于差异审计和回溯，不能用于生成新结论。
4. 引用消息中的历史内容不能压过当前切片主语义；例如引用上一轮报价但当前问的是开票，则当前切片归财务开票确认。

人工精读要求：

1. 用户要求人工精读时，禁止用关键词规则、批量分类脚本或模式匹配替代逐切片语义理解。
2. 每个切片必须先阅读原始 `content` 与角色顺序，再判断当前切片的主语义、复用价值和风险边界。
3. 如发现正在使用规则代码批量生成结论，必须立即停止，回到逐切片阅读和人工判断。
4. 允许使用数据库查询读取原文、使用定点 SQL 回写已人工确认的切片；但回写清单必须来自逐切片人工判断，而不是自动分类结果。
5. 每个批次必须在历史文档中说明处理方式是“人工精读”还是“规则化重判”；两者不能混用、不能混称。

## 11. 直接入库阈值

`入库_切片` 必须收紧使用。能填字段、能归类、能看出场景，不等于可以直接入库。

直接入库只适用于满足以下条件的切片：

1. 内容可以独立指导类似场景销售回复。
2. 不依赖具体项目、具体联系人、具体金额、具体账号、具体物流、具体供应商路径或附件素材。
3. 参考回复是可复用销售动作或话术，而不是单次执行记录。
4. 使用边界清楚，检索出来不会让销售误套价格、参数、承诺或客户隐私。

以下内容即使字段完整，也不得直接 `入库_切片`，应为 `待人工确认` 或不入库：

| 内容类型 | 默认处理 |
|---|---|
| 具体金额、报价、发票、运费、付款、收款账号、税号、银行账号 | `待人工确认`，不可直接复用数值 |
| 具体型号、尺寸、颜色、Logo、激光/水晶贴、字幕位置、文件参数 | `待人工确认`，不可直接复用参数 |
| 物流报备、表单纠错、引用消息、单次执行进度 | `待人工确认` 或 `不入库_无上下文价值` |
| 视频号、附件、图片、案例素材只有引导语或请对方查看 | `待人工确认` 或 `不入库_无上下文价值` |
| 只包含销售签名、联系方式、名片、账号信息 | `待人工确认` 或 `不入库_强个案` |
| 客户画像、岗位、平台、供应商、邮箱电话等信息，无可复用销售回复 | 不直接入库 |

`service_intro` 的直接入库条件更严格：只有当销售原文包含清晰、可独立复用的服务能力介绍或业务承接话术时，才可直接入库；如果只是项目执行、附件提示、视频提示、联系方式、报价付款、物流或表单细节，一律不直接入库。

`待人工确认` 是质量控制状态，不是失败状态。凡是有潜在价值但存在素材缺失、强个案、上下文依赖或发布风险的切片，应优先进入 `待人工确认`。

## 12. 财务开票付款语义

含税、开票、发票、收据、财务章、税点、普票、专票、银行账号、开户行、纳税人识别号、付款、支付、未付、到账、对公、抵扣、作废重开、抬头等，属于财务/开票/付款条件确认。

处理规则：

1. 默认商业阶段为 `商务转化`，商业场景为 `quotation_confirmation / 方案确认与正式报价`。
2. 不得因为原文中引用了上一轮“预算、最低价、报价”内容，就把当前切片归为 `budget_cut / 预算收紧`；当前切片的主语义是含税、开票或付款时，应按财务确认处理。
3. 不得归为 `service_intro / 业务深度介绍`；财务开票沟通不是服务能力介绍。
4. 默认 `待人工确认`，不直接发布，因为具体金额、税率、账号、抬头、开票方式、收据方式和付款流程都强依赖当前订单和财务要求。
5. `slice_title` 应明确写财务动作，例如“财务开票付款条件确认”“开票税点与含税价格确认”“发票抬头与付款账号确认”。
6. `scenario_match_reason` 必须写明主语义是含税/开票/发票/收据/付款/账号确认，而不是复述上一轮报价或预算。
7. `usage_boundary` 必须提示金额、税率、账号、抬头、发票处理方式不可直接套用。

示例：

| 原文 | 正确处理 |
