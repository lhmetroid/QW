# 原始数据准备与人工操作说明

## 一、文档目的

本文档用于统一说明当前知识库方案及后续“业务事实库 + 生成参考库”增强方案下，项目涉及的原始数据、字段定义、人工操作步骤，以及代码与大模型处理数据的逻辑。

本文档有两个使用目标：

1. 指导业务、运营、销售、知识库管理员准备当前版本确实需要人工整理的可入库数据。
2. 指导项目实施人员理解当前系统已经自动读取、自动生成、自动沉淀了哪些业务上下文数据。
3. 指导后续增强方案需要补哪些规范字段，但不把它们误当成“当前都要手工补进知识库”的必备项。

说明：

- 本文档不改代码，只说明现状与建议。
- 本文档中的“当前已实现”指当前仓库已有能力。
- 本文档中的“建议补齐口径”指为了后续增强方案落地，建议统一的数据口径，不等于当前都要人工补录，更不等于默认进入当前知识库。

## 二、总体数据原则

### 2.1 总体目标

系统的目标不是“尽可能收很多资料”，而是：

- 让系统能找到**最合适、最安全、最可追溯**的知识证据；
- 让系统在证据不足时**宁可保守，不可乱答**；
- 让后续自动回复建立在**业务阶段正确、证据有效、状态可用、范围匹配**的前提下。

### 2.2 原始数据必须满足的原则

1. **真实**
   - 必须来自真实业务资料、真实会话、真实邮件、真实报价、真实流程文件。
   - 禁止把模拟数据、测试数据、猜测信息混入正式知识源。

2. **原子化**
   - 一条知识只表达一个知识点。
   - 一条事实只表达一个业务事实。
   - 不允许把报价、最低收费、加急规则、流程步骤、案例说明混在同一条数据里。

3. **可追溯**
   - 每条知识或事实都必须能追溯到原始来源。
   - 必须保留来源类型、来源文件、来源位置、时间、操作者等信息。

4. **可判定**
   - 报价、能力、流程、限制条件等高风险内容，必须能明确判断是否可用于回复。
   - 不能只保留模糊文本，而没有结构字段。

5. **可治理**
   - 必须支持草稿、审核、发布、归档、版本、生效期、命中日志、人工反馈。

6. **空值明确**
   - 不确定就留空，不允许编造。
   - 未知值用空值，不用“默认写死”来掩盖事实缺失。

## 三、原始数据种类总览

结合当前方案与增强方案，建议把原始数据分成两大类。

### 3.1 第一类：生成参考类数据

用于回答“怎么说、依据是什么、规则是什么、有哪些案例或模板可以参考”。

当前系统已支持的主要类型：

- 常见问答
- 能力知识
- 流程知识
- 报价知识
- 历史案例
- 销售话术
- 邮件整理知识
- 结构化报价规则

### 3.2 第二类：业务事实类数据

用于回答“当前这个客户线程处于什么业务阶段、已经发生了什么、还缺什么信息、能不能进入下一步回复动作”。

这部分不是“当前知识库正式导入资料”的同义词。

当前系统已经通过会话记录、LLM1 摘要、CRM 画像、运行状态返回等方式持有一部分业务事实或业务上下文；只是这些数据目前分散在不同表和接口返回中，还没有被统一整理成完整独立的“事实层”。

建议提前准备的事实类型：

- 线程主信息
- 客户画像事实
- 产品意图事实
- 报价事实
- 订单事实
- 回款事实
- 物流事实
- 售后问题事实
- 邮件线程事实
- 附件解析事实

## 四、当前必须准备的原始数据

本节是当前系统已经能直接消化、且应优先准备的数据。

## 4.1 正式知识导入数据

### 4.1.1 适用范围

适用于以下正式知识导入入口：

- 导入中心 -> 常见问答
- 导入中心 -> 报价规则
- 导入中心 -> 流程规范
- 导入中心 -> 能力清单
- 导入中心 -> 历史案例
- 导入中心 -> 销售话术
- 导入中心 -> 邮件整理

这些数据会直接进入正式知识库草稿，不先进候选池。

### 4.1.2 当前统一标准字段

当前正式知识 Excel 建议按统一 20 列准备。

| 中文字段名 | 系统字段名 | 是否必填 | 数据类型 | 定义 | 填写要求 |
|---|---|---:|---|---|---|
| 标题 | `title` | 是 | 文本 | 知识标题、问题、规则名、场景名 | 必须简洁明确，不宜过长 |
| 内容 | `content` | 是 | 长文本 | 知识正文、答案、规则、步骤、案例正文 | 必须保留原始业务含义 |
| 知识类型 | `knowledge_type` | 否 | 枚举 | 业务含义分类 | 可为空，系统按导入类型默认补 |
| 切片类型 | `chunk_type` | 否 | 枚举 | 证据形态分类 | 可为空，系统按导入类型默认补 |
| 服务 | `business_line` | 否 | 枚举 | 所属业务线 | 建议填，无法判断可空 |
| 语种 | `language_pair` | 否 | 枚举 | 语种方向 | 翻译类建议填写 |
| 服务范围 | `service_scope` | 否 | 枚举 | 适用资料或业务范围 | 如医学资料、法律资料、普通资料 |
| 客户层级 | `customer_tier` | 否 | 枚举 | 适用客户层级 | 仅当知识有限制时填写 |
| 优先级 | `priority` | 否 | 数字 | 检索排序优先级 | 默认 50 |
| 风险等级 | `risk_level` | 否 | 枚举 | 风险高低 | 报价、合同、税费建议高风险 |
| 生效日期 | `effective_from` | 报价类建议必填 | 日期 | 知识或规则生效起点 | 报价类强烈建议填写 |
| 失效日期 | `effective_to` | 报价类建议必填 | 日期 | 知识或规则失效时间 | 报价类强烈建议填写 |
| 单位 | `unit` | 报价类建议必填 | 枚举 | 报价计价单位 | 仅结构化报价规则使用 |
| 币种 | `currency` | 报价类建议必填 | 文本 | 报价币种 | 默认一般为 `CNY` |
| 价格 | `price_min` | 报价类建议必填 | 数字 | 最低价或单价 | 需与正文一致 |
| 最高价格 | `price_max` | 否 | 数字 | 区间报价上限 | 有区间时填写 |
| 最低收费 | `min_charge` | 否 | 数字 | 起步价、最低收费 | 建议明确填写 |
| 加急倍率 | `urgent_multiplier` | 否 | 数字 | 加急系数 | 如 `1.5` |
| 税费 | `tax_policy` | 否 | 文本 | 含税、不含税、税点说明 | 尽量标准化 |
| 标签 | `tags` | 否 | 文本 | 辅助识别标签 | 可写来源、主题、备注 |

### 4.1.3 当前系统认可的知识类型

| 中文名称 | 系统值 | 定义 |
|---|---|---|
| 能力知识 | `capability` | 公司能不能做、支持哪些语种、支持哪些服务范围 |
| 报价知识 | `pricing` | 价格、单位、最低收费、税费、折扣、加急费 |
| 流程知识 | `process` | 下单、付款、审核、交付、售后、开票等步骤 |
| 常见问答 | `faq` | 普通解释、问答、话术、案例摘要、定义 |

### 4.1.4 当前系统认可的切片类型

| 中文名称 | 系统值 | 定义 |
|---|---|---|
| 规则 | `rule` | 可执行、可判断的正式规则 |
| 问答 | `faq` | 一问一答式标准知识 |
| 案例 | `example` | 历史项目经验、案例材料 |
| 话术模板 | `template` | 销售可复用表达模板 |
| 限制条件 | `constraint` | 不能做、需确认、需补充资料等边界条件 |
| 定义 | `definition` | 术语、字段、概念解释 |

### 4.1.5 当前系统实际使用的知识分类口径

系统在辅助拆分和检索时，实际上还使用更细的知识分类口径：

| 中文知识分类 | 系统归一化结果 |
|---|---|
| 报价限制条件 | `knowledge_type=pricing` + `chunk_type=constraint` |
| 能力知识 | `knowledge_type=capability` + `chunk_type=rule` |
| 流程规则 | `knowledge_type=process` + `chunk_type=rule` |
| FAQ常见问答 | `knowledge_type=faq` + `chunk_type=faq` |
| 案例 | `knowledge_type=faq` + `chunk_type=example` |
| 邮件模板 | `knowledge_type=faq` + `chunk_type=template` |
| 名词定义 | `knowledge_type=faq` + `chunk_type=definition` |

### 4.1.6 正式知识原始数据的最低质量要求

1. 标题必须能表达主题。
2. 内容必须保留原文业务含义。
3. 一行只表达一个知识点。
4. 涉及价格时，必须同时准备结构化报价字段。
5. 涉及有效期的知识，必须准备生效日期、失效日期。
6. 涉及客户分层、服务范围、语种方向时，建议准备结构字段，不建议只藏在正文里。

## 4.2 结构化报价规则原始数据

### 4.2.1 为什么必须单独准备

当前系统明确要求：

- 报价模板不等于报价规则。
- 报价正文不等于结构化报价规则。
- 价格、单位、最低收费、税费、加急倍率，必须尽量进入 `pricing_rule`。

1. 价格数字必须能追溯到原文。
2. 没有明确数字时，不要编造。
3. 同一适用范围不要出现冲突价格。
4. 没有结构化报价规则的报价知识，不应直接发布。

4. 候选不是正式知识，不能直接参与正式检索。

## 五、当前已存在的业务上下文数据与建议补齐口径

本节不是“当前都要人工准备并导入知识库”的清单。

更准确地说，本节分成两层：

- 第一层：当前代码里已经存在、已自动读取或已自动生成的业务上下文数据。
- 第二层：为了后续增强方案更稳定，建议补齐或显式化的统一字段口径。

默认原则：

- 这部分数据很多并不进入当前知识库正式切片。
- 其中一部分来自 CRM 读取、一部分来自会话摘要、一部分来自运行链路状态。
- 只有明确适合作为知识证据的内容，才应进入当前知识库或报价规则。

## 5.2 建议补齐的统一字段口径

本节保留的是“为了后续更显式、更可治理，建议补齐的规范字段”。

这些字段名很多是我按统一口径整理出来的，并不代表当前代码里已经完整建表或完整落地；它们的作用是：

- 把当前分散在会话、摘要、CRM、附件里的信息整理成更稳定的事实层；
- 让后续阶段判断、状态一致性校验、效果分析更容易做；
- 默认不进入当前知识库正式切片，除非其中某部分被审核后转化为正式知识或正式规则。

## 六、原始数据定义必须统一的关键口径

## 6.1 业务阶段口径

后续增强方案必须统一业务阶段，否则线程事实层会失真。

建议统一为：

- 初始接触
- 需求确认
- 资料收集中
- 报价准备中
- 正式报价中
- 谈判中
- 下单中
- 付款中
- 交付中
- 发货中
- 售后中
- 已完成
- 已暂停
- 已流失

## 6.2 “可回复”口径

建议后续统一：

- **可回复证据**：可以直接进入自动回复上下文
- **仅人工参考证据**：只能给人看，不能直接给模型
- **待审核证据**：需修正或补字段后才能进入可回复

## 6.3 报价口径

必须统一以下定义：

- 单价和最低收费不是同一个字段
- 含税和不含税必须明确区分
- 加急倍率必须明确基数
- 报价有效期必须有起止时间
- 历史案例中的价格不得直接等同于当前报价规则

## 6.4 案例口径

案例只用于：

- 证明做过类似项目
- 提供表达骨架
- 提供场景经验

案例不能直接用于：

- 当前固定报价
- 当前固定能力承诺
- 当前固定交付承诺

## 七、完整人工操作步骤

本节按“数据准备 -> 入库 -> 审核 -> 发布 -> 检索验证 -> 业务使用 -> 反馈治理”完整说明。

## 7.1 前置准备

### 7.1.1 环境准备

人工操作前先确认：

1. `.env` 已配置数据库、Embedding、LLM。
2. 系统可正常打开 `http://127.0.0.1:8000/`。
3. 知识库管理页可正常进入。
4. 若要处理真实企微记录，企微会话存档配置已就绪。

### 7.1.2 数据整理准备

人工准备原始资料时建议按以下顺序：

1. 收集资料。
2. 去重。
3. 按一个知识点一条的原则拆分。
4. 对报价类资料补结构字段。
5. 对不确定内容标记为空，不要主观补全。

---

## 7.2 正式知识 Excel 导入步骤

适用：

- 常见问答
- 报价规则
- 流程规范
- 能力清单
- 历史案例
- 销售话术
- 邮件整理

### 7.2.1 准备 Excel

1. 进入知识库管理。
2. 打开 `导入中心`。
3. 选择导入类型。
4. 下载模板。
5. 按模板填写数据。

### 7.2.2 预览校验

1. 返回 `导入中心`。
2. 上传 Excel。
3. 点击 `预览`。
4. 查看失败行、跳过行、可导入行。
5. 修正原表格后重新预览。

### 7.2.3 确认导入

1. 预览通过后点击 `确认导入`。
2. 系统生成导入批次编号。
3. 数据写入正式知识草稿。
4. 报价类内容会尽量生成结构化报价规则。

### 7.2.4 审核发布

1. 进入 `批次` 或 `待审核`。
2. 点击 `提交审核`。
3. 审核人点击 `审核同意`。
4. 点击 `发布`。
5. 如门禁失败，按提示补字段、补规则、修正文案后再次发布。

---

## 7.3 长文本辅助拆分导入步骤

适用：

- 一整份长文档
- 一大段制度说明
- 一段混合了多个知识点的原始资料

### 7.3.1 操作步骤

1. 准备资料标题。
2. 准备长文本原文。
3. 调用“辅助拆分导入”入口。
4. 系统用 LLM-1 把长文本拆成多个候选知识切片。
5. 所有切片先进入 `draft`，等待人工审核。

### 7.3.2 人工审核重点

1. 是否一条切片只表达一个知识点。
2. 是否把价格错误留在普通文本里而没有结构化 `pricing_rule`。
3. 是否错误地把案例当能力规则。
4. 是否错误地把模板当正式规则。
5. 是否存在原文没有、模型补出来的内容。

---

## 7.4 会话 / 历史资料 / 邮件抽取到候选池步骤

适用：

- 真实企微会话抽取
- 历史案例文本抽取
- 邮件整理表抽取
- 销售纠错反馈转候选

### 7.4.1 会话或文本抽取

1. 准备 `session_id` 或文本数组。
2. 调用抽取入口。
3. 系统使用 LLM-1 从原文中抽取候选知识。
4. 结果写入 `知识候选`。

### 7.4.2 邮件 Excel 抽取

1. 准备邮件整理 Excel。
2. 上传至邮件抽取入口。
3. 系统解析 Excel 记录。
4. 系统调用 LLM-1 抽取候选知识。
5. 结果写入 `知识候选`。

### 7.4.3 人工审核候选

1. 进入 `知识候选`。
2. 查看候选状态、来源类型。
3. 打开详情。
4. 若内容不准确，点击 `编辑候选`。
5. 修正后点击 `转为草稿知识`。
6. 转草稿后，再走正式审核发布流程。

---

## 7.5 从销售反馈补知识步骤

### 7.5.1 操作步骤

1. 在 `企微实时智能` 中查看建议回复。
2. 若建议有问题，点击 `需修正`。
3. 填写修正意见。
4. 系统把问题写入命中日志或生成候选来源。
5. 进入 `知识候选` 查找对应记录。
6. 编辑后转为草稿知识。
7. 审核、发布。

### 7.5.2 人工修正重点

1. 修正错误的知识类型。
2. 修正错误的切片类型。
3. 补足结构化报价规则。
4. 删除没有原文依据的内容。

---

## 7.6 检索测试步骤

### 7.6.1 操作步骤

1. 打开 `检索测试`。
2. 输入真实客户问题。
3. 选择业务线、知识类型、语种、服务范围等条件。
4. 点击 `检索`。
5. 观察主命中、补充证据、同文档关联切片、置信度、人工复核标记。

### 7.6.2 人工判定重点

1. 是否命中了正确知识分类。
2. 是否命中了结构化报价规则。
3. 是否错误召回了同文档无关切片。
4. 是否错误放行了需要人工复核的问题。
5. 是否出现宽泛 query 却给出过度确定的结果。

---

## 7.7 回归测试步骤

### 7.7.1 操作步骤

1. 打开 `回归测试`。
2. 选择分组、类别、风险等级、业务线。
3. 运行回归。
4. 查看结果。
5. 对失败用例定位到知识文档、切片或报价规则修复。

### 7.7.2 人工重点

1. 报价问题必须命中 `pricing_rule`。
2. 能力问题必须命中能力证据。
3. 流程问题必须命中流程规则。
4. 需要人工复核的问题必须被标记。
5. 宽泛 query 不应直接给出确定性证据。

2. 查看 `命中日志`。
3. 把错误命中、高风险命中、人工复核高频问题转为候选。
4. 审核候选并转草稿。
5. 发布新版本。
6. 复跑检索测试与回归测试。

## 八、当前代码与 LLM 的数据处理逻辑

本节按数据流说明系统当前如何处理数据。

## 8.1 正式知识导入逻辑

### 8.1.1 处理目标

把业务人员准备的正式知识资料，变成：

- `knowledge_document`
- `knowledge_chunk`
- 必要时的 `pricing_rule`

### 8.1.2 当前处理逻辑

1. 读取 Excel。
2. 校验必要列。
3. 归一化知识类型、切片类型、业务线、语种、服务范围。
4. 若检测到报价内容，尽量抽取结构化 `pricing_rule`。
5. 对 `标题 + 内容` 调用 Embedding。
6. 写入 `draft` 状态的文档和切片。
7. 等待人工审核与发布门禁。

### 8.1.3 相关代码入口

- 知识分类配置：`backend/main.py` 中 `KB_KNOWLEDGE_CLASS_CONFIG`
- 报价规则推断：`backend/main.py` 中 `infer_pricing_rule_candidate`
- 正式入库主链路：`backend/main.py` 各导入接口与 `_create_pricing_rule`
- 数据模型：`backend/database.py`

---

## 8.2 长文本辅助拆分逻辑

### 8.2.1 处理目标

把一段混合知识点的长文本，拆成多个草稿切片。

### 8.2.2 LLM 处理逻辑

系统会给 LLM-1 一个“知识拆分助手”提示词，要求：

- 只依据原文整理；
- 一条切片只表达一个知识点；
- 只能使用规定的知识分类；
- 明确价格信息要尽量抽成 `pricing_rule`；
- 不确定字段填空，不得编造。

### 8.2.3 系统落库逻辑

1. 调用 `run_llm1_json_prompt`。
2. 校验返回的 `chunks`。
3. 归一化知识分类与字段。
4. 生成 Embedding。
5. 写入 `draft` 文档与切片。
6. 若可提取价格，则同时写入 `pricing_rule`。

### 8.2.4 相关代码入口

- 接口：`/api/kb/import/assisted_text`
- 提示词构造：`backend/main.py` 中 `import_assisted_text`
- LLM 调用：`backend/intent_engine.py` 中 `run_llm1_json_prompt`

---

## 8.3 候选知识抽取逻辑

### 8.3.1 处理目标

从高噪声资料中先抽取“候选知识”，而不是直接生成正式知识。

### 8.3.2 当前支持来源

- 会话消息
- 文本数组
- 邮件 Excel
- 销售反馈

### 8.3.3 LLM 处理逻辑

系统会给 LLM-1 一个“候选抽取助手”提示词，要求：

- 只允许根据原文抽取明确存在的知识；
- 没有明确知识点时返回空列表；
- 不能兜底编造常见问答；
- 报价、折扣、税费、赔付、能力承诺必须更高风险；
- `pricing_rule` 只能填原文中明确出现的数字和单位。

### 8.3.4 系统落库逻辑

1. 拼接会话或原始记录文本。
2. 调用 LLM-1 返回 `items`。
3. 校验每个候选对象。
4. 写入 `knowledge_candidate`。
5. 候选默认状态为 `candidate`。
6. 后续人工转草稿。

### 8.3.5 相关代码入口

- 候选抽取函数：`backend/main.py` 中 `_llm_extract_candidate_items_from_text`
- 候选建模：`backend/database.py` 中 `KnowledgeCandidate`
- 反馈转候选：`/api/kb/candidates/from_feedback`
- 候选转草稿：`/api/kb/candidates/{candidate_id}/promote`

---

## 8.4 企微实时会话到智能回复的逻辑

### 8.4.1 主链路目标

从真实会话生成销售建议，但必须受知识证据约束。

### 8.4.2 当前数据流

`MessageLog -> LLM-1 摘要 -> CRM 画像 -> query_features -> retrieve_knowledge_v2 -> format_knowledge_context -> LLM-2 -> validate_sales_assist_output -> 安全输出`

### 8.4.3 分步骤说明

#### 第一步：会话存储

- 真实企微消息进入 `message_logs`
- 当前会话最近消息会被取出作为上下文

#### 第二步：LLM-1 提取结构化摘要

LLM-1 按固定字段输出：

- 话题
- 核心诉求
- 关键事实
- 待办项
- 风险
- 待确认项
- 状态

相关代码：

- `IntentEngine.slow_track_analyze`
- `IntentEngine._run_llm1`
- `SYSTEM_PROMPT_LLM1`

#### 第三步：CRM 画像融合

系统读取 CRM：

- 公司
- 行业
- 商机
- 合同
- 生命周期阶段
- 客户层级
- 回款风险

相关代码：

- `IntentEngine.get_crm_context`
- `backend/crm_profile.py`

#### 第四步：query 特征推断

系统结合摘要和 CRM 推断：

- 业务线
- 知识分类意图
- 语种方向
- 服务范围
- 是否存在宽泛 query

相关代码：

- `IntentEngine.infer_query_features`

#### 第五步：知识检索

系统使用 `retrieve_knowledge_v2` 做：

1. 结构化过滤
2. 关键词候选
3. 向量候选
4. 排序打分
5. 生成主命中、补充证据、同文档关联切片
6. 生成状态：`ok / manual_review_required / low_confidence / ambiguous_query / no_applicable_knowledge`

相关代码：

- `IntentEngine.retrieve_knowledge_v2`

#### 第六步：证据格式化

系统把知识证据按三类组织给 LLM-2：

- 规则型知识
- FAQ 型知识
- 案例型知识

相关代码：

- `IntentEngine.format_knowledge_context`
- `IntentEngine.build_evidence_refs`

#### 第七步：LLM-2 生成回复

LLM-2 按固定模板输出：

- 企微回复参考
- 跟进思路说明

相关代码：

- `IntentEngine.generate_sales_assist_bundle`
- `SYSTEM_PROMPT_LLM2`

#### 第八步：输出后校验

系统在展示前会再次校验：

- 回复中的价格数字是否来自 `pricing_rule`
- 能力或流程承诺是否有 active 知识证据
- 知识检索是否已标记人工复核
- CRM 高风险客户是否需要更保守

如果不通过，则输出保守安全回复，不直接使用 LLM 原文。

相关代码：

- `IntentEngine.validate_sales_assist_output`
- `IntentEngine.build_safe_assist_response`

## 8.5 当前发布门禁与质量治理逻辑

### 8.5.1 发布前结构校验

发布前系统检查：

1. 文档是否有版本号
2. 是否至少有一条知识切片
3. 报价知识是否有生效期
4. 报价知识是否有结构化 `pricing_rule`

相关代码：

### 8.5.2 质量治理

系统会检查：

- 已发布报价切片是否缺失结构化规则
- 是否存在重复规则
- 是否存在冲突规则
- 文档或规则是否过期

### 8.5.3 命中日志

系统会记录：

- query
- 命中切片
- 分数
- 未命中原因
- 人工复核标记
- 最终回复
- 反馈结果

## 九、增强方案下建议补充的处理逻辑

本节不是当前已实现，而是为了让后续方案落地时，数据准备不返工。

## 9.1 建议新增业务事实层处理链路

建议未来主链路升级为：

`线程原始资料 -> 线程事实提取 -> 事实审核 / usable_for_reply 判定 -> 生成参考检索 -> LLM 生成 -> 状态一致性校验`

## 9.2 建议新增四维质量评分

建议对事实或可复用生成片段增加：

- 主题明确度
- 信息完整度
- 可复用度
- 证据可靠度

衍生结果建议包括：

- `kb_useful_score`
- `publishable`
- `usable_for_reply`

## 9.3 建议新增线程后效回流

建议准备并回流以下标签：

- 客户是否回复
- 是否推进到下一阶段
- 是否转正式报价
- 是否成交
- 是否发生售后

这样后续才能做：

- `business_effect_score`
- 模板排序增强
- 优秀回复筛选

## 十一、最终结论

当前系统已经具备：

- 正式知识导入
- 候选知识抽取
- 报价规则独立治理
- 检索质量门禁
- 企微会话知识证据驱动回复
- 输出后安全校验

但为了支撑后续增强方案，建议现在就统一准备两类数据：

1. **生成参考类数据**
   - 解决“能依据什么说”

2. **业务事实类数据**
   - 解决“当前这个客户线程处于什么阶段、现在能不能这么说”

只有这两类数据同时准备到位，后续系统才能从“安全回复”继续升级为“业务阶段正确、证据充分、真正可商业化落地的自动辅助回复系统”。
