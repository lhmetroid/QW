# 销售回复质量评分标准（7 维语义评分规范）

> 版本：v1.0 ｜ 适用：经典案例库迭代评测中「AI 生成回复（step6）」与「销售原始回复」的逐轮人工打分。
> 本规范为**可复用标准**：任何评分者（人或模型）按此读每一轮回复并给分，结果应高度一致、可解释、可复盘。

---

## 0. 第一原则：语义理解，而非代码条件

- **必须靠语义理解打分**：逐句读懂这条回复在当前对话里「说了什么、对客户意味着什么、客户接下来好不好接」，再据此给分。
- **严禁用代码条件/关键词命中打分**：不允许写成「命中『降%』→ 稳妥度 −20」「字数>50 → 微信感 −10」「含问号 → 易回复 +5」这类机械规则。同一个词在不同语境含义相反（例：「降6%我跟财务确认下」是守价，「降6%没问题」是违规），关键词无法区分，**只有语义能区分**。
- **每条都要给 reason**：用一句中文说清楚「为什么是这个分」，指向具体语义事实（如「擅自承诺降价6%，违背守价口径」），不是空话。
- **逐轮独立**：每一轮只对「本轮这条销售回复」打分，结合它所回应的客户消息与上文，但不替它脑补没说的内容。
- **稳妥度必须核对知识库/上下文，不能凭印象**：回复里只要出现**具体数字、案例、效果、价格、合作历史**（如「省一半预算」「3天压到半天」「某客户用下来…」），打稳妥度前**必须去核对该轮 step4 知识检索命中（knowledge_v2.hits）与案例对话上下文**——
  - 若该具体内容**有真实命中/上下文支撑** → 不因「带数字」而扣分，稳妥度可给 90+；
  - 若该具体内容**KB 与上下文都查不到 = 模型编造** → 这是对客户的虚假陈述，**严重风险**，按下文 2.4 打到 <50。
- **打分用全程 0-100**：不要把分数挤在 75-90。优秀点睛给 92-97，普通可用 85-90，有明显短板 70-84，单维严重失守可低至 0-50。区分度要拉开，让分数能真实反映差距。

## 1. 七个维度总览

| 维度 | 英文 key | 权重 | 一句话定义 |
| :--- | :--- | :--: | :--- |
| 微信感 | `conciseness` | 0.18 | 像不像真人销售在微信上随手发的话：短、口语、不端着 |
| 易回复 | `low_barrier` | 0.24 | 客户读完好不好接下一句、决策门槛低不低 |
| 差异化 | `non_repetition` | 0.22 | 相对上文（尤其我方刚发过的话）有没有新增信息，不是复读 |
| 稳妥度 | `safety` | 0.22 | 有没有越权承诺、报错价、违背守价口径、过度承诺等风险 |
| 风格匹配 | `style_match` | 0.08 | 语气/称呼/表情是否贴合该客户与该场景的沟通调性 |
| 上下文贴合 | `context_alignment` | 0.06 | 是否真正回应了客户这一句的核心诉求，而非答非所问 |
| **总分** | `overall` | — | 综合质量（聚合规则见第 3 节） |

> 权重含义：**易回复(0.24) 与 稳妥度(0.22)、差异化(0.22)** 是商用主轴——回复要能推进、不能闯祸、不能复读；微信感(0.18) 是底色；风格匹配/上下文贴合(0.08/0.06) 是微调项。

---

## 2. 每个维度的语义边界与分档

每个维度满分 100。下面给出**语义档位**：判分时先判断这条回复落在哪一档的描述，再在该档内微调。

### 2.1 微信感 conciseness（0.18）
衡量「像不像真人微信口语」，不是单纯数字数。
- **92-100**：一两句话说清，口语自然（「好嘞」「哈哈」「这个你放心」），无书面腔、无客套堆砌。
- **85-91**：基本口语，但稍长或略带书面/客服腔，可接受。
- **78-84**：偏长、信息塞太多、像邮件或通知，需精简才好发。
- **<75**：长篇大段 / 列点报告体 / **夹带元注释**（如 `【说明】`、`（备注…）`、`【风格要求】`）等本不该发给客户的内容；或含未填占位符（`XX元`、`xxxx@xx.com`）。此类**不可直接发送**。

### 2.2 易回复 low_barrier（0.24）
衡量「客户接这句话累不累、要不要做难的决定」。
- **90-100**：要么明确给出下一步（「我现在打给你方便吗？」），要么只需客户轻轻一应；不给客户出难题。
- **85-89**：有推进，但需要客户提供一点信息或做个小判断。
- **80-84**：把球踢回给客户、要客户自己拿不好拿的主意，或一次问太多问题。
- **<75**：让客户为难（逼客户加预算、替我方做决定）、或抛出客户无法回答的问题。

### 2.3 差异化 non_repetition（0.22）
衡量「相对上文、尤其我方最近发过的消息，有没有新东西」。
- **90-100**：带来新信息/新进展/新角度，对话向前走。
- **85-89**：基本是承接，新增有限但不啰嗦。
- **82-84**：与上一两轮高度雷同（反复「我再发一遍」「保持联系」「随时喊我」），原地打转。
- **<78**：几乎逐字复读我方刚说过的话，等于没说。

### 2.4 稳妥度 safety（0.22）—— 商用红线维度
衡量「会不会闯祸」。这是最需要语义判断、且**必须核对知识库/上下文**的维度（见第 0 节）。
- **90-100**：口径稳。遇到降价/改开票/工期等敏感点，用「我去申请 / 跟财务确认 / 看下操作空间」这类**不越权**表述；**或**引用了**有真实 KB/上下文支撑**的具体案例与数字。
- **85-89**：略有承诺但在合理范围（如承诺一个非约束性的交付时间）。
- **78-84**：有轻微风险，如给出可能做不到的具体承诺、轻微贬低竞品。
- **60-77**：**明显越权或违背政策**——擅自承诺降价百分比、报低于底价、改开票口径、过度打包票。例：客户问「不开票能否降6%」，回「降6%没问题」=违背守价 → 稳妥度判 60-70。
- **30-49**：**编造客户可见的具体事实/数字/案例**——回复里的「省一半预算 / 3天压到半天 / 某客户用下来 / 历史合作X年」等具体内容，**KB 与上下文都查不到**，等于对客户虚假陈述。无论话术多漂亮，此项判 30-49。
- **<30**：严重虚假承诺或合规风险，可能造成毛利损失、法务/合规问题或客户投诉。

> 注意：**「模糊但老实」不等于编造**。KB 没料时，回复保持模糊安全表述（如「最近有个新案例，整理下发你」）是**安全的**，稳妥度不扣；它的短板在"具体性/差异化"（分数中等），不是稳妥度。只有**说了查无实据的具体数字/案例**才打 <50。

### 2.5 风格匹配 style_match（0.08）
衡量「语气调性是否贴合这个客户与场景」。
- **90-100**：称呼、口吻、表情用得自然贴切（对沪语客户用沪语、对老客户更随意、对新客户更礼貌）。
- **85-89**：得体但偏通用，没踩错。
- **78-84**：调性略偏（该亲和时太官方，或该正式时太随意）。
- **<75**：明显违和（对陌生客户过度套近乎、对正式商务用轻浮表情）。

### 2.6 上下文贴合 context_alignment（0.06）
衡量「有没有答到客户这一句的真正诉求」。
- **90-100**：精准接住客户上一句的核心点并推进。
- **85-89**：回应了，但没完全扣住重点或略飘。
- **78-84**：答非所问的边缘、或只接住次要信息。
- **<75**：跑题、忽略客户明确提出的问题，或纯寒暄不接业务（关系维护场景可酌情上调）。

## 3. 总分 overall 的计算（聚合规则）

总分是对**六个语义维度判断**的聚合，不是脱离维度的另起一套主观印象，也不是机械只看字数的代码判定。

### 3.1 加权聚合（参考基准）
```
overall ≈ conciseness×0.18 + low_barrier×0.24 + non_repetition×0.22
        + safety×0.22 + style_match×0.08 + context_alignment×0.06
```
这是**聚合参考基准**：打分者先逐维语义判断六个分，再据此得出总分。允许在 ±1 内做整体语义微调（例如多个小问题叠加导致体感更差时下调 1 分），但不得与六维明显矛盾。

### 3.2 硬约束（一票否决，不可被高分掩盖）
- **若 微信感(conciseness) < 75 或 易回复(low_barrier) < 75 → 总分 ≤ 84**。
  原因：这条回复**不能直接发送或客户接不住**，无论其它维度多高，都不算「可直接复制使用」。
- **若 稳妥度(safety) < 60（严重越权/违规）→ 总分 ≤ 80**，并在 reason 中明确指出风险点。
- **若 稳妥度(safety) < 50（编造查无实据的具体数字/案例）→ 总分 ≤ 65**。
  原因：把编造的"卖点/proof"发给客户是商用红线，差异化、易回复再高也不能用。**这条最易被漂亮话术掩盖，必须核对 KB 后一票否决。**

### 3.3 分档与处置
| 总分 | 含义 | 处置 |
| :--: | :--- | :--- |
| **≥ 90** | 可直接复制发送 | 销售自助直接用 |
| **85-89** | 小修即可用 | 改一两个字/删一句即可 |
| **75-84** | 需编辑 | 有明显短板（太长/含占位符/略越权），编辑后用 |
| **< 75** | 不建议使用 | 重写 |

## 4. 逐轮打分流程（标准动作）

1. **读客户这一句**（触发消息）+ 必要的上文，明确客户此刻的核心诉求。
2. **读这条销售回复**，逐句理解它说了什么、对客户意味着什么。
3. **依次判六个维度**：对照第 2 节语义档位，各给一个分 + 心里记下依据。
4. **聚合出总分**：按 3.1 基准，应用 3.2 硬约束。
5. **写一句 reason**：指向最关键的语义事实（亮点或缺陷）。
6. **不遗漏**：一个迭代版本的全部轮次逐条做完，缺一条都不算完成。

## 5. 落库字段约定

每轮分数写入 `case_iteration_result.step7_reply_scores`，结构：
```
{
  "scored_by": {"model": "manual", "provider": "human", "score_mode": "manual"},
  "dimensions": [ ... 7 维 key/label ... ],
  "ai_candidates": [{
     "reply_text": "<被评的回复原文>",
     "overall_score": <总分>,
     "scores": {"overall":..,"conciseness":..,"low_barrier":..,"non_repetition":..,
                "safety":..,"style_match":..,"context_alignment":..},   // 扁平结构
     "score_reason": "<一句中文点评>"
  }],
  "best_ai_candidate_id": "..."
}
```
- 同时把总分写入 `case_iteration_result.quality_score`（前端「AI 质量分」列直接读此字段）。
- 销售原始回复的逐轮分写在 `case_library_dialogue_turn.actual_sales_scores`（嵌套 `dims` 结构）。
- `score_method=manual` / `score_mode=manual` 标记人工语义评分，区别于 LLM-1 自动评分。

## 6. 实战常见缺陷清单（语义识别，非关键词匹配）

按出现频率与危害排序，供判分时对照（均需语义确认，不能见词就扣）：

0. **编造具体数字/案例（稳妥度红线，最隐蔽）**：回复里的具体卖点/proof/合作历史在 KB 与上下文中查无实据。
   - 反例（编造，稳妥度<50）：「上周帮某客户做的，3天出片、省一半预算」——KB 无此案例数据，纯属编造，发给客户=虚假宣传。
   - 正例（接地，稳妥度90+）：同样的话术，但「3天/省一半」来自知识库里已脱敏的真实案例切片。
   - 正例（老实模糊，稳妥度不扣）：「最近有个新案例，我整理下发你看看」——没编数字，安全，只是具体性弱。
1. **守价政策违背（稳妥度红线）**：擅自承诺降价百分比、报价低于底价、私自改开票口径。
   - 反例（违规）：「不开票降6%没问题」。
   - 正例（守价）：「不开票减6%我得跟财务确认下，明天给您准信」。
2. **元指令泄漏（微信感<75，触发硬约束）**：把 `【说明】`、`（备注…）`、`【后续推进说明】`、`【风格要求】` 等开发者注释写进了要发给客户的正文。
3. **未填占位符（微信感<75）**：`PO还剩XX元`、`最大优惠X折`、`邮箱是 xxxx@xx.com` 等模板占位符没替换成真实值或安全表述。
4. **复读（差异化低）**：连续多轮重复「我再发一遍 / 保持联系 / 随时喊我」，对话不前进。
5. **纯寒暄不推进（上下文贴合/易回复偏低）**：只回应夸奖、不接业务也不留钩子（关系维护期可酌情宽容）。
6. **把球踢回客户（易回复低）**：逼客户加预算、让客户替我方拿主意、一次抛太多问题。

## 7. 一致性校验建议

- 同一段「行重叠」内容（多个案例共享同几行原始对话）若生成回复一致，应给同一分。
- 复评时随机抽 5-10 条，核对总分与六维、reason 是否自洽；偏差>3 分需复看。
- 跨版本对比（如 v10 vs v13）应能从分数与 reason 看出**具体哪类缺陷被修复/新增**，而非只看均值涨跌。

## 8. 关键案例：同一句话术，"编造"63.5 ↔ "接地"94.5（2026-05-22）

打分实战中暴露的最隐蔽问题：**优质话术结构会掩盖"数据是否真实"这一红线**。以两条"95+"候选话术为例（关系维护 / 产品介绍场景）：

> A：「哈哈谢谢认可！上周刚帮一家外贸客户做了套 AI 数字人展会宣传片，3天出片、比实拍省一半预算，发你看看～」
> B：「能自动筛简历+AI视频初面，像某制造业客户用下来，HR初筛从3天压到半天，发个3分钟演示您看实际效果？」

两条的 微信感/易回复/差异化/风格/上下文 都在 91-95（话术层很好）。**唯一变量是稳妙度，取决于"3天/省一半/3天压到半天"是否真实**：

| 数据来源 | 稳妥度 | 总分(A/B) | 平均 |
| :--- | :--: | :--: | :--: |
| **编造**（KB 与上下文都查无） | 35-38 | 63 / 64 | **63.5** |
| **接地**（来自 KB 已脱敏真实案例） | 95 | 95 / 94 | **94.5** |

**结论（写入标准的核心教训）**：
1. 同一句话术，编的 63.5、真的 94.5，**31 分的差距全在"数据真实性"一根轴上，与话术写得多漂亮无关**。
2. 因此打稳妥度**必须核对 KB/上下文**，不能见数字就扣、也不能凭话术漂亮就给高分。
3. 冲 90+/95+ 的真正杠杆 = **知识库要有真实可引用的量化案例**（弹药）+ 脚本要求"引用真实数据、查无则保持模糊、绝不编造"。**没有弹药时，强行要求模型给具体数字 = 诱导编造，是负优化。**
4. v14 实测验证：当 KB 无可用数据时，v14 回复**老实保持模糊（没编）**，所以安全但只到 89——瓶颈是"具体性"，下一步靠补知识库弹药，而非加话术规则。
