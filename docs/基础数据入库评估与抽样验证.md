## 1. 总结论

4 类数据不能都按同一种方式直接入正式知识库。

- 销售开发类邮件：可作为候选知识抽取来源，适合先入 `knowledge_candidate`，再人工审核、合并、去重、转正式知识。
- 除开发类日常业务沟通邮件：可作为候选抽取来源，但信息噪声较多。随机样本多为付款、确认、收件、文件版本等事务沟通，适合抽取流程/话术/风险点，不适合整批直接发布。
- 业务知识、业务流程、话术、案例：可作为基础知识来源，但原始 Excel 缺标准表头，需先转换为标准导入模板；涉及报价、支付、费用的条目需要补结构化 `pricing_rule` 或降为候选审核。
- 笔译报价规范：常规 `.docx`/zip 路径未发现 `word/document.xml`，但旧 Word 二进制内可恢复 UTF-16 正文；已拆分为结构化报价规则并写入知识库草稿。

## 3. 各类数据入库建议

### 3.1 销售开发类邮件

适合入库方式：

- 先作为 `email_excel` 来源进入候选池。
- LLM 从正文中抽取明确出现的能力、案例、流程、服务范围。
- 人工审核候选知识，去除营销寒暄、客户隐私、重复表达。
- 审核后再转为正式知识，走发布门禁。

本次随机 5 条验证：

- 解析成功。
- 真实 LLM 抽取成功。
- 实际写入 `knowledge_candidate` 38 条。
- 数据价值较高，能抽出服务能力、行业案例、服务流程等知识点。

### 3.2 除开发类日常业务沟通邮件

适合入库方式：

- 先进入候选池，不建议直接发布。
- 优先抽取可复用的业务规则、跟进话术、合同/发票/收款流程。
- 对单纯确认、付款催办、收件、文件版本沟通，应允许 LLM 返回空候选。

本次随机 5 条验证：

- 解析成功。
- 真实 LLM 抽取链路成功。
- 生成 0 条候选。

该结果不是报错。抽样内容包括付款跟进、发票发送、样机反馈、文件链接等，缺少可沉淀为通用知识的明确内容，因此系统按规则没有兜底生成 FAQ。

### 3.3 业务知识、业务流程、话术、案例

适合入库方式：

- 原始文件可作为知识库基础来源。
- 但原始文件只有两列内容，缺少标准表头和结构化字段。
- 需要先转换为标准模板字段：`标题`、`内容`、`知识类型`、`切片类型`、`业务线`、`服务范围`、`风险等级`、`生效日期`、`失效日期` 等。
- 涉及报价、价格、收费、支付、费用的内容需要补结构化 `pricing_rule`，否则会被发布校验或门禁阻止。

本次随机 5 条验证：

- 原始解析出 711 条。
- 已生成标准导入测试文件：`测试数据/基础数据抽样验证_20260420/business_knowledge_sample5_standard_import.xlsx`
- 标准预览结果：
  - 3 条有效。
  - 2 条失败。
- 失败原因：
  - `如何更新报价信息?`：`pricing_rule_missing_value`
  - `支付流程和条件是什么？`：`pricing_rule_missing_value`

这说明该文件可以作为基础数据，但需要清洗规则：凡是标题/正文触发报价、支付、费用类风险词，应补结构化报价/付款规则，或者先进入候选池人工确认。

### 3.4 笔译报价规范

当前文件 `docs/笔译报价.doc` 可以作为报价基础数据，但不能按普通 `.docx` 方式读取。

实际检查结果：

- 常规 zip/docx 路径未发现 `word/document.xml`，包内只看到 theme 资源。
- 旧 Word 二进制正文中可恢复 UTF-16 文本，已抽取到 `测试数据/笔译报价抽取_20260420/笔译报价_raw_text.txt`。
- 抽取命中关键原文：`翻译报价单`、`1000中文字符`、`中译英或日`、`法/德`、`英译日`、`日译英`。
- 已生成结构化入库文件：`测试数据/笔译报价抽取_20260420/笔译报价_结构化入库.xlsx`。
- 后台标准 Excel 预检结果：59 条有效、0 条失败、0 条跳过。
- 已真实写入 AISaleData PostgreSQL：批次 `pricing_excel_60a34f67c2e9`，创建 59 条 `draft` 知识、59 条 `draft pricing_rule`。

拆分粒度：

- 按语种方向拆分，例如 `中译德` 单独落为 `language_pair=zh->de`。
- 按资料类型拆分，例如普通/技术、合同条款、公关新闻稿、菜单品牌文学类、幻灯片、母语润稿、双外语翻译、排版服务。
- 按计价单位拆分，例如 `per_1000_chars`、`per_slide`、`per_english_word`、`per_japanese_char`。
- 示例落库结果：`中译德法/德报价 400元/1000中文字符`，`price_min=400.00`，`currency=CNY`，`unit=per_1000_chars`。

注意：

- 当前报价已入库为待审核草稿，不直接进入线上 active 检索。
- 报价属于高风险知识，需要人工审核后再发布，发布时仍走结构化校验与发布门禁。

## 4. 本轮代码适配

已补充邮件解析表头别名，使真实邮件文件可解析：

- `原始邮件内容`
- `本次邮件内容提纯后`
- `OriThisEmail`
- `PureEmailText`

并跳过第二行英文表头，避免把 `OriThisEmail / PureEmailText` 误当作正文。

## 5. 后续建议

建议把基础数据入库拆成三条管线：

- 邮件类：`Excel -> EmailImportService -> LLM 候选抽取 -> candidate -> 人工审核 -> 发布`
- 业务知识类：`两列表格 -> 标准模板转换 -> 预览校验 -> draft -> review -> 发布门禁 -> active`
- 报价规范类：`结构化报价表 -> pricing_rule -> 发布门禁 -> active`

报价类数据不要只以正文形式入库，必须形成结构化 `pricing_rule`，否则后续报价检索和发布门禁会持续拦截。
