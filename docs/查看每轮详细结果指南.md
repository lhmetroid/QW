# 查看每轮迭代详细结果指南

> 适用范围：「经典案例迭代优化」页面（60 案例批量跑 7 步链路）每轮测试的全部相关内容怎么看、看哪里、文件路径都列清楚。
>
> 路径约定：本文里所有相对路径都从仓库根 `D:\items\QW\` 起算。

## 0. 最终分析目标

经典案例迭代优化不是为了把测试均分做高，而是为了最终商业化的 AI 智能回复：生成内容要让销售在真实企微侧边栏里直接复制就能用，能减少销售思考和编辑成本，并且不引入价格、承诺、税务、交期、客户关系等风险。

因此每轮分析必须看完整链路，不能只看最后的 AI 回复和总分。所有中间输入输出都是分析对象：

| 环节 | 必看问题 |
| --- | --- |
| 案例源数据 | 核心对话、背景对话、人工原始分是否合理；触发点是否是真实客户问题 |
| 触发切分 | `visible_messages` 是否只包含触发点及之前；`post_trigger_messages` 是否没有提前喂给 LLM；`actual_sales_replies` 是否从触发后第一段销售回复开始 |
| 第1步摘要 | 主题、核心诉求、风险、待确认事项是否抓准 |
| 第2步 CRM | 真实案例 not_found 要排查 external_userid/session_id；案例库合成 ID not_found 可视为预期 |
| 第3步线程事实 | 是否把本轮业务事实、客户意图、待办项提取对 |
| 第4步知识检索 | 区分知识库缺数据、检索太严、命中不相关、命中但生成没用上 |
| 第5步 LLM-1 对比 | 是否发现摘要/知识/风险冲突 |
| 第6步生成 | 话术是否口语化、短、稳妥、可复制；是否多问无必要问题 |
| 第7步评分 | 总分是否和分项、评分理由一致；高分是否真达到商业可用 |

## 1. 一轮迭代到底由哪些"产物"组成

每点击一次 **▶ 启动新一轮迭代** 都会生成一个 `run`（迭代记录），包含 60 个 `result`（每个案例 1 条）和 1 份"得分分析 + AI 自检"+ 3 份 AI 工具分析 + 1 份 LLM-1 汇总。

| 产物 | 数据库表 / 文件 | 说明 |
| --- | --- | --- |
| 迭代汇总（每轮 1 行） | `case_iteration_run` | 版本号、状态、平均/最低/最高质量分、Git SHA、备份状态 |
| 60 个逐案例结果 | `case_iteration_result` | 7 步链路全部 JSON 快照 + 评分 + 耗时 |
| 案例源数据（人工标注） | `case_library_case` | 12 场景 × 5 案例（来自 `sales_scenario_cases_v2.md`）|
| 后端运行日志 | `logs/caselib_be4.out` / `logs/caselib_be4.err` | 每轮跑时的 stdout/stderr |
| Git 备份 commit | `git log` | 跑完后自动 `chore(caselib): iteration vN auto-backup` |

## 3. 数据库直接查（最准确，开发/排查必备）

数据库连接见 `.env` 的 `DATABASE_URL`。下列 SQL 都按"v{N}"对应的 run 取。

### 3.1 取最新 5 轮迭代的汇总
```sql
SELECT version_no, status, total_cases, success_cases, failed_cases,
       ROUND(avg_quality_score, 2) AS avg, ROUND(min_quality_score, 2) AS min, ROUND(max_quality_score, 2) AS max,
       avg_latency_ms, git_short_sha, git_dirty, backup_status,
       triggered_at, finished_at
FROM case_iteration_run
ORDER BY version_no DESC
LIMIT 5;
```

### 3.2 取某一轮的 60 个案例结果（按质量分升序，先看最差）
```sql
SELECT scenario_code, scenario_rank, quality_score, kb1_eval_score, kb2_eval_score,
       quality_status, latency_ms, LEFT(step6_sales_advice, 60) AS advice_preview,
       error_message
FROM case_iteration_result
WHERE run_id = (SELECT run_id FROM case_iteration_run WHERE version_no = :v)
ORDER BY quality_score ASC NULLS FIRST;
```

### 3.3 取某个案例的完整 7 步 JSON
```sql
SELECT scenario_code, scenario_rank,
       step1_summary, step2_crm_info, step3_thread_business_fact,
       step4_knowledge, step5_llm1_compare,
       step6_sales_advice, step6_compare_advice,
       step7_reply_styles, step7_reply_scores,
       quality_score, kb1_eval_score, kb2_eval_score,
       quality_status, latency_ms, stage_status, error_message
FROM case_iteration_result
WHERE run_id = :run_id AND scenario_code = 'S09' AND scenario_rank = 4;
```

### 3.4 取某轮的"三 AI 分析 + 汇总"
```sql
SELECT version_no, analysis_codex, analysis_antigravity, analysis_claude_code, analysis_summary,
       improvement_analysis, ai_next_step_plan
FROM case_iteration_run
WHERE version_no = :v;
```

## 4. 后端 API 直接调（不开 UI 时用）

后端默认端口 `8071`，所有路径前缀 `/api/case_lib`：

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| GET | `/api/case_lib/iterations` | 列出所有迭代（含三 AI 分析与汇总字段）|
| GET | `/api/case_lib/iterations/{run_id}` | 单轮迭代 + 60 条 result |
| POST | `/api/case_lib/iterations/start` | 触发新一轮迭代（body: `{mode, change_summary, triggered_by}`）|
| PUT | `/api/case_lib/iterations/{run_id}/analysis` | 提交某 AI 工具的分析（body: `{source: codex\|antigravity\|claude_code, content: "..."}`）。三者齐全自动触发 LLM-1 汇总 |
| POST | `/api/case_lib/iterations/{run_id}/analysis/regenerate_summary` | 手动重跑 LLM-1 汇总 |
| GET | `/api/case_lib/cases` | 60 个案例的源数据（含人工原始分、核心对话）|
| POST | `/api/case_lib/reimport` | 重新解析 `sales_scenario_cases_v2.md` 并写入 `case_library_case`|

### 示例：用 curl 提交 Codex 的分析
```bash
curl -X PUT "http://127.0.0.1:8071/api/case_lib/iterations/{run_id}/analysis" \
  -H "Content-Type: application/json" \
  -d '{"source":"codex","content":"## 本轮发现\\n1. ...\\n## 下一步\\n1. ..."}'
```

返回里有 `summary_status`：
- `llm1_summarized`：三者齐全，已用 LLM-1 综合
- `fallback_concat_no_llm1`：LLM-1 没配置，回退拼接
- `fallback_llm1_empty` / `fallback_llm1_error`：LLM-1 调用失败，回退拼接
- `skipped_incomplete`：还有 AI 未填，未生成汇总

## 7. 评分维度（人工 vs AI 完全一致）

7 个维度（来自 [intent_engine.py:2451](../backend/intent_engine.py)）：

| key | label | overall 公式权重 |
| --- | --- | --- |
| overall | 总分 | （由下 6 项加权得出） |
| low_barrier | 易回复 | 0.24 |
| non_repetition | 差异化 | 0.22 |
| safety | 稳妥度（validation 警告/封堵扣分） | 0.22 |
| conciseness | 微信感（按字数，≤50 字 96 分，>80 字 62 分） | 0.18 |
| style_match | 风格匹配 | 0.08 |
| context_alignment | 上下文贴合 | 0.06 |

`overall = conciseness*0.18 + low_barrier*0.24 + non_repetition*0.22 + safety*0.22 + style_match*0.08 + context_alignment*0.06`

人工回复和 AI 候选**走同一函数 `score_reply_candidates`、同一 prompt、同一套维度**——唯一差别：人工回复 `style=None`，不参与 style_match 加成。先算启发式，再让 LLM-1 重打；LLM-1 失败回退启发式。

### 商业化可复制评分口径

分数含义要按“销售是否能直接复制使用”解释：

| 分数段 | 解释 |
| --- | --- |
| 90+ | 可直接复制，无需补信息；关键事实完整，风险稳妥，微信口吻自然 |
| 85-89 | 基本可用，但还有轻微措辞或推进细节可改 |
| 75-84 | 可参考，需要销售修改后再发 |
| <75 | 不建议直接用 |

评分复核时要特别防止“理由承认问题但总分很高”。如果出现以下情况，原则上不应给 90+：
- 缺关键事实，例如已交付、付款、发票、交期、价格口径没有说清。
- 总分与分项冲突，例如 `conciseness < 75`、`context_alignment < 80` 但 `overall >= 90`。
- 评分理由包含“未直接回应”“缺乏具体行动”“略显冗长”“不够具体解决问题”。
- 回复需要销售再判断、再删改，不能直接复制。

## 8. 表格里 3 个分数怎么取的（v3 后修复）

| 列 | 计算口径 |
| --- | --- |
| **AI 质量分** | `reply_scores_v2.ai_candidates` 里 `overall_score` 的最大值 |
| **私域库分** | `ai_candidates` 中 `knowledge_source == "knowledge_v2"` 的候选 overall 最高 |
| **外部库分** | `ai_candidates` 中 `knowledge_source == "knowledge_external_api"` 的候选 overall 最高 |
| **人工原始分** | 静态：来自 `sales_scenario_cases_v2.md` 的 `**质量分（最高）**` 字段（v2.md 里 82% 都是 70/100，所以同分密集）|
| **Δ（AI − 人工）** | 上面两列直接相减；绝对值 ≥1 高亮 |

> 历史 bug（v2 之前）：`_caselib_run_real` 取的是不存在的 `total/kb1_score/kb2_score` 字段，导致永远落到启发式兜底分 65.x。已在 [backend/main.py](../backend/main.py) `_caselib_run_real` 评分提取段修复。

> v3 数据纠偏（2026-05-19）：发现 11 条 `ai_candidates.score_reason=llm1_failed` 且没有 `overall_score`，但结果表还残留 82-95 的旧 `quality_score`。这些已改为 `quality_score=NULL`、`quality_status=real_score_missing`、`status=failed`，不再计入均分。评分失败就是失败，不能回退假分或沿用旧分。

### 销售原始回复分为空时

按当前要求，销售原回复不用每次重新打分。若 `actual_sales_replies[0].overall_score` 为空且 `score_reason=llm1_failed`：

1. 先确认是不是评分模型失败，而不是销售回复不存在。
2. 不重跑销售评分时，可使用 `case_library_case.quality_score_md` 回填总分。
3. 回填时必须标记 `score_reason=filled_from_quality_score_md_no_rescore`，避免后续误认为这是本轮 LLM 新评分。

## 9. 为什么有些步骤会空

| 步骤为空 | 原因 | 修复 |
| --- | --- | --- |
| 第5步·大模型1对比摘要 = `{}` | LLM-1 对比模型未配置 | `.env` 填上 `LLM1_COMPARE_API_URL/KEY/MODEL` |
| 第6步·销售建议（对比模型）= 空 | LLM-2 对比模型未配置 或 评测模式未开启 | 同上，并保证 `_caselib_run_real` 发起请求时 body 带 `evaluation_full_pipeline: True` |
| 第7步候选评分 = 空 | 同上 | 同上 |
| 私域库分 / 外部库分 = `-` | `reply_scores_v2.ai_candidates` 为空（评分未跑 / 评分回退启发式失败）| 检查 LLM-1 是否可用 |
| 第2步·客户画像 = `not_found` | 案例库用合成 `caselib_ext_*` 调 CRM，**这是预期**，不是 bug |
| "无标题" 案例 | `sales_scenario_cases_v2.md` 中原始标题就是"（无标题）" | 已加自动从核心对话首条客户消息回退；执行 `python backend/_case_lib_import.py` 刷新即可 |
| `llm1_failed` 但表格显示高分 | 历史残留分或错误兜底 | 必须置空 `quality_score`，标为 `real_score_missing/failed`，不计入均分 |
| 截图看到 90+ / 92+ | 不代表真实高质量 | 打开第7步检查分项、评分理由、AI候选是否实际有 `overall_score` |

## 10. 一轮迭代完整流转图

```
①触发：前端「▶ 启动新一轮迭代」
   └→ POST /api/case_lib/iterations/start
       └→ INSERT case_iteration_run (status=queued, version_no=N)
       └→ 后台线程 _caselib_run_iteration_background

②逐案例：60 个 case 串行
   for each case in case_library_case (按 scenario_code, rank):
     ├ 拼合成 external/sales userid 注入 message_logs
     ├ POST http://127.0.0.1:8071/api/wecom/sidebar_assist
     │   body 带 evaluation_full_pipeline=True
     │   → 拿到 7 步完整 JSON（含 LLM1对比 / LLM2对比 / scoring）
     ├ 解析 body → quality_score / kb1 / kb2 / 7步内容
     └ INSERT case_iteration_result

③收尾
   ├ 算 avg / min / max / latency
   ├ _caselib_compose_improvement_analysis → improvement_analysis（机器基线）
   ├ _caselib_compose_ai_next_plan         → ai_next_step_plan（机器基线）
   ├ _caselib_git_backup                   → 自动 commit + push
   └ UPDATE case_iteration_run SET status='success', finished_at=now()

④人工填三 AI 分析
   for src in [codex, antigravity, claude_code]:
     PUT /api/case_lib/iterations/{run_id}/analysis (source=src, content=...)
   └ 第 3 个保存时自动触发 _caselib_compose_three_ai_summary
     └ LLM-1 综合 → analysis_summary

   > [!IMPORTANT]
   > **【AI 边界与协同规范（严禁越界）】**
   > * **各行其职**：每个 AI 助手（Codex, Antigravity, Claude Code）在分析时，**必须且仅能**填写属于自己名称的文本框，严禁越界填充其他 AI 的文本框！
   > * **严禁提前代理汇总**：任何非最后一个保存的 AI，**绝对不要**提前通过代码或 API 强行写入/更新底部的 `analysis_summary`（LLM-1 综合汇总）。
   > * **汇总触发权归属**：底部的综合汇总必须且只能由**最后一个填满（第 3 个保存）**的 AI 触发调用，或由人工通过专用的汇总重置/生成端点发起。非最后一个 AI 代理汇总属于严重越界行为，会打乱协同排班与评分的隔离性。

## 12. v3 三AI综合分析结论（2026-05-19）

> **最终目的**：所有分析和优化均服务于”商业化AI智能回复”落地——销售拿到AI生成话术，无需任何修改，直接在企微复制发给客户。评分标准的锚点是”可直接使用”，而非”格式正确/无硬伤”。前期案例测试不用重跑，只对本次已跑出的各参数（从step1到step7的所有输入输出，不仅仅是最后的分数）及流程代码做分析和优化建议。

### 核心结论：v3 AI质量分整体虚高约 5~8 分

经 Codex、Antigravity、Claude Code 三方独立分析一致确认，v3 平均分 87.63 存在系统性虚高，修正后实际有效均分约 **79~82**，三大根因：

1. **公式未覆盖**：v3 运行时代码直接使用 LLM-1(qwen14b) 返回的 `overall` 字段，而非加权公式值（后续 commit 才修复）
2. **conciseness 系统性偏低**：23/49 个案例字数 ≤50字 却被打了 60~70 分——qwen14b 将”≤50字应高分”误解为”50分封顶”，短回复本该得加分反被拉低
3. **safety 普遍虚高**：49 个案例中 45% safety=100，78% safety≥90，普通跟进话语不应给满分

**关于”92分是否真高质量”**：否。v3 最高评分案例（47字，口语化推进）按修正公式估算约 83~87 分；部分高分回复篇幅超百字，完全不能直接使用。**高分≠可直接使用。**

### 11个 real_score_missing 案例

S06-3、S06-5、S02-4、S05-2、S07-2、S07-3、S06-4、S03-2、S07-1、S03-3、S03-1 的 AI 回复生成成功，只是 LLM-1 评分超时/失败（`score_reason=llm1_failed`），`quality_score=NULL`，不纳入均分统计。需单独补评分（不需要重跑全链路）。

### 销售原始分说明

`step7_reply_scores.actual_sales_replies` 字段 60/60 均已入库，评分合理（4字”去问下哦”→30分；28字幽默破冰→90分）。前端读取路径正确（`actual_sales_replies`），若 UI 显示空白请刷新或检查 API 响应格式。
