# 特征提取模型速度对比与实施建议（2026-05-12）

## 1. 当前特征提取实现原理

当前企微侧边栏主入口是：

- `backend/main.py -> /api/wecom/sidebar_assist`

在接口流程中，如果当前会话没有可复用的 `IntentSummary`，或者请求带了 `force_refresh`，会执行第一阶段特征提取：

- `IntentEngine._run_llm1(session_id, context)`
- `IntentEngine.build_llm1_request(...)`
- `IntentEngine.run_llm1_request(...)`

`build_llm1_request(...)` 会把最近对话按下面形式拼成 `conversation_text`：

```text
[客户]: ...
[销售]: ...
```

然后注入 `SYSTEM_PROMPT_LLM1`，要求模型输出纯 JSON，字段包括：

```text
topic, core_demand, key_facts, todo_items, risks, to_be_confirmed, status
```

当前配置里 `STAGE1_USE_LLM2=true`，因此第一阶段实际走的是 LLM-2 主配置，即 `deepseek-chat`，不是本地 `LLM1_MODEL=qwen14b`。

特征提取不是最终回复，但它影响后面多个环节：

- CRM/会话事实合并：用于理解客户当前主线、已有事实、待确认项。
- 知识库 1/2 检索：用于形成更准确的检索意图和关键词。
- LLM-2 主生成：作为最终回复 prompt 的结构化上下文。
- 质量校验：用于判断回复是否遗漏客户当前需求、时间点、待办和风险。

因此特征提取的提速不能通过减少上下文、删减字段、弱化事实提取来做。这样虽然可能降低耗时，但会破坏后续知识库检索和最终回复质量，容易出现答非所问。

测试输入：

- 不是只把“客户最后一句”喂给模型；表格里的“客户最后一句”只是用于标识测试锚点。
- 实际输入使用现有 `SYSTEM_PROMPT_LLM1`，包含角色要求、输出 JSON 字段、最后一句优先、多事项拆分、`key_facts` 结构等完整要求。
- 每条样本取 anchor 消息之前可见的最近最多 15 条企微上下文；case_4 实际只有 4 条，其余样本为 15 条。
- 上下文使用当前业务格式 `[客户] / [销售]` 拼接后替换 `{{conversation_text}}`。
- 先 warmup，再统计热启动调用耗时。

实际 prompt 模板如下：

```text
你是一个严谨的销售线索分析专家。请根据下方的客户企微对话记录，提取结构化特征。
要求以纯JSON对象格式输出（不可包含```json等markdown标识），字段包括：
topic, core_demand, key_facts, todo_items, risks, to_be_confirmed, status

额外要求：
1. 优先围绕“客户最后一句”识别当前这轮真实推进点，不要被更早的话题带偏。
2. 若客户最后一句只是确认收到、确认时间、承诺稍后发资料、或简短回复 OK/收到/好，不要把它误判成新的大需求。
3. 若对话里同时出现多个时间点/多个会议/多个事项，必须拆开，不能合并成一个事件。
4. `key_facts` 必须尽量输出以下结构（缺什么留空，不要编造）：
   - `latest_customer_reply_type`: `ack_only` / `schedule_confirmation` / `promise_send_later` / `question` / `new_request` / `other`
   - `latest_customer_reply_goal`: 用一句话写客户这句回复真实想表达什么
   - `timeline_items`: 数组，每项包含 `time`, `topic`, `status_scope`（只能填 `ongoing` 或 `future`）, `status`, `owner`, `next_action`
   - `current_focus`: 当前这轮最该回应的一个点
   - `separate_topics`: 若存在多条并行事项，在这里拆开列出
5. `topic` 写当前主线；`core_demand` 写客户当前最需要我方承接/确认/推进的点。
6. 如果客户已经回答了我方上一问，`todo_items` 里不要再重复同一个追问，而是写我方下一步动作。
7. 若客户明确给了两个时间点，例如“6月某会议”和“本周三某会议”，必须在 `timeline_items` 中拆成两条，分别标记 `future` / `ongoing`。

【企微对话实录】:
[客户]: ...
[销售]: ...
[客户]: 当前锚点消息
```

本次新增测试只更新本文档，没有新增 Python 脚本文件。

## 7. 结论

### 是否能把特征提取压到 0.5 秒以内

按本次 `10.0.0.222` 的五条样本热启动结果，当前 3B 级模型最快在 0.63-2.28 秒之间，平均约 1.09 秒，且质量和 JSON 稳定性不足。因此在不牺牲质量的前提下，当前这台 Ollama 主机上还不能把正式特征提取稳定压到 0.5 秒以内。

如果目标是 0.5 秒以内，需要另行验证专门的小模型、蒸馏模型、约束解码、GPU 推理服务或规则前置缓存。但这类方案必须经过质量回归，不能直接替换当前特征提取。

### 14B

`10.0.0.222` 的 14B 输出较稳定，5/5 JSON 合法，热启动平均约 5.31 秒，比当前 deepseek 五条样本平均约 8.84 秒快。但 14B 仍是秒级，不适合作为“秒开”体验的唯一优化点。

`192.168.31.102` 的 14B 平均约 54.83 秒，明显不可作为在线主路径。

适合方向：

- 作为本地/内网备用特征提取模型。
- 在 deepseek 抖动或外部 API 慢时做 fallback。
- 继续扩大样本验证质量后，再评估是否替换当前第一阶段。

### 7B

`10.0.0.222` 的 7B 平均约 3.06 秒，JSON 合法率 5/5，是本次最值得继续验证的候选。但它在短句确认、最后一句聚焦、多话题分离上仍有偏差风险。

`192.168.31.102` 的 7B 平均约 12.75 秒，慢于当前 deepseek 平均值，不建议用于在线替换。

适合方向：

- 扩大到至少 30-50 条 5.11 样本做质量对比。
- 重点检查 `latest_customer_reply_type`、`current_focus`、`todo_items` 是否会带偏最终回复。
- 如果质量合格，可作为特征提取主候选，比当前 deepseek 明显快。

### 3B

`10.0.0.222` 的 3B 级模型平均约 1.09 秒，但 5 条中 1 条 JSON 失败，且多个 case 输出明显偏薄。

不建议方向：

- 不建议直接替换正式特征提取。
- 不建议让它决定最终知识库检索和 LLM-2 生成上下文。

可尝试方向：

- 用作前置轻分类，例如识别“收到/确认/稍后发/时间确认/文件已发”等简单场景。
- 用作 Step 0 的 `instant_reply_preview` 信号，先给前端展示低风险预览，但不跳过原 6 步链路。
- 用作缓存命中或规则判断的辅助，不作为最终事实来源。

### 192.168.31.102

192 机器本次测试不适合承载在线特征提取：

- 14B 和 7B 的 token/s 明显低于 `10.0.0.222`。
- 14B 出现一次超过 2 分钟的模型加载，说明模型常驻或硬件资源存在问题。
- `qwen3-coder:480b-cloud` 可调用，但平均约 29.70 秒，且是 coder cloud 模型，不适合作为特征提取主模型。

### case_1：PPT 明天检查、13 号早上正式使用

````text
你是一个严谨的销售线索分析专家。请根据下方的客户企微对话记录，提取结构化特征。
要求以纯JSON对象格式输出（不可包含```json等markdown标识），字段包括：
topic, core_demand, key_facts, todo_items, risks, to_be_confirmed, status

额外要求：
1. 优先围绕“客户最后一句”识别当前这轮真实推进点，不要被更早的话题带偏。
2. 若客户最后一句只是确认收到、确认时间、承诺稍后发资料、或简短回复 OK/收到/好，不要把它误判成新的大需求。
