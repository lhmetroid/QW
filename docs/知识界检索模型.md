# 知识库检索模型对比（2026-05-12）

## 1. 分析范围

本文只分析企微智能回复链路中的知识库检索阶段：

- 知识库1：本地业务知识库，代码路径为 `IntentEngine.retrieve_knowledge_v2(...)`。
- 知识库2：外部销售知识库 API，代码路径为 `_search_sales_kb_api(...)`，接口为 `POST /kb-units/qa-retrieve`。

本次重点是用 `10.0.0.222:11434` 上的 embedding 模型复测知识库1，并用同 5 条 query 复测当前知识库2 API 耗时。所有测试都做了预热，尽量排除首次模型加载干扰。

## 2. 当前实现原理

知识库1检索流程：

```text
LLM-1 特征提取
-> infer_query_features(...)
-> build_retrieval_query(...)
-> retrieve_knowledge_v2(...)
-> query embedding
-> DB 过滤 active/effective/business_line/knowledge_type/service_scope
-> structured candidates + keyword prefilter
-> semantic + keyword + bm25 + exact phrase + intent + metadata 综合打分
-> 返回 hits/evidence_context/evidence_refs
```

知识库1当前已有向量情况：

| 项 | 值 |
|---|---|
| active chunks | 1454 |
| embedding_provider | `ollama` |
| embedding_model | `qllama/bge-m3:latest` |
| embedding_dim | 1024 |

因此，知识库1不能随便把 query embedding 换成其他模型。若 query 用 `qwen3-embedding`、`bge-large`、`nomic`，但库内 chunk 仍是 `qllama/bge-m3`，语义分数会失真；必须全库重新 embedding 后才有正式可比性。

知识库2检索流程：

```text
build_retrieval_query(...)
-> POST https://knowledgebase.speedasia.net/kb-units/qa-retrieve
-> 返回 answers
-> 归一化为 knowledge_external_api hits
```

知识库2当前 API 只传 `question`，不支持在本系统侧指定 `10.0.0.222` 的 embedding 模型。因此本轮知识库2只能测当前 API 真实耗时和返回结果，不能像知识库1一样切换 embedding 模型。

## 7. case_4 候选数偏大的原因（修复前后）

现有流程不是“先检索前 5 条，再对 5 条打分”。实际是：

```text
1. 根据 active / effective_time / business_line / knowledge_type / service_scope 等结构化条件筛候选
2. structured_filters 最多取 KB_CANDIDATE_LIMIT 条，当前默认 300
3. 根据 query_terms 做 keyword_prefilter，再最多取 KB_CANDIDATE_LIMIT 条，当前默认 300
4. 两路候选按 chunk_id 合并去重
5. 对合并后的所有候选做 semantic / keyword / bm25 / exact / intent / metadata 等综合打分
6. 排序后截取 top5
```

所以 top5 是最后一步，不是候选生成阶段。

说明：下表和本节 `kb_case_4` 的具体数字，都是在本次把 `KB_CANDIDATE_LIMIT` 从 `500` 改到 `300` 之前实测得到的历史结果，用于解释旧链路为什么会慢；若按最新代码重跑，`kb_case_4` 的结构化上限将变为 `300`，不再是 `500`。

5 条样本在改限前代码下的候选规模如下：

| Case | structured_filters | keyword_prefilter | 合并后候选 | 说明 |
|---|---:|---:|---:|---|
| kb_case_1 | 12 | 7 | 12 | 小候选集 |
| kb_case_2 | 48 | 0 | 48 | 结构化命中为主 |
| kb_case_3 | 6 | 0 | 6 | 候选很小 |
| kb_case_4 | 500 | 0 | 500 | 结构化命中达到上限 |
| kb_case_5 | 48 | 0 | 48 | 结构化命中为主 |

这也回答了“1400 多个切片是不是都检索到了”这个问题：不是。当前库里 active chunk 虽然是 `1454`，但现有流程并不是把 1454 条全部拿出来打分；它先做结构化过滤和关键词预筛，再只对合并候选打分。改限前的 5 个样本里，只有 `kb_case_4` 达到 `structured_filters=500` 上限，其余都远小于全库规模。

`kb_case_4` 的实际条件如下：

| 项 | 值 |
|---|---|
| query | 我懂你意思了。。。；客户需要我方完成6个PPT的排版修改，并处理额外2个PPT的抄袭问题，同时确认最终报价和交付时间。；PPT排版与修改服务交付；ongoing |
| query_features | `business_line=general`；`knowledge_type=[pricing, process, faq]` |
| query_terms | 报价、交付、客户、我懂你意思了、客户需要我方完成6个ppt的排版修改、并处理额外2个ppt的抄袭问题、同时确认最终报价和交付时间、ppt排版与修改服务交付、ongoing |
| 修复前 structured_filters | 500 条，达到上限 |
| 修复前 keyword_prefilter | 500 条，达到上限 |
| 修复前合并后候选 | 653 条 |
| 修复后 keyword_prefilter_terms | 我懂你意思了、客户需要我方完成6个ppt的排版修改、并处理额外2个ppt的抄袭问题、同时确认最终报价和交付时间、ppt排版与修改服务交付 |
| 修复后 structured_filters | 500 条，达到上限 |
| 修复后 keyword_prefilter | 0 条 |
| 修复后合并后候选 | 500 条 |
| 最终 top5 | 5 条 |

判断：这不是代码执行 bug，而是当前检索策略的设计结果。但它是一个性能和质量风险点。

主要原因：

- `business_line=general` 且 `knowledge_type=[pricing, process, faq]` 太宽，直接把大量通用报价、流程、FAQ 放进候选池。
- 修复前 `query_terms` 里包含 `报价`、`交付`、`客户` 这类高频词，keyword_prefilter 命中面很大。
- 修复前 structured 和 keyword 两路各自最多取 500，合并后可能超过 500，本例达到 653。
- 只有合并后才排序 top5，因此候选越大，SQL 拉取、Python 打分、排序都会变慢。

这类原因确实会导致现有知识库1向量检索慢，尤其是 broad query、general 场景、报价/交付/客户这类高频词混在一起时。

本轮已做的低风险修复：

- `keyword_prefilter` 不再直接吃完整 `query_terms`，而是先过滤 `客户`、`报价`、`交付`、`流程`、`active/ongoing` 这类泛词。
- 结构化候选和关键词候选先只取 `chunk_id`，合并去重后再一次性加载详情，避免把两路最多各 `500` 条完整行都拉回 Python。

修复效果：

- `kb_case_4` 候选数从 `653` 降到 `500`。
- `kb_case_4` 总耗时从 `3844.04ms` 降到 `3068.81ms`。
- `kb_case_4` top1 未变化，仍是 `商机7问完整版 · 应用背景及对手：用途及如何解决`。

### 总览

| Case | 当前 .env 总耗时 | 原 KB1 | 候选数 | top1 |
|---|---:|---:|---:|---|
| kb_case_1 | 1675.97ms | 1276.37ms | 12 | 分批交稿项目应按优先批次和剩余批次拆分报价与交期 |

说明：上表是 `retrieve_knowledge_v2(...)` 端到端总耗时；下表是检索主路径的分步耗时，到“排出 top5”为止，因此下表求和会略小于总耗时。

### 分步耗时

| 步骤 | 中文说明 | kb_case_1 | kb_case_4 | kb_case_5 |
|---|---|---:|---:|---:|
| 1 | 归一化 query，补充 alias hints | 209.47ms | 13.80ms | 13.49ms |
| 2 | 调用当前 `.env` embedding 模型生成 query 向量 | 923.65ms | 858.88ms | 1558.43ms |
| 3 | 读取线程事实 `ThreadBusinessFact` | 7.95ms | 5.82ms | 5.23ms |
| 4 | 构造 active/有效期/业务线/知识类型过滤 SQL | 0.34ms | 0.25ms | 0.43ms |
| 5 | 解析 `query_terms`、`keyword_prefilter_terms`、intent profile | 1.00ms | 0.06ms | 0.10ms |
| 6 | 执行结构化过滤候选 `structured_filters`，只取 `chunk_id` | 6.82ms | 8.79ms | 6.14ms |
| 7 | 执行关键词预筛 `keyword_prefilter`，只取 `chunk_id` | 20.97ms | 85.65ms | 26.91ms |
| 8 | 两路候选按 `chunk_id` 合并去重 | 0.00ms | 0.04ms | 0.01ms |
| 9 | 一次性加载合并后的候选详情 | 37.58ms | 1233.75ms | 131.26ms |
| 10 | 对全部候选逐条综合打分 | 3.37ms | 80.06ms | 10.55ms |
| 11 | 排序并截取 top5 | 0.01ms | 0.17ms | 0.03ms |

结论：

- 常规样本里，KB1 的主要耗时仍是 embedding 调用，当前约 `0.86s-1.56s`。
- `kb_case_4` 修复后最大的耗时已不再是“两路各拉 500 条完整候选”，而是结构化宽筛后一次性装载 `500` 条候选详情：`1233.75ms`。
- 打分本身不是最大瓶颈。即使 `500` 条候选，综合打分也只有 `80.06ms`。
- 所以现阶段 KB1 的慢主要分成两类：普通 case 卡在第 2 步 embedding；broad case 卡在宽候选装载，而不是“top5 打分 bug”。

### 第 2 步 embedding 能否通过代码提速

可以提一部分，但边界很明确：

- 已做代码优化 1：复用 `requests.Session`，减少每次重新建连开销。
- 已做代码优化 2：对完全相同的 query 文本做 5 分钟短期 embedding cache。
- 相同 query 连续调用实测：第 1 次约 `1286.40ms`，第 2 次约 `0.01ms`，第 3 次约 `0.01ms`。

这说明：

- 如果同一条 `retrieval_query` 被重复调用，代码层缓存可以把第 2 步几乎压平。
- 但对“新的、没见过的 query”，首轮耗时仍主要取决于远端 embedding 模型本身；当前 `.env` 这台模型的热启动首轮通常还是 `0.9s-1.6s`。
- 所以只靠代码优化，不能把所有新 query 的 embedding 稳定压到 `0.5s` 内；若要继续压首轮时延，仍要换更快的 embedding 服务或更强硬件。

### 对优化的直接含义

可优化点：

- 已完成：对 `query_terms` 做停用词/高频词过滤，弱化 `客户`、`报价`、`交付` 这种泛词的 `keyword_prefilter` 影响。
- 已完成：structured 和 keyword 两路先只取 ID，合并后再一次性加载详情。
- 下一步可做：对合并后的总候选再设上限，例如先按优先级、业务线、精确短语缩到 100-200 条再打分。
- 对 `business_line=general` 且 `knowledge_type` 很宽的场景，加更细的 `knowledge_class` 或 `service_scope`。
- 给常见 query 或相同 `retrieval_query` 加短期缓存。
- 若保持当前候选策略，只换 embedding 模型只能优化一部分耗时，不能解决 case_4 这类 SQL 候选过宽问题。

## 9. 非同空间模型结果

这些模型只做了速度和失败模式观察，不建议直接上线替换，因为库内 1454 条 chunk 仍是 `qllama/bge-m3` 向量。

| 模型 | 热启动平均耗时 | 主要问题 |
|---|---:|---|
| `qwen3-embedding:0.6b` | 965.74ms | 与现有 BGE-M3 chunk embedding 空间不一致，semantic_score 接近 0 |
| `bge-large:latest` | 907.93ms | 与现有 BGE-M3 chunk embedding 空间不一致，semantic_score 接近 0 或负数 |
| `nomic-embed-text:latest` | 813.27ms | 768 维，和当前 1024 维 chunk 不匹配，semantic_score 为 0 |

结论：这些模型如果要正式评估，必须先把全量 `knowledge_chunk` 重新 embedding，并记录新的 `embedding_model` / `embedding_dim`，再跑回归测试。不能只替换 query embedding。

## 11. 结论

### 知识库1

当前代码层已经完成的低风险修复：

- `keyword_prefilter` 去掉高频泛词，避免 `客户 / 报价 / 交付 / active / ongoing` 直接把候选池打满。
- 两路候选先只取 `chunk_id`，合并后再一次性加载详情。
- embedding 调用复用 HTTP session，并给完全相同 query 加 5 分钟短期缓存。

可直接推进的候选是：

```text
EMBEDDING_API_URL=http://10.0.0.222:11434
EMBEDDING_MODEL=bge-m3-cpu:latest
EMBEDDING_DIM=1024
```

原因：

- 与现有知识库 chunk embedding 同属 BGE-M3 1024 维，测试中命中结果和分数与 `bge-m3:latest` 一致。
- 5 条样本平均中位耗时约 `1.12s`，比原 KB1 平均 `2.83s` 明显更快。
- 除候选数很大的 case_4 外，其余样本约 `0.50s-0.73s`。

需要注意：

- `bge-m3:latest` 也可用，但测试中偶发过 500；干净复测后正常，仍建议作为备选。
- `qwen3-embedding`、`bge-large`、`nomic` 不能直接替换当前 KB1 query embedding。
- 如果要评估新 embedding 模型，需要做全库重嵌和质量回归。
- 代码修复后，`kb_case_4` 已从 `653` 候选降到 `500`，总耗时从 `3844.04ms` 降到 `3068.81ms`，但 broad query 仍偏慢，下一步应继续压结构化宽筛候选。

### 知识库2

知识库2当前仍是慢环节：

- 5轮重跑平均约 `7.54s`，中位数约 `7.63s`。
- 25次中 2 次 read timeout，成功率 `92%`。
- 本系统无法通过切换 222 embedding 模型直接优化 KB2。

适合的优化方向：

- 对同一 `retrieval_query` 做短期缓存，消息不变时复用 KB2 结果。
- KB2 与 KB1 并行调用，不要串行阻塞。
- KB2 增加服务端索引/缓存/预热，目标先压到 `1s-2s`。
- 对 KB2 设置更清晰的降级策略：例如 `2s` 内无结果先返回 KB1，KB2 后补。

### 速度目标判断

知识库1使用 `10.0.0.222 / bge-m3-cpu` 后，常规样本有机会接近 `0.5s-0.8s`；但候选数过大的场景仍会到 `3s` 左右。要稳定压到 `0.5s` 内，不能只换 embedding 模型，还需要控制候选规模和缓存。

知识库2目前不具备 `0.5s` 条件，应优先做异步/缓存/超时降级。
