# 企微应用实时提醒实现方案 (V1.0)

## 1. 方案目标
本方案旨在通过企业微信自建应用，实现对客户对话的实时监听与智能化解析。通过“低延迟规则引擎”与“深度 LLM 意图分析”双轨模式，确保销售人员能够即时捕捉高风险/高价值信号，并获得结构化的客户分析支持。

---

## 2. 核心架构设计：双轨制与异步管线
根据项目技术框架，系统采用 **FastAPI + 异步任务队列 (RQ/Celery)** 的组合，确保高并发下的实时响应。

- **实时网关 (FastAPI)**: 
    - 负责企微回调验证、解密及消息下发。
    - 运行 **旁路轨道 (Fast Track)**：毫秒级正则匹配 10 类强信号并直接触发提醒。
- **异步分析管线 (Background Jobs)**:
    - 采用 **RQ/Celery** 异步处理。
    - **阶段一 (LLM-1 提取)**: 使用 `10.0.0.210` 接口，将上下文转为结构化 JSON。
    - **阶段二 (LLM-2 辅助)**: 使用 DeepSeek，结合 **pgvector** 检索知识库，生成建议回复。
- **文本上下文聚合**:
    - 仅处理文本、位置、系统事件等元数据。
    - 图片与文件识别为“即时事件类型”，不再进行下载或外部存储。

### 处理流程
1. **[实时]** FastAPI 接收消息 → 解密 → 旁路规则匹配 → 发送即时提醒。
2. **[异步]** 消息入队 → LLM-1 生成 Schema V1 摘要 → 存入 PostgreSQL。
3. **[按需]** 触发辅助回复指令 → 检索知识库 (RAG) → LLM-2 生成回复稿。

## 4. 意图轨道：LLM 结构化概述 (Schema V1)
当触发结构化分析时，LLM 必须严格按照以下字段返回结果：

| 字段 | 说明 | 填充规则 |
| :--- | :--- | :--- |
| **主线话题** | 核心业务主线 | 最多2个，无法判断填“过渡/寒暄”。 |
| **核心诉求** | 最终要达成的目标 | 指向明确诉求，不录入推测。 |
| **关键事实** | 规格、时间、钱、量 | 只要出现则必须记录，严禁幻觉。 |
| **待跟进事项** | 需要销售执行的任务 | 按优先级排序。 |
| **风险/顾虑** | 负面点/犹豫点 | 结合旁路 5, 8, 10 类信号。 |
| **待确认信息** | 模糊点/补问点 | 标记双方尚未对齐的信息。 |
| **对话状态** | 当前业务生命周期 | **枚举值**: 需求收集/报价/合同/执行/结算/售后/过渡。 |

## 5. 实现具体步骤

### 第一步：企业微信权限与凭证准备
1. **登录企微后台**: 获取 `CorpID` 和自建应用的 `Secret`、`AgentID`。
2. **配置回调接口**: 
   - 准备公网可访问的 URL (HTTPS)。
   - 设置 `Token` 和 `EncodingAESKey`。
   - 使用您提供的 `private_key.pem` 处理加密流（若涉及自定义加密）。

### 第二步：消息网关建设 (后端)
1. **签名验证**: 按照企微官方算法校验 `msg_signature`, `timestamp`, `nonce`。
2. **解密逻辑**: 将 XML/JSON 密文转为明文。
3. **消息入库**: 将消息存入 Redis 列表，每个 `ExternalUserId` 维护一个最近 20 条的消息滑动窗口。

### 第三步：并行分析执行
1. **Fast Track**: 直接对单条明文进行 Regex/Search 匹配，命中即调用 `message/send` API。
2. **Slow Track**: 
   - 触发条件：消息停顿超过 3 分钟，或对话更新满 5 轮。
   - 调用 LLM (推荐 Qwen2.5) 生成 JSON 定格式输出。

### 第四步：提醒卡片设计 (TextCard)
提醒消息应具备层次感，防止消息轰炸：
- **标题**: `[AI实时提醒] - 类别`
- **内容**: 提取的消息原文片段。
- **卡片底部**: “一键生成回复模板” 或 “查看历史全貌”。

## 6. 数据安全与隐私保障
1. **加解密规范**: 所有对话在磁盘存储前必须加密。
2. **数据隔离**: 不同销售负责的客户数据在缓存层必须隔离。
3. **合规审计**: 记录所有的 AI 推送历史，用于评估准确率。
