### 一、两版一致且经双方实测验证的结论

1. 数据画像一致: 分布 95+ =0 / 90-94=211 / 85-89=81 / 75-84=2 / <60=2; 子维度上限 low_barrier max94(mean89.3)、safety max92(mean88.7)、conciseness max95、non_repetition max95、style max93、context max94。
2. 公式覆盖已修复: 296 条中 293 条满足 overall = conc*0.18+low*0.24+nonrep*0.22+safe*0.22+style*0.08+ctx*0.06, 3 条红线硬封顶。代入子维度上限理论最高约 93.88 -> 解释实际天花板 94。
3. 核心结论一致: v16 的问题不是文案到顶, 而是评分标尺被 prompt 压死 -- low_barrier(0.24)恒<=94、safety(0.22)恒<=92, 数学上给不出 v2.0 标准 3.1 要求的"本轮最优解 95+", 211/296 挤在 90-94。
4. 交叉验证通过(Claude 实测 Codex 独有断言): 最高分样本"好嘞, 那我现在打过来?大概5分钟, 你看方便不?"确为 overall94/low_barrier94/safety92; note 三关键词(未直接回应/缺关键事实/缺具体行动)在 v16 命中数=0、overall恰82=0(故 P1-1 非本轮主因成立); S11 最高=90。Codex 依据扎实。

### 二、合并后的下一步优化项(按风险隔离)

**P0-1 [已验证] safety 评分 prompt 旧口径与 v2.0 标准 2.4 冲突 | 评分标尺 | 风险=中**
- 位置: backend/intent_engine.py 两处评分 prompt(AI候选 ~2328 + 销售原始回复 ~2395), 必须同步改。
- 现状: "普通跟进/确认/报价类给70-75; 只有规避违约/法律/品牌风险才90+" -- 与标准 2.4(口径稳的正常确认/收尾就该90-100)冲突, 是 safety 永不上95的根因。
- 改后: 口径稳/不超额/不催逼/不编数字的正常跟进确认收尾给90-100; 敏感点用"我去申请/找财务确认"不越权也可90-100; 未核实承诺/轻微贬竞品78-84; 越权让利/报低价/方式票/编造数字案例按红线降分。

**P0-2 [已验证] low_barrier 缺 95+ 高分锚点 | 评分标尺 | 风险=中**
- 位置: 同上两处。现状只有"一句话结束、没有多余客套", 无95+分档, 自封顶94。
- 改后: 客户已收尾/只需轻确认时一句干净确认不给负担=95-100; 给出明确低成本下一步(现在打给你5分钟/今天下班前给准信/发明细表对照)=95-100; 有推进但需客户多判断=85-94; 把球踢回/一次问多个=70-84。

**P0-3 [已验证] 占位符/元注释红线一致性 -- 注意区分"评分修复"与"生成修复" | 风险: 评分侧低 / 生成侧中**
- 评分侧(两版共识, Codex 主张): conciseness prompt 补红线 -- 含未填占位符(xxxx@/XX元/X折/POXX)或元注释(【...】/企微回复参考/风格要求/备注)-> conciseness<60、严重<40。消除"总分56但微信感88"的自相矛盾。仅让评分更准。
- 生成侧+链路侧(Claude 实测补充, Codex 未覆盖, 这是真正抬最低分的关键):
  - 实测真相: S09-1 模型在【企微回复参考】抬头前塞对话且编造"大连项目/制造业展会案例"(KB无据); S10-1 生成占位符邮箱 xxxx@xxx.com 且整段无抬头。split_sales_assist_output 对 S09-1 本能抽出干净正文, 但 DB 被评的是原始整段 -> caselib 评分存了未过 extract 的原文(抽取缺口)。
  - 修复: (a)改 main.py SYSTEM_PROMPT_LLM2: 强制只输出抬头+正文、禁止抬头前后追加文字、禁占位符、禁编造未在KB/上下文出现的数字与案例; (b)专项定位修复 caselib 评分存原文的链路抽取缺口。
  - 价值: 修后 S09 类干净正文可评到~90, 两个56分作为可发文案才真正消失; 仅做评分侧修复(conciseness<60)只是把坏文案打得更准, 不抬最低分。

### 三、对"能否提升质量/最低分/平均分"的合并判断

- 评分/诊断质量: P0-1/P0-2/P0-3评分侧能提升 -- 开95+通道、拉开聚集、红线一致。
- 生成质量/最低分: 仅改评分 prompt 不会抬最低分(P0-3评分侧反而让红线案例更低=正确); 真正抬最低分必须靠 P0-3 生成侧(禁占位符/编造/格式违规)+链路抽取缺口修复。
- 平均分: P0-1/P0-2 抬 safety/low_barrier 上限会小幅抬均分(~+1), 但两版共识不以涨分为成功标志, 须严格抽检防 v3 式系统性虚高。
