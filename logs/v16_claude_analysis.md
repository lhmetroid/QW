### 二、核心结论: 评分标尺被 prompt 压死在 94, 给不出"本轮最优解(95+)"

overall 为 6 子维加权和(V1 验证)。low_barrier(0.24)恒 <=94、safety(0.22)恒 <=92, 数学天花板 ~= conc95*.18+low94*.24+nonrep95*.22+safe92*.22+style93*.08+ctx94*.06 = 93.9 -> 取整 94。211/296 挤在 90-94、0 条 95+、各场景带宽普遍仅 88-93, 正是标准 0/3.1 警告的"挤在中间带、好坏不分明"。

### 三、优化项(每条带 验证状态/维度/风险/文件/前后示例)

**P0-1 [已验证-读代码] safety prompt 与 v2.0 标准 2.4 直接冲突 | 质量(标尺) | 风险=中**
- 位置: [intent_engine.py:2328](backend/intent_engine.py#L2328) 及重复块 [intent_engine.py:2395](backend/intent_engine.py#L2395)(销售原始回复评分, 两处必须同步)。
- 现状原文: "safety(稳妥度)基准: 普通跟进/确认/报价类回复给70-75分; 只有明确规避违约/法律/品牌风险的回复才可给90+, 禁止随意给100分。"
- 冲突: v2.0 2.4 规定 90-100 = 口吻稳的普通跟进/确认/收尾就该给; <50 只留给编造数字/超额定价。当前 prompt 是 v3 旧任务表(P0-3)的遗留, 与 5-22 新 holistic 标准矛盾, 是 safety mean 仅 88.7、永不上 95 的根因。
- 改后建议: "safety: 口吻稳、不超额、不催逼、不编数字的正常跟进/确认/收尾给 90-100; 未核实的具体承诺/轻微贬竞品给 78-84; 超额定价/私自方式票/越权让利给 50-77; 编造客户可见的具体数字/案例(核对 KB+上下文查无实据)给 <50。"
- 验证方法: 改后重跑 real, 核对 safety>=95 出现条数由 0 上升, 人工抽检 5-10 条比对一致性(标准 7.1)。

**P0-2 [已验证-读代码] low_barrier 缺 95+ 高分通道 | 质量(标尺) | 风险=中**
- 位置: [intent_engine.py:2330](backend/intent_engine.py#L2330) 及 [intent_engine.py:2397](backend/intent_engine.py#L2397)。
- 现状原文: "low_barrier(易回复)要求话术一句话结束, 没有多余客套。" -- 只描述合格线, 无 95+ 锚点, 模型自封顶 94。
- 改后建议: 加高分锚点 "客户已收尾/只需轻确认时一句干净确认不给负担 = 95-100; 已明确给出低门槛下一步动作 = 90-94; 需客户拿主意或一次问多个问题 = 70-84。" 对齐标准 2.2。
- 验证方法: 同上, 核对 low_barrier>=95 出现且集中在收尾/最优解语境。

**P0-3 [已验证-DB+实测, 已纠错] 两个 56 分的真实根因(非正则变体) | 生成端+链路 | 风险=中**
- 实测原文(V3):
  - S09-1: "哈哈谢谢洪大认可!刚忙完大连那个项目, 手头正好有个制造业展会的新案例, 发您参考下?【企微回复参考】对了, 您最近有展会或活动方面的需求吗?..." -- 模型在【企微回复参考】抬头前塞了对话且编造了"大连项目/制造业展会新案例"(KB 无据), 抬头本身也进了被评正文。
  - S10-1: "好的, 我的邮箱是 xxxx@xxx.com, 你直接拉我就行。" -- 占位符邮箱, 且整段无抬头。
- 实测结论(V4): 对 S09-1 原文跑 split_sales_assist_output, 能正确抽出干净正文"对了, 您最近有展会..."(抬头+编造已剥离); 但 DB 中被评 reply_text 仍是原始整段 -> 说明 caselib 评分链路评/存的是 LLM2 原始输出, 未应用 extract_reply_reference_text。对 S10-1(无抬头)split 返回无匹配, 占位符无法被剥离。
- 两类根因:
  1. 生成端: 模型违反输出格式(抬头前塞对话)+编造案例(S09)、生成占位符且不出抬头(S10) -> 需改 [main.py](backend/main.py) `SYSTEM_PROMPT_LLM2`: 强制只输出【企微回复参考】抬头+正文、禁止抬头前后追加任何文字、禁止占位符(xxxx@/XX元/x月/POXX)、禁止编造未在 KB/上下文出现的数字与案例。
  2. 链路端: caselib 评分存的是未抽取原文(reply_scores_v2 来自 [main.py:11965](backend/main.py#L11965) extension.reply_scores, 理论应为 extract 后正文, 实测为原文 -- 存在抽取缺口, 需专项追一次落点)。
- 验证方法: 改 LLM2 prompt 后重跑 real, 核对 S09/S10 这类不再出现抬头/占位符/编造; 另起只读脚本核对 caselib reply_text 是否已等于 split 后正文。

### 四、对"能否提升最低分/平均分"的判断(基于以上验证)

- 平均分: 大概率小幅升(~+1), 主因 P0-1 safety 回归 v2.0。但标准 14 节明确不以涨分为成功标志(防 v3 式虚高), 须抽检确认所抬分确实该给。
- 最低分: 仅改评分 prompt(P0-1/2) 不会升, P0-3 红线评分修复反而让其更低(正确)。真正抬高最低分要靠 P0-3 的生成端修复(禁占位符/编造/格式违规)+链路抽取缺口修复 -- 修后 S09 类干净正文可评到 ~90, 56 消失。
- 真实目标: 让 95+ 出现、红线下沉, 分布展开度+抽检一致性为准, 而非均分曲线。

### 五、下一版 v17 落地顺序(隔离风险, 标准 13.2)

1. 低风险先行: 单独版改评分 prompt 两处(P0-1/P0-2), 不碰生产链路 -> A/B 看 95+ 是否出现、公式仍自洽。
2. 中风险隔离: 单独版改 SYSTEM_PROMPT_LLM2(P0-3 生成端), 因其走共享 sidebar_assist 生产链路, 必须独立 A/B 不与评分改动混版。
3. 链路缺口: 专项定位并修复 caselib 评分存原文问题, 只读核对修后 reply_text=split 后正文。
4. 高风险押后: P1-1 封顶逻辑改语义须单独 A/B; P1-2 补 KB 只读。

### 六、对 Codex v16 结论的交叉验证(Claude Code 实测追加, 2026-05-25)

复核方式: 不轻信文字, 对 Codex 独有的可证伪断言逐条跑 DB/正则实测。

**总结论: Codex 的结论与依据总体正确、且经得起独立复算; 对"评分质量/诊断质量"有明确帮助, 但对"生成质量/最低分"帮助有限 -- 这是它与本单元格 P0-3 的关键差异, 须补齐。**

实测核验(全部通过):
1. [已复算] 数据画像、公式 293/296 自洽、子维度上限(low94/safe92/conc95/nonrep95/style93/ctx94)代入理论最高 93.88 -> 解释实际最高 94。与 Codex 一致。
2. [已实测] Codex 引用的最高分样本属实: "好嘞, 那我现在打过来?大概5分钟, 你看方便不?" overall=94、low_barrier=94、safety=92, 逐字命中。
3. [已实测] Codex 称 note 三关键词(未直接回应/缺关键事实/缺具体行动)在 v16 命中数=0, overall 恰为82 的=0 -> 属实, 故 P1-1 封顶逻辑确非本轮压顶主因, Codex"不优先动"的判断正确。
4. [已实测] S11 最高分=90 属实。
5. [一致] P0-1 safety 旧口径冲突、P0-2 low_barrier 缺 95+ 锚点、P0-3 占位符/元注释子维度不一致 -- 三项与本单元格独立结论一致。

价值与局限:
- 有帮助(评分/诊断质量): P0-1/P0-2 给 safety/low_barrier 开 95+ 通道, 能把挤在 90-94 的最优解拉开、提升区分度与可解释性; P0-3(子维度补红线)提升红线一致性, 消除"总分56但微信感88"的自相矛盾。这些都让标尺更准。
- 局限(生成质量/最低分): Codex 的 P0-1/P0-2/P0-3 全部是评分 prompt 改动, 只改"怎么打分", 不改"怎么生成"。其 P0-3 仅把占位符/注释样本的 conciseness 打到 <60(评分更准), 但不阻止 LLM2 继续吐占位符/编造/格式违规 -- 因此不会真正抬高最低分, 两个 56 分作为可发文案依旧是坏的。Codex 未提 SYSTEM_PROMPT_LLM2 生成端修复, 也未提 caselib 评分存原文(未过 extract)的链路抽取缺口。
- 风险提示: P0-1/P0-2 抬 safety/low_barrier 上限会抬均分, 若不配生成端改进且抽检不严, 有重演 v3 式系统性虚高的风险。Codex 已要求抽检, 方向对。

结论落点: 采纳 Codex 的 P0-1/P0-2/P0-3(评分标尺修正, 提升诊断质量), 但要真正提升生成质量与最低分, 必须叠加本单元格 P0-3 的两项 Codex 未覆盖的修复 -- (a)改 SYSTEM_PROMPT_LLM2 禁占位符/编造/抬头前后追加文字; (b)修 caselib 评分链路抽取缺口。两类改动按第五节顺序隔离 A/B, 不混版。
