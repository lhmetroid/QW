为了让多个 AI 工具之间能“接力、恢复、验证、重试”而人为约定的**项目状态文件**。

你可以理解为：

```
Codex / Hermes / Claude Code / Antigravity 的对话会断但项目目录里的 markdown 文件不会断
```

所以我们把任务状态写到文件里，让任何一个工具都能继续。

---

## 这几个文件分别是什么作用？

### 1）`TASKS.md`：任务清单

作用：记录大任务拆成哪些小任务，哪些完成了，哪些未完成。

例如 30 页项目文档开发，可以拆成：

```
# TASKS## 当前大目标完成 v13 项目文档开发、验证和优化。## 任务清单- [x] Task 1：整理现有项目结构和数据来源- [x] Task 2：修复 v13 测试提速方案中的已知问题- [ ] Task 3：补充 CRM 邮件读取和联系人行业识别逻辑- [ ] Task 4：生成完整验证报告- [ ] Task 5：最终复查、清理临时文件、更新项目进展
```

Hermes 读取它后，就知道：

```
第一个未完成任务是 Task 3下一步应该调用 Codex CLI 继续 Task 3
```

---

### 2）`PROGRESS.md`：当前进度

作用：记录“现在做到哪一步”“最后一次成功在哪里”“当前是否卡住”。

它比 `TASKS.md` 更细，适合恢复中断。

例如：

```
# PROGRESS## 当前状态正在执行 Task 3：补充 CRM 邮件读取和联系人行业识别逻辑。## 已完成- 已检查 CRM 数据表结构- 已生成部分测试脚本- 已定位邮件数据来源字段## 当前卡点Codex 上次执行到 `scratch/find_nonempty_crm_email_tables.py` 时等待授权。## 下一步继续运行检查脚本，确认哪些表包含有效邮件数据。## 最近验证尚未完成最终验证。
```

Hermes 恢复时会先看这里，而不是完全依赖上一次聊天记录。

---

### 3）`TASK_HANDOFF.md`：交接文件

作用：当一个工具做到一半，换另一个工具继续时，用它交接上下文。

比如你在 VS Code Codex 插件里已经做了一半，想让 Hermes + Codex CLI 继续，就让插件先生成：

```
# TASK_HANDOFF## 当前目标继续修复 v13 测试提速方案，补全 CRM 邮件读取和联系人行业识别逻辑。## 已完成- 修改了 inspect_pg_sqlserver_mails.py- 修改了 inspect_crm_usr_email.py- 新增 find_nonempty_crm_email_tables.py- 已生成部分轮询结果指南## 未完成- 还未确认最终 CRM 邮件字段来源- 还未重新生成完整报告- 还未运行最终验证## 已修改文件- scratch/inspect_pg_sqlserver_mails.py- scratch/inspect_crm_usr_email.py- scratch/find_nonempty_crm_email_tables.py- 查看每轮详细结果指南.md## 下一步建议1. 继续运行 CRM 检查脚本2. 确认有效邮件字段3. 更新项目进展文档4. 运行最终验证命令## 注意事项不要删除现有数据文件；不要覆盖人工整理的 md 文档。
```

它的作用相当于：

```
把 VS Code 插件/Antigravity/Codex 桌面版里的对话状态导出成文件让 Hermes 可以读取并继续
```

---

### 4）`VALIDATION.md`：验证命令

作用：告诉 Hermes 和 Codex：任务做完后要怎么检查是否成功。

例如 Node 项目：

```
# VALIDATION## 必须执行```bashgit diff --checknpm testnpm run lintnpm run build
```

## 通过标准

- 命令全部返回 0
- 没有新增明显错误
- 关键输出文件已生成

```
如果是 Python 项目：```markdown# VALIDATION## 必须执行```bashgit diff --checkpython -m pytestpython build_521_questions.py
```

## 通过标准

- pytest 通过
- 目标 HTML / JSON / MD 文件生成
- 输出数量符合预期

```
Hermes 完成任务后会按这里跑验证，而不是随便判断“看起来完成了”。---## 这些文件没有怎么办？没有也可以，但建议创建。如果没有，Hermes 只能靠：```text当前聊天git diff目录结构文件名猜测
```

这样恢复能力会弱很多。

建议你在项目根目录执行：

```
touch TASKS.md PROGRESS.md TASK_HANDOFF.md VALIDATION.md
```

如果还没有通用规则文件，也建议加：

```
touch AGENTS.md
```

---

## `AGENTS.md` 要不要？

建议要。

`AGENTS.md` 是给所有 AI Agent 看的项目规则文件，比 `TASKS.md` 更基础。

作用是规定：

```
这个项目怎么运行哪些文件不能动改代码后必须验证什么遇到授权怎么办哪些命令危险任务完成后怎么记录
```

例如：

```
# AGENTS## 项目规则- 所有 AI 工具必须先阅读 TASKS.md、PROGRESS.md、TASK_HANDOFF.md、VALIDATION.md。- 每次只处理一个小任务。- 修改代码前先查看 git status。- 修改完成后必须运行 VALIDATION.md 中的验证命令。- 每完成一个任务，必须更新 PROGRESS.md。- 任务中断、失败、token 不足、需要人工授权时，必须更新 TASK_HANDOFF.md。- 不允许删除大量文件。- 不允许修改生产环境配置。- 不允许提交、推送远程仓库，除非用户明确要求。
```

这个文件很重要，建议以后所有项目都统一放。

---

## 以后都统一用这些文件名吗？

**建议统一用这些文件名。**

原因：

```
1. Hermes / Codex / Claude Code / Antigravity 都容易理解2. 每个项目结构一致，后续好维护3. 中断恢复时不用重新解释4. 方便你复制同一套工作流
```

推荐标准文件名：

```
AGENTS.md          项目规则 / AI 操作规范TASKS.md           任务清单PROGRESS.md        当前进度TASK_HANDOFF.md    交接上下文VALIDATION.md      验证命令和通过标准
```

如果你已有类似文件，也可以不重复创建，但建议逐步统一。

例如你现在项目里可能已有：

```
项目进展.md预处理相关.md知识库最终方案.md查看每轮详细结果指南.md
```

这些可以继续保留。  
但建议新增一个统一入口：

```
TASK_HANDOFF.md
```

里面可以引用原文件：

```
# TASK_HANDOFF## 相关文档- 项目进展.md- 查看每轮详细结果指南.md- 知识库最终方案.md## 当前应继续的任务见 TASKS.md 中第一个未完成任务。
```

## 推荐你现在直接创建一套

在项目根目录，比如：

```
cd /mnt/d/items/QW
```

```

```

## 通过标准

- 命令返回 0
- 没有明显语法错误
- 关键输出文件存在

```
---## 以后给 Hermes 的提示可以统一成这样```text请加载 codex skill。当前目录是 Git 项目。请先读取：- AGENTS.md- TASKS.md- PROGRESS.md- TASK_HANDOFF.md- VALIDATION.md- git status- git diff你作为任务主管，通过 Codex CLI 执行任务，不要控制 Windows 的 Codex 插件或桌面版。请继续 TASKS.md 中第一个未完成任务。每次只处理一个小任务。完成后运行 VALIDATION.md 中的验证命令。如果验证通过，更新 TASKS.md、PROGRESS.md、TASK_HANDOFF.md。如果失败、token 不足、需要人工授权或登录失败，停止并记录原因。
```

结论：**这些文件不是强制要求，但强烈建议以后统一使用。它们是 Hermes 控制 Codex CLI 长任务时的“状态存储层”和“恢复依据”。**
