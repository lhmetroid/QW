# 企业微信侧边栏 - 完整技术框架设计

---

## 1. 整体技术方向

### 前端形态：H5 页面（而非小程序）

**理由：**
- **开发效率高**：React + 现有 Web 技术栈，代码复用率高，1-2 人能快速交付
- **侧边栏适配好**：H5 天然支持响应式布局，侧边栏宽度（300-400px）兼容性好
- **迭代成本低**：H5 可以直接上传 CDN，无需发版、审核，快速迭代和 A/B 测试
- **和老系统对接灵活**：H5 通过 REST API 调用后端，不受小程序沙箱限制
- **未来迁移简单**：第三方应用时，H5 页面可以直接复用，只改后端接入层

**小程序不选的原因：**
- 侧边栏小程序在企微中的形态和权限有限制
- 需要发版审核，迭代周期长
- 小程序容易出现权限和上下文获取不稳定的问题
- 技术栈切换（如果团队有 H5 基础）

---

### 后端应用形态：企业微信自建应用（而非第三方应用）

**理由：**
- **当前目标是内部验证**：不需要立即承担第三方应用的商业化和授权管理复杂度
- **上线速度快**：自建应用配置简单，无需 ISV 预授权、永久授权码等流程
- **客户联系和权限清晰**：自建应用在企业内安装，客户联系数据获取稳定
- **快速反馈**：内部验证产品是否高频、是否对销售有帮助

**为什么不现在就做第三方应用：**
- 第三方应用需要完整的授权体系（预授权、永久授权码、token 管理）
- 需要多租户隔离设计和数据安全认证
- 需要提供服务商后台和企业配置面板
- 这些工作量大，会严重拖累内部验证的进度

---

## 2. 两个选择之间的关系

```
阶段 1（现在）：自建应用验证         阶段 2（未来）：第三方应用商用
─────────────────────────────      ──────────────────────────────

企业微信自建应用                     企业微信第三方应用
  ↓ 调用                             ↓ 调用
H5 页面（React）                      H5 页面（React）← 复用
  ↓ 调用                             ↓ 调用
[企微接入层] ← 需要改造               [新的企微接入层] ← 改造点
  ↓ 调用                             ↓ 调用
[业务服务层] ← 保持不变               [业务服务层] ← 复用
  ↓ 调用                             ↓ 调用
PostgreSQL（多租户 schema）           PostgreSQL（多租户 schema）← 复用
  ↓ 调用                             ↓ 调用
内部 CRM + AI 服务 API               内部 CRM + AI 服务 API
```

**关键点：**
- H5 页面结构 ✅ 阶段 1-2 都用
- 业务逻辑层 ✅ 阶段 1-2 都用
- 数据库 schema ✅ 阶段 1-2 都用（多租户设计）
- 企微接入层 ⚠️ 阶段 2 需要改造（token 来源、授权流程）

## 3. 后端分层设计（为迁移预留）

```
阶段 1 - 自建应用：

企微自建应用 Token（固定）
         ↓
┌──────────────────────────┐
│ wecom_adapter/           │
├──────────────────────────┤
│ • Token 管理（自建应用）  │
│ • JS-SDK 签名            │
│ • 上下文解析             │
│ • 企微 API 调用          │
└──────────┬───────────────┘
           ↓
┌──────────────────────────┐
│ business_service/        │
├──────────────────────────┤
│ • 客户信息               │
│ • 商机数据               │
│ • AI 摘要与回复          │
│ • 跟进记录               │
└──────────┬───────────────┘
           ↓
  CRM + AI 服务 API


阶段 2 - 第三方应用：

第三方应用授权 Token（来自 permanent_code）
         ↓
┌──────────────────────────┐
│ wecom_adapter/           │
├──────────────────────────┤
│ • Token 管理（第三方）    │  ← 改这部分
│ • JS-SDK 签名            │
│ • 上下文解析             │
│ • 企微 API 调用          │
└──────────┬───────────────┘
           ↓
┌──────────────────────────┐
│ business_service/        │
├──────────────────────────┤
│ • 客户信息               │  ← 不变
│ • 商机数据               │
│ • AI 摘要与回复          │
│ • 跟进记录               │
└──────────┬───────────────┘
           ↓
  CRM + AI 服务 API
```

### wecom_adapter（企微接入层）

**职责**：隔离所有企微相关逻辑
- Token 管理（缓存、刷新）
- JS-SDK 签名接口
- 上下文解析（成员身份、客户 ID）
- 企微 API 调用
- 权限验证

**为什么分离**：第三方应用阶段只需改这层，业务逻辑层完全不感知企微接入方式的改变

### business_service（业务逻辑层）

**职责**：与企微无关的业务逻辑
- customer_service：客户信息、ID 映射
- opportunity_service：商机查询
- summary_service：AI 摘要（调用 AI API）
- reply_service：推荐回复（调用 AI API）
- followup_service：跟进记录
- audit_service：审计日志

**为什么分离**：这层可被多个入口调用（H5、PC 页面等），第三方应用阶段完全复用，无需修改

## 4. 多租户数据库设计（支撑迁移）

当前阶段虽然只服务一个公司，但数据库设计**必须支持多租户**：

```sql
-- 所有表都有这两个字段
CREATE TABLE customers (
    id BIGINT PRIMARY KEY,
    tenant_id VARCHAR(50) NOT NULL,    -- 当前内部应用 = 公司 ID
    corp_id VARCHAR(50) NOT NULL,      -- 企微 corp_id
    external_user_id VARCHAR(100),
    crm_customer_id VARCHAR(100),
    ...
    UNIQUE(tenant_id, external_user_id),
    INDEX(tenant_id)
);
```

**好处：**
- 当前：tenant_id 就是公司 ID，查询时加 WHERE tenant_id = '固定值'
- 未来：tenant_id 从 API 请求中动态读取，自动支持多企业

## 5. 技术栈总结

| 层 | 技术 | 为什么选它 |
|----|------|----------|
| **前端** | React + Ant Design | H5 开发快，能快速验证侧边栏可用性 |
| **后端** | Python FastAPI | 轻量、高效，1-2 人能快速开发和维护 |
| **数据库** | PostgreSQL | 支持多租户隔离、复杂查询、JSON 字段 |
| **部署** | H5 CDN + FastAPI 云服务 | H5 快速迭代，后端可弹性扩展 |

## 6. API 接口规范

**统一响应格式**：

```json
{
  "code": 0,
  "message": "success",
  "data": {...}
}
```

**核心 API 清单**：

| 端点 | 方法 | 说明 |
|------|------|------|
| `/api/wecom/jsapi-ticket` | GET | 前端获取 JS-SDK 签名票据 |
| `/api/customer/{customer_id}` | GET | 获取客户详情 |
| `/api/customer/{customer_id}/summary` | GET | 获取 AI 摘要 |
| `/api/customer/{customer_id}/reply` | GET | 获取推荐回复 |
| `/api/customer/{customer_id}/opportunity` | GET | 获取商机信息 |
| `/api/customer/{customer_id}/followup` | GET | 获取跟进记录 |
| `/api/customer/{customer_id}/followup` | POST | 新增跟进记录 |

## 7. 关键技术决策

| 决策 | 选择 | 理由 |
|------|------|------|
| **认证方式** | 企微 JS-SDK 签名 + 简单 Token | 不用 OAuth 复杂流程，H5 直接调用后端 API |
| **缓存策略** | Redis（可选）或数据库查询 | 1-2 人项目，先不加 Redis，DB 直接支持 |
| **异步任务** | Celery（可选）或同步调用 | 先做同步，等性能瓶颈再加异步 |
| **日志审计** | 数据库表 audit_logs | 简单有效，满足审计需求 |
| **监控告警** | 日志 + 简单指标 | 用公司现有监控系统 |

---

## 8. 迁移到第三方应用的改造点

| 改造点 | 当前状态 | 第三方应用状态 |
|--------|---------|-----------------|
| **Token 来源** | 固定企微自建应用 token | 改为从授权企业的 permanent_code 换取 |
| **tenant_id** | 硬编码或配置中指定公司 ID | 从 API 请求中提取（基于 corp_id） |
| **权限范围** | 自建应用可见范围 | 第三方应用授权范围 |
| **wecom_adapter/** | 当前实现 | 改造 token 管理和授权流程 |
| **business_service/** | 完全复用 | 无需改动 |
