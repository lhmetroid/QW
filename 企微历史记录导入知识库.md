# 企微历史记录导入知识库

更新时间：2026-05-15

## 0. 执行基准

本文档是“企微历史记录导入知识库”的当前执行基准。后续可以按新要求迭代，但没有新要求时，预处理、页面展示、字段入库、质量分和发布判断都必须严格按本文口径执行，不再回到旧的单行判断、旧阶段/旧环节或“纯推广一律不入库”的逻辑。

本文保留用户原始要求中的关键表达；为避免执行歧义，部分内容已改写为更标准的字段定义、流程和边界说明，但不得删减原始目标：将已有历史企微中有价值、可检索、可复用、不会误导的内容整理进入知识库。

## 1. 目标

将企微历史会话数据整理成可检索、可复用、可审核、可发布的销售知识库数据。

核心目标不是保存聊天记录本身，而是从已有历史企微对话中抽取“以后销售在相同场景下能检索到、能参考、不会误导”的知识单元。

最终入库结果需要满足：

1. 可准确检索：标题、商业阶段、商业场景、服务类型、语种、客户类型等字段能帮助检索定位。
2. 可复用：内容应是通用话术、有效处理方式、业务知识、可迁移案例或可归纳经验。
3. 可追溯：保留客户 ID、销售 ID、时间、来源批次、原始行号或原始消息范围，不弄乱来源。
4. 可审核：预处理只生成候选和判断结果，正式进入知识库前需要人工复核。
5. 可治理：低复用、强个案、纯确认、乱码、无问无答、无上下文价值的内容不应污染正式知识库。纯推广、群发模板若可复用且不误导，可以作为话术或模板候选入库。

## 1.1 当前企微历史入库总逻辑

当前流程必须分成四步，后续测试、LLM 批处理和人工复核都按这个顺序执行：

1. 数据导入：企微原始历史数据已经进入 `wecom_raw_import`，原始消息、客户 ID、销售 ID、时间、批次等信息必须保留，不在此步做入库判断。
2. 切片生成：把企微消息拆成对话组，也就是问答切片。切片是后续最小处理单位，不是颜色本身，但颜色用于辅助识别切片边界。客户多行问、销售多行答在此步合并完成。每个切片必须有唯一切片号；同一切片内多行共享同一个切片号，不同切片不能重复。
3. 入知识库预处理：对第 2 步形成的切片整体做语义理解，生成知识标题、完整知识切片、入库正文、商业阶段、商业场景、服务类型、语种、质量分、处理结论、命中依据、使用边界、发布目标等字段。此步严禁回到逐行处理；逐行只用于追溯原始消息，不作为知识库判断单位。
4. 质量分和人工筛选：第 3 步应已完成所有字段值和处理结论。人工复核只判断是否入库、是否调整质量分、是否拆分或合并异常切片、是否发布到正式知识库；不是重新按每行做字段生成。

当前处于第 3 步测试阶段：字段值暂由人工/本次测试归纳完成，后续应由 LLM 按本文边界批量生成。每个字段值都必须服务于知识库检索、复用、审核或发布，不能只是对原始内容的简单描述。

## 1.2 强制执行：切片号先冻结，再做入库预处理

为避免再次出现“第 3 步预处理时跨切片、重切片、逐行判断”的偏差，后续一律按以下死规则执行：

1. 第 2 步先生成固定切片号 `slice_id`，这是后续所有处理的硬约束字段。
2. 第 3 步只允许读取同一个 `slice_id` 内的原始行，并只允许回写该 `slice_id` 内的预处理字段。
3. 第 3 步不得新增、删除、拆分、合并或改写 `slice_id`；不得把相邻切片合并成一个知识切片；不得把一个 `slice_id` 内的字段写到另一个 `slice_id`。
4. 若人工或程序发现切片边界错误，必须退回第 2 步修正切片号；修正完成后，再重新执行第 3 步。不能在第 3 步边预处理边修边界。
5. 前端必须显示 `slice_id`，人工复核时先确认当前行所属切片号，再看入库字段。没有切片号的行只能处于“待切片”状态，不能进入入库预处理。
6. 自动预处理任务的输入单位必须是“一个 `slice_id` 对应的全部原始行”，不是单行、不是颜色块、不是 `group_key`、不是当前页面视觉连续块。
7. 自动预处理任务的输出必须一次性写回该 `slice_id` 下所有行，且同一 `slice_id` 内所有切片级字段完全一致。
8. 商业场景不是切片号。同一个商业场景可以包含多个独立切片；例如同属“转介绍请求与存量裂变”的对话，可以拆成“通过联系人验证后的首句问候”“客户提供转介绍联系人后的感谢承接”“客户简单确认不入库”等多个切片。不得因为商业阶段、商业场景、质量分或处理结论相近，就把不同问答组写成同一个 `slice_id`。

执行顺序固定为：

```text
原始行 -> 第2步生成/确认 slice_id -> 锁定 slice_id -> 第3步按 slice_id 生成入库字段 -> 质量分/人工复核 -> 发布
```

例外只允许发生在第 2 步：例如发现 23-29 不应是一个切片，必须先拆成 23-24、25-27、28-29 三个 `slice_id`；拆完后，第 3 步分别对三个切片处理，不能再跨切片取问、取答或合并正文。

## 2. 原始需求依据

本节来自 `项目进展.md:8939-8975` 的补充原始需求，作为当前方案必须遵守的基础口径。

原始要求包括：

1. 当前企微全部对话数据需要被抽取和分析。
2. 需要先通过代码和 `项目进展.md` 理解当前知识库逻辑、各类知识库数据要求及格式。
3. 最终目的是整理出好的、可用的、能准确检索的知识库数据入库。
4. 初步构想是按环节入库，每轮作为切片；一轮一般是一问一答，始终以客户作为问、销售作为答。
5. 环节示例包括报价、咨询产品、售后等。
6. 知识库字段需要整理清楚，例如服务/产品、语种等，并额外增加阶段字段：开发、商机跟进、合同跟进、售后、财务等。
7. 需要明确找不到答、找不到问的数据对如何处理。
8. 企微聊天可能一个问题多行发送，需要合并；但时间跨度较长、跨天、已出现结束性质语句时不能简单合并。
9. 合并仅限连续客户消息或连续销售消息，中间插入另一方则不合并；上一个环节已断开也不合并。
10. 涉及引用时，引用可跨行形成新问答，但要避免重复；原始问不应再重复绑定其他答。
11. 客户引用销售原文并明确回答我方问题时，需要判断是否形成新的切片；若只是“好的”这类简单确认，则不入库。
12. 示例中：
    - “客户问能否提供阿拉伯语翻译 + 销售说明可以并追问方向”作为第一个切片。
    - “客户引用销售并回答阿拉伯译中 + 销售进一步承接报价”作为第二个切片。
    - 最后“好的”只是简单确认，不作为数据对入库。
13. 上述示例对应新增字段：阶段为商机跟进，标题可为“阿拉伯语译中翻译的报价沟通”。
14. 示例仅供参考，实际处理需结合真实数据和已有代码补充所有情况。
15. 数据中的客户 ID、销售 ID、时间等信息不得弄乱。
16. 下一步整理出的数据组之间要有明确间隔，便于区分。
17. 企微中出现的人名、文件名、乱码、图片、视频等需要统一标准化；以可通用知识库数据为标准，其他内容文字不得随意改写。
18. 整个过程必须避免中文乱码。

## 4. 关键修订结论

### 4.1 切片是最小整体，不是单行

一个切片是知识库最小整体，通常由客户方向和销售方向组成。

切片号是执行约束字段，对应 `wecom_raw_import.slice_id`：

1. 每个切片只有一个切片号。
2. 同一切片内所有原始行必须共享同一个切片号。
3. 不同切片的切片号不能重复。
4. 入库预处理、质量分、处理结论、发布状态、知识标题、完整知识切片、商业阶段、商业场景、命中依据和使用边界都必须按切片号聚合生成。
5. 若同一切片内不同行出现不同值，应视为数据错误或旧逻辑残留，必须先修正切片号或切片级字段，再做入库判断。

例如当前前 6 行：

1. 行 1-3 是一个切片候选。
2. 行 4-6 是另一个切片候选。
3. 行 7 属于同一大环节下的客户简单确认或结束性信息，但没有可用销售答复，不入库。
4. 行 8 开始若语义进入新的主动触达、关系维护、转介绍或老客唤醒场景，应作为新的知识库候选单元开始；即使当前可视块只有销售单向消息，也不能仅因“无客户问”自动不入库。

同一切片内的多行必须共享同一个质量分、处理结论、发布状态和切片号。行级字段可以显示，但不能把行级判断当作最终知识库单元。

紫色底色的“仅销售消息”属于特殊企微切片类型：销售主动触达模板。它不是标准客户问、销售答的问答切片，但可以构造“触发场景/适用条件”作为触发字段，例如“长期未互动客户需要轻量唤醒”“节后开工需要低压力问候”“希望客户转介绍潜在需求方”。同一切片内连续或多行销售消息合并为参考回复，必要时做脱敏和轻量标准化；拟发布到知识库的正文对齐正式知识切片正文。

这类切片是否入库，只看可复用性、检索价值、是否会误导和质量分，不再因为没有客户显性提问直接归为 `不入库_无问`。

例外情况：

如果可明确判断某一行只是对上文的简单确认，例如“好的、明白、收到”，而后续行与下一句组成完整问答，则可以排除该简单确认，把后续问答作为切片。但这涉及复杂语义理解，当前阶段不作为自动处理重点，优先人工复核。

### 4.2 处理备注的定义

`process_note` 不是“解读客户的话”，而是“预处理备注”。

它应说明：

1. 该行属于哪个切片候选。
2. 该切片为何入库或不入库。
3. 若不入库，原因是无答、无问、纯确认、乱码、强个案、时效性过强、复用价值低、无上下文价值等。
4. 若入库，说明该切片的可复用价值，例如“客户忙时的改约处理”“客户未收到邮件时的处理方式”“客户拒绝礼品需求后的转向话术”。
5. 若只是切片内组成部分，备注应标明“同切片，随切片统一判断”，不能让同一切片内问和答出现互相矛盾的结论。

### 4.3 质量分评价标准

质量分应评价“切片作为知识库被检索和引用时是否有价值”，不是评价单句是否看起来完整。

建议按 0-100 评分：

1. 90-100：高度通用、可直接作为标准话术或强业务规则；检索出来通常不会误导。
2. 80-89：高价值，客户问题和销售回应都清晰，有明确复用场景。
3. 70-79：有入库价值，但需要标题、字段或备注限定适用边界。
4. 60-69：弱参考，可能适合候选池或人工参考，不宜直接发布为强知识。
5. 40-59：上下文过强、个案化明显、容易误导，默认不入正式知识库。
6. 0-39：无信息量、纯确认、乱码、无问无答、无上下文价值，不入库。纯推广和群发模板不能自动低分，应看是否可复用、可检索、不会误导。

行 1-3 “客户忙，不便通话，销售提出下午 2 点再联系”是通用场景处理，有入库价值。它不应因为客户话短而低估，也不应只给销售行单独入库。

行 4-6 “客户问来意 / 指引联系孔经理 / 销售说明多次联系孔经理但未接通”过于具体，且可能误导后续检索结果，因为不是每个联系人电话都打不通。该切片应判为 `不入库_强个案`，质量分不高于 45；不得进入 `入库_切片` 或 `待人工确认`，也不应高于行 1-3。

### 4.4 商业阶段和商业场景必须按语义判断

不能因为某个联系人会话存档中的前几条消息是当前数据开头，就简单判断为“初次接触”。

原因：

1. 企微会话存档并不一定从真正第一次沟通开始。
2. 早期历史记录可能因未开启会话存档缺失。
3. 老客户新加企微或重新加企微时，数据开头也可能不是初次接触。
4. 商业阶段和商业场景应依据语义、客户关系、上下文信号、已有客户历史和场景定义判断。

商业场景的主场景判断以“主要触发语义和主要销售动作”为准：

1. 如果消息明显以节日、新年、开工、祝福问候作为触发，后面顺带带出业务介绍或轻量探需，主场景优先归为 `基础跟进 / 节气/开工问候 / greetings`。
2. 如果核心动作是客户刚添加、首次建立服务认知、销售自我介绍并系统说明服务范围，且没有明显节日问候触发，才归为 `基础跟进 / 新客激活 / lead_activation`。
3. 如果核心动作是沉默老客、长期未服务客户的重新激活，归为 `基础跟进 / 老客唤醒 / re_activation`。
4. 如果内容无法识别、未知消息类型、只有无意义占位，不应强行归入任何商业阶段或商业场景；前端应显示为空或无法识别，不回退到旧 `stage/section` 造成误导。

结合当前前 30 行样例补充：

1. 页面第 13 行虽然包含自我介绍、公司能力和轻量探需，但开头“新年好、开工大吉、祝团队业绩长虹”是主触发，后续业务介绍只是节日问候后的自然承接，因此应归为 `基础跟进 / 节气/开工问候 / greetings`，不归为 `新客激活`。
2. 页面第 14 行为“未知消息类型”，没有可识别客户问、销售答、销售动作或业务语义，应归为无意义或无法识别内容，处理结论为不入库，商业阶段、商业场景和场景编码均留空。
3. 商业场景不是按消息位置、底色或是否第一条判断，而是按切片整体语义判断；销售主动触达、节日问候、案例分享、转介绍请求都可以形成候选知识切片，但必须有明确复用边界。

当前商业场景判断应优先参考 `llm_business_scenario_standards` 表：

1. 核心表：`llm_business_scenario_standards`。
2. 关键字段：`category`、`scenario`、`scenario_full`、`scene_value`、`features`、`strategy`。
3. 用途：作为 22 个商业场景规范和语义特征依据，辅助判断当前对话属于哪个商业阶段和商业场景。

### 4.5 系统验证消息可以作为切片的问

类似行 18 “我通过了你的联系人验证请求，现在我们可以开始聊天了”虽然是系统消息，但在企微历史导入知识库场景中，它实际代表客户同意添加后的起点。

如果后续销售第一句是有效开发话术，则该系统消息可以作为切片的“问/触发语”参与入库。

原因：

1. 本项目目标是把已有历史企微中有价值内容进入知识库。
2. 问答切片中的“问”、标题、环节等都是检索要素。
3. 如果删掉触发语，只保留销售话术，会影响后续检索定位。

处理原则：

1. 系统验证消息本身不作为独立知识入库。
2. 它可与销售首次有效问候/开发话术组成切片候选。
3. 标题应体现“客户同意添加后的首次触达”或具体业务触达场景。
4. 若销售后续无上下文价值，则不入库。纯推广、群发模板本身可能有入库价值，是否入库应按可复用性、检索价值和是否会误导来判断。

### 4.6 紫色底色销售单向消息的入库口径

前端紫色底色表示当前可视块只有销售消息、没有客户显性回复。该颜色只是提示“不是标准问答块”，不是“不入库”的判断。

处理原则：

1. 有复用价值的销售单向触达应作为“企微主动触达模板”进入候选或正式知识库。
2. 这类切片的“问”可以是虚拟触发语/适用场景，不需要伪造客户原话。
3. “参考回复 / 销售答”使用同一切片内销售方向的实际话术；如销售连续多行发送、跨行补充、同一模板分多条发出，必须按原顺序合并到同一个 `slice_answer` 字段，再做必要脱敏、去具体联系方式和标准化表达。
4. 商业场景按主要销售意图判断，常见包括：
   - 老客户或长期未互动客户唤醒：`基础跟进 / 老客唤醒 / re_activation`。
   - 节日、新年、开工问候：`基础跟进 / 节气/开工问候 / greetings`。
   - 转介绍请求：`客情经营 / 转介绍请求与存量裂变 / referral`。
   - 案例或视频号案例分享：若主要目的仍是唤醒老客户，可归入老客唤醒；若检索目标是找案例话术，可归入 `价值呈现 / 案例分享 / case_sharing`。
   - 公司能力或服务范围介绍：若用于激活沉默客户，可归入老客唤醒；若用于能力说明，可归入 `价值呈现 / 业务深度介绍 / service_intro`。
5. 是否入库按质量分判断：高质量直接作为模板切片，边界不清或素材缺失进入待人工确认，明显无上下文价值、乱码、过期强时效内容才不入库。
6. `不入库_无问` 只用于既无法构造有效触发场景、又没有复用价值的单向内容；不能用于排除所有紫色底色消息。

### 4.7 详情侧栏应展示入知识库详细信息

当前点击“详情”显示完整 `group_key` 对话组，只适合查看原始上下文，不符合“入知识库详情”的目标。

目标详情侧栏应先展示“拟发布知识结构”，再展示“预处理检索字段”。

拟发布知识结构应尽量对齐现有知识库详情：

1. 知识标题：对应 `knowledge_document.title` 或 `knowledge_chunk.title`。
2. 知识分类：例如企微回复模板、企微候选模板、案例、能力知识等。
3. 服务：对应正式知识库服务字段。
4. 商业阶段 / 商业场景：用于检索加权和人工审核。
5. 知识切片：对应一个完整的入库最小单元，必须把触发场景/客户问、参考回复/销售答、使用边界作为一个整体展示和审核；字段可以拆分存储，但页面主展示不能把问和答表现成两个独立知识。
6. 入库正文：对应未来 `knowledge_chunk.content`，显示销售可复用的话术、模板、案例正文或规则正文。它是知识切片中的正文部分，不等于完整切片本身。

预处理检索字段用于审核和检索增强：

1. 触发场景 / 客户问：标准问答中来自客户方向；主动触达模板中为适用条件。
2. 参考回复 / 销售答：同一切片内销售方向多行合并后的话术。
3. 使用边界：说明哪些场景可用、哪些信息需要人工调整、哪些强个案不能直接照搬。
4. 商业场景编码：内部稳定编码。
5. 命中依据：解释为什么命中该商业场景。
6. 质量分和评分理由。
7. 处理结论。
8. 不入库原因或待人工确认原因。

字段区别必须明确：

1. 知识标题：总结整个问答或主动触达模板，用于检索列表和人工识别，例如“客户开会不便通话时约定下午回拨”。标题不是客户原话，也不应只截取第一句；标题文字中不需要额外加“切片”“模板”等泛化词。类似“基础跟进 - 老客唤醒触达模板”的标题只重复分类，不包含客户触发、销售动作和使用边界，不能作为合格标题。
2. 触发场景 / 客户问：标准问答中是客户方向的触发内容，多行客户消息必须合并显示；销售主动触达模板中是“触发场景/适用条件”，例如“长期未互动客户暂无明确提问，销售希望用案例做低压力触达”，不得伪造客户原话。
3. 参考回复 / 销售答：销售方向的实际回应或经脱敏后的参考话术。同一切片内连续销售消息、多条补充说明、分开发送的同一模板内容，都必须按原顺序合并显示；不能只取第一条销售消息，也不能让同一切片的不同行显示不同销售答。
4. 知识切片：问答组成的完整最小整体。详情页必须先把“触发场景 / 客户问 + 参考回复 / 销售答 + 使用边界”作为一个完整切片展示，确保同一 `slice_id` 的任意行点击后看到的是同一份整体。
5. 入库正文：准备写入正式知识库 `knowledge_chunk.content` 的正文，优先对齐现有知识库切片展示方式。它通常是可复用的销售回复、模板、案例正文或规则正文，是完整知识切片的一部分，不应被误解为整个切片。
6. 使用边界：说明该话术适用条件、需要人工调整的信息，以及不能直接照搬的强个案内容。
7. 商业场景编码：内部稳定编码，用于检索、统计、接口传参和后续改名兼容；业务侧可弱化展示，但预处理和审核时必须保留。
8. 命中依据：说明“为什么”命中该商业场景，必须引用语义信号，例如客户忙、销售约定回拨、销售做转介绍请求、销售做案例分享等；不能只写“按当前切片语义归入某场景”这种结论复述。

同一 `slice_id` 的任意一行被点击时，详情侧栏必须先显示同一份完整知识切片，再显示结构化字段明细。完整知识切片包含知识标题、触发场景、参考回复和使用边界；字段明细包含知识分类、服务、商业场景、入库正文、质量分、处理结论、商业场景编码、命中依据、来源行号范围和发布目标。行级原始消息只在折叠上下文中展示，不能覆盖拟发布知识结构。

完整原始对话可以作为“上下文”折叠区或单独入口，不应替代入库详情。

### 4.8 入库正文、参考回复和使用边界的区别

这三个字段不能混用：

1. 入库正文：正式写入 `knowledge_chunk.content` 的正文。它是后续检索、召回、生成回复时最核心的文本字段。对企微回复模板而言，它通常就是经过标准化后的销售可复用话术，因此很多场景会与“参考回复”相同。
2. 参考回复：预处理和审核字段，对应 `wecom_raw_import.slice_answer`。它必须保留“销售方向原文多行合并后的回复内容”，用于说明这条知识来自哪个销售答复。参考回复不能改写语气、不能替换成更礼貌或更标准的话术、不能把多行内容总结成新句子；最多只在页面另行展示标准化预览。它不一定独立进入正式知识库列；发布时通常作为入库正文的来源。
3. 使用边界：预处理和治理字段，对应 `wecom_raw_import.usage_boundary`。它不应作为可直接发给客户的话术，不应混入 `knowledge_chunk.content`。正式入库时应写入 `knowledge_chunk.structured_tags.usage_boundary`，作为检索加权、人工审核、LLM 生成时的适用条件和风险约束。

当前正式知识库落位建议：

| 预处理字段 | 正式知识库落位 | 后续用途 |
|---|---|---|
| `slice_title` | `knowledge_document.title` / `knowledge_chunk.title` | 列表展示、检索标题、人工识别 |
| `normalized_content` | `knowledge_chunk.content` | 正式入库正文，参与检索和回复生成 |
| `slice_question` | `knowledge_chunk.structured_tags.trigger_scenario` | 触发场景/客户问，用于检索增强和审核，不直接作为回复正文 |
| `slice_answer` | `knowledge_chunk.structured_tags.reference_reply`，通常也是 `normalized_content` 的来源 | 溯源和审核；当 `normalized_content` 与它相同时不需要重复作为正文存两份 |
| `usage_boundary` | `knowledge_chunk.structured_tags.usage_boundary` | 适用条件、变量替换要求、禁止照搬内容、生成约束 |
| `business_stage` | `knowledge_chunk.business_stage` | 正式检索字段 |
| `business_scenario_code` | `knowledge_chunk.business_scenario_code` | 正式检索字段 |
| `business_scenario_name` | `knowledge_chunk.structured_tags.business_scenario_name` | 中文展示和审计快照 |
| `scenario_match_reason` | `knowledge_chunk.structured_tags.scenario_match_reason` | 审核和模型调试 |

因此，“入库正文”和“参考回复”在企微话术类知识中可能接近，但不能反向污染：参考回复永远以原始销售回复合并为准；入库正文只能在参考回复基础上做必要脱敏、乱码清理、附件占位等最低限度标准化，不得改写原意、语气和措辞，不得删除正常表情标签。若两者完全相同，页面可显示为“参考回复：同入库正文”，避免造成重复知识的误解。

使用边界的合格标准：

1. 必须说明适用场景，例如老客唤醒、客户忙无法通话、邮件未收到后补发资料。
2. 必须说明需要人工替换或调整的信息，例如服务范围、客户关系、资料内容、发送时间、语气。
3. 必须说明不能直接照搬的风险，例如具体联系人、具体电话路径、强个案、缺少附件正文、过期活动。
4. 不能只写“按质量分决定”“人工复核后使用”这类空泛话；这种值不达标，只能作为临时兜底。

当前数据库核对结论：

1. `wecom_raw_import` 暂存表已有 `stage`、`section` 字段，但这是企微原始数据预处理字段，不是正式知识库字段；它们不代表正式知识库已经沉淀了可复用的阶段/环节字段。
2. `knowledge_document` 没有独立的商业阶段、商业场景列；可扩展信息主要存放在 `source_meta`、`tags` JSON 字段。
3. `knowledge_chunk` 已有独立商业阶段、商业场景编码列：`business_stage`、`business_scenario_code`；`structured_tags` 只用于保留场景中文名、展示值、命中依据等审计信息。
4. 当前旧知识库里确实存在 `scenario_label`、`intent_label`，但它们是旧的启发式标签，主要在 `knowledge_document.tags`、`knowledge_chunk.structured_tags` 中，不等同于新的商业场景标准。
5. `llm_business_scenario_standards` 是当前应作为企微商业阶段和商业场景判断标准的核心表，但现有企微预处理和正式知识库字段尚未与它完成对齐。

字段区别：

| 字段 | 当前位置 | 含义 | 当前状态 |
|---|---|---|---|
| 临时阶段 | `wecom_raw_import.stage` | 方案阶段和前 182 行试处理使用的销售大阶段 | 仅作为预处理参考，不作为正式知识库字段 |
| 临时环节 | `wecom_raw_import.section` | 方案阶段和前 182 行试处理使用的细分环节 | 与商业场景高度重叠，后续不再单独作为正式字段 |
| 商业阶段 | `llm_business_scenario_standards.category` | 22 个标准场景的上层分类，如基础跟进、价值呈现、异议处理 | 新标准字段，应写入正式知识切片 |
| 商业场景 | `llm_business_scenario_standards.scenario` / `scenario_full` | 具体销售场景，如新客激活、案例分享、已有稳定供应商 | 新标准字段，应作为检索和详情展示核心字段 |
| 商业场景编码 | `llm_business_scenario_standards.scene_value` | 商业场景的稳定代码，如 `lead_activation` | 内部存储和检索过滤使用，前端不直接展示英文代码 |
| 旧场景标签 | `knowledge_chunk.structured_tags`、`knowledge_document.tags` | 旧启发式标签，如 quotation、process、payment | 旧字段，只能兼容历史数据 |
| 旧意图标签 | `knowledge_chunk.structured_tags`、`knowledge_document.tags` | 旧启发式标签，如 pricing、script、pre_sales_touch | 旧字段，只能兼容历史数据 |

后续正式方案应新增或明确映射：

1. 不建议继续保留“阶段、环节、场景/意图”三套相近字段。推荐合并为两层：`商业阶段` 和 `商业场景`。
2. 正式知识库底层直接给 `knowledge_chunk` 增加独立列：`business_stage`、`business_scenario_code`，并建立索引；中文展示名从 `llm_business_scenario_standards` 表读取。
3. `knowledge_chunk.structured_tags` 只保留审计快照和辅助信息，例如 `business_scenario_name`、`business_scenario_full`、`business_scenario_display`、`scenario_match_reason`。
4. 企微详情侧栏展示“商业阶段 / 商业场景 / 命中依据”，不再展示容易混淆的“阶段 / 环节 / 场景/意图”三列。
5. `stage`、`section` 只作为企微暂存层预处理参考；`scenario_label`、`intent_label` 只作为现有智能回复和旧知识库兼容信息，不作为新数据标准。

## 5. 字段定义

### 5.1 原始暂存表字段

`wecom_raw_import` 用于存放原始消息和预处理结果。

关键字段：

| 字段 | 说明 |
|---|---|
| `import_batch_id` | 导入批次 |
| `sender` / `receiver` | 原始发件人/收件人 |
| `group_id` | 群聊 ID，私聊为空 |
| `msg_time` | 消息时间 |
| `content` | 原始消息内容 |
| `from_id` / `to_id` | 企微内部 ID |
| `role` | `customer` / `sales` / `unknown` |
| `group_key` | 当前导入时按空行或对话边界生成的粗分组 |
| `row_index` | 原始导入序号，可能是 0 基且可能因源数据缺行而不连续；页面验收说的“第 N 行”优先按企微数据页 `#` 列或表内导入顺序 `id` 对齐 |
| `stage` | 临时阶段，企微暂存/预处理字段；正式知识库不沿用 |
| `section` | 临时环节，企微暂存/预处理字段；正式知识库不沿用 |
| `business_line` | 服务类型 |
| `language_pair` | 语种。只有明确出现语言方向、语言对或翻译语种时才填写；国家、行业、出口贸易、国外项目、视频标题等不能推断语种。`business_line=多元` 表示服务类型覆盖多类业务，不等于多语种；不涉及明确语言方向时填 `不适用`；明确涉及翻译但无法判断语种时填 `未知` |
| `quality_score` | 预处理质量分，0-100 |
| `process_note` | 处理备注，不是客户话语解读 |
| `process_result` | 处理结论 |
| `row_status` | `pending` / `processed` / `published` / `rejected` / `skipped` |

当前 `wecom_raw_import` 已补充切片级字段；若后续数据量和审核流复杂度提高，可再建立单独的企微切片候选表：

| 当前切片字段 | 说明 |
|---|---|
| `kb_unit_id` | 知识单元候选 ID，可用于同一知识单元下多切片归组 |
| `slice_id` | 切片号。第 2 步切片生成后的唯一聚合键；同一切片内多行相同，不同切片不重复；第 3 步预处理和第 4 步筛选必须按它聚合 |
| `slice_title` | 知识标题预览，用于检索和人工审核；标题总结完整问答或模板，应表达触发场景、销售动作或可复用价值，不额外添加“切片”“模板”等泛化词，不用“商业阶段 - 商业场景”代替标题 |
| `slice_row_start` / `slice_row_end` | 来源行范围，优先用于追溯原始导入序号；页面展示和人工验收需要同时能对应企微数据页 `#` 列 |
| `slice_question` | 触发场景/客户问；标准问答中合并客户方向多行消息；销售主动触达模板可写“触发场景/适用条件”，不得伪造客户原话 |
| `slice_answer` | 参考回复/销售答；合并同一切片内销售方向的连续消息、多条补充和分开发送的同一模板，通常为销售实际话术或经脱敏后的模板话术 |
| `normalized_content` | 拟发布入库正文预览，对齐 `knowledge_chunk.content`；它不是完整知识切片，完整切片由 `slice_question` + `slice_answer` + `usage_boundary` 共同组成 |
| `slice_quality_score` | 切片级质量分，同一切片内多行必须一致 |
| `slice_process_result` | 切片级处理结论，同一切片内多行必须一致 |
| `slice_status` | 切片级状态，如 `processed` / `skipped` / `published` |
| `publish_target` | 目标知识库结构 |
| `skip_reason` | 不入库原因或待人工确认原因 |
| `business_stage` | 商业阶段，中文值来自 `llm_business_scenario_standards.category` |
| `business_scenario_code` | 商业场景编码，来自 `llm_business_scenario_standards.scene_value` |
| `business_scenario_name` | 商业场景中文名，来自 `llm_business_scenario_standards.scenario` |
| `scenario_match_reason` | 命中该商业场景的语义依据 |
| `usage_boundary` | 使用边界，说明适用条件、需要人工调整的信息、不能直接照搬的强个案内容 |

### 5.2 正式知识库字段

正式发布应进入现有知识库结构：

| 表 | 作用 |
|---|---|
| `knowledge_document` | 知识主文档，保存标题、来源、类型、状态、版本、业务线等 |
| `knowledge_chunk` | 实际参与检索的知识切片 |
| `pricing_rule` | 结构化报价规则，仅价格类知识需要 |
| `knowledge_candidate` | 候选池，可用于待审核内容 |

企微历史切片通常映射为：

1. `source_type = wecom_history` 或类似值。
2. `knowledge_type` 根据内容判断为 `faq` / `process` / `capability` / `pricing`。
3. `chunk_type` 可为 `template` / `example` / `rule` / `faq`。
4. `business_line` 使用系统现有枚举：`translation`、`printing`、`interpretation`、`multimedia`、`exhibition`、`gifts`、`general`。
5. `language_pair` 使用系统既有语种编码或空值。
6. `usable_for_reply` 和 `allowed_for_generation` 需由质量分和人工审核决定。

## 6. 切片流程

### 6.1 原始数据读取

输入为企微 CSV，字段包括：

`发件人 | 收件人 | 群ID | 时间 | 内容 | from | tolist`

要求：

1. 保留原始顺序。
2. 保留客户 ID、销售 ID、时间、群 ID。
3. 避免中文乱码。
4. 空行或明确对话边界可用于生成粗 `group_key`，但不能直接等同于最终切片。

### 6.2 粗分组

先按联系人/会话/空行/导入批次生成粗分组。

粗分组只用于上下文查看和分页展示，不是最终入库单元。

### 6.3 连续消息合并

同一角色连续消息可合并为同一方向文本，但必须满足：

1. 同一客户或同一销售连续发送。
2. 中间无另一方插入。
3. 时间间隔不长，当前初始规则为小于 4 小时。
4. 不跨天，除非人工确认是同一话题连续补充。
5. 上一句没有“好的、下次再说、拜拜、先这样”等结束性质。
6. 没有进入新的商业场景、话题或触达目的。

### 6.4 问答配对

标准切片为：

1. 客户问。
2. 销售答。

可接受的扩展：

1. 客户同意添加后的系统验证消息 + 销售首次有效触达。
2. 客户引用销售并补充实质信息 + 销售承接回答。
3. 客户多行问题合并 + 销售多行回答合并。
4. 销售主动触达模板：没有客户显性提问，但能构造明确触发场景/适用条件，例如老客户唤醒、节气/开工问候、转介绍请求、案例分享、能力介绍。

不入库：

1. 标准问答中找不到客户问，且无法构造有效触发场景/适用条件。
2. 找不到销售答。
3. 客户只是“好的、收到、OK、谢谢”等简单确认。
4. 销售后续无上下文价值，无法归纳为可复用话术、模板、案例或规则。
5. 强时效、强个案、容易误导的内容。
6. 乱码、未知消息类型、无意义表情。
7. 仅用于了解客户行业、组织、渠道、预算、供应商、个人情况的客户画像信息，若没有可复用销售回复，不进入回复知识库；这类信息可进入 CRM、客户画像或销售备注。
8. 外部供应商价格、客户自述采购价格、客户具体渠道成本，不能直接作为我方报价规则或报价知识；若记录，只能作为客户情报或待人工确认，不直接发布。

### 6.5 引用处理

前 30 条样例尚未出现真实引用消息；页面前 182 行中已出现真实引用样例：销售引用客户“易拉宝也图文印刷120一个”后仅回复“够卷”。该样例说明，引用结构出现后仍要判断引用后的新内容是否有知识价值，不能因为有引用就自动入库。

引用逻辑只在数据中明确出现“引用/回复某条消息/带引用块”的结构时启用，不能把普通上下文承接误当作引用，否则会打乱已生成的切片号和问答关系。

在没有明确引用结构的数据中，不执行引用拆分、引用合并或跨行重绑。普通前后文承接仍按现有 `slice_id` 聚合处理，不能因为后续消息看起来像回应，就重新拆坏已经确认的问答切片。

引用规则：

1. 销售引用客户原文并给出新回复：引用内容可作为问，新回复作为答，形成新切片。
2. 若引用内容已经被其他切片作为问并已有明确答，不重复绑定。
3. 客户引用销售原文并给出实质信息：该实质信息可作为新问或补充槽位。
4. 客户引用销售原文后只说“好的、收到”：不入库。
5. 销售引用客户原文后只做情绪化反应、赞同、吐槽或简单确认，例如“够卷、是的、收到”，不入库。
6. 引用文本要标准化，但不得改写原意。

## 7. 标准化规则

标准化只用于形成可复用、可检索、可审核的知识正文，不得改变原始业务含义。原始 `content` 必须保留，标准化结果写入 `slice_question`、`slice_answer`、`normalized_content` 等字段。

### 7.1 脱敏规则

1. 销售姓名替换为 `[销售姓名]`，除非姓名本身是公司公开署名且人工确认可保留。
2. 客户联系人姓名替换为 `[客户联系人]`。
3. 手机号码、座机、分机等联系方式替换为 `[联系电话]`。
4. 邮箱替换为 `[邮箱地址]`。
5. 微信号、企微号、QQ、会议号等账号替换为 `[账号信息]`。
6. 具体公司名原则上保留，因为它可能是行业、标杆案例或客户类型检索要素；如涉及隐私、合同限制或强个案，再替换为 `[客户公司]`。
7. 我方公司名不脱敏，例如“事必达”“上海嘉赛”“嘉赛”可直接保留，这是通用品牌和服务主体，不属于客户隐私。
8. 案例中的公司名默认保留，除非该案例涉及保密客户、合同限制、投诉纠纷或强个案风险；保留公司名有助于行业、标杆和案例检索。
9. 具体项目名、订单号、合同号、报价单号、发票号替换为 `[项目/单据编号]`，除非它们是正式知识规则的必要索引。

### 7.2 附件和多媒体规则

1. 文件名替换为 `[附件文件]`，除非文件名本身是关键业务检索要素，例如语种、行业、服务类型、报价主题。
2. 图片、视频、语音分别替换为 `[图片内容]`、`[视频内容]`、`[语音内容]`。
3. 视频号、案例视频、图片案例如果没有可读正文，不直接发布为正式知识；可进入待人工确认，要求补充视频/图片内容摘要。
4. 未知消息类型不入库，但保留原始行用于追溯。

### 7.3 文本清理规则

1. 保留原始话术中的语气、标点、表情标签和换行，除非是乱码、未知消息类型或无法阅读的占位内容。
2. 保留能体现语气、关系强弱、客户态度的必要口语表达，例如“麻烦您”“方便吗”“不着急”等。
3. 修正常见乱码和断行，但不得随意改写客户或销售原意。
4. 连续空格、无意义换行可轻量压缩；多条连续消息合并时用换行保留原始发送层次。
5. 中英文、数字、语种、服务名称不得误改，避免影响检索。

### 7.4 问答和模板正文规则

1. 标准问答切片：客户方向多行内容合并写入 `slice_question`，销售方向多行内容合并写入 `slice_answer`。
2. 销售主动触达模板：`slice_question` 写触发场景/适用条件，`slice_answer` 写同一切片内合并后的销售话术；必须在备注中说明这是虚拟触发场景，不是客户原话。
3. 系统验证消息可作为首次触达触发语，但不单独入库。
4. `normalized_content` 是拟发布入库正文预览，对齐 `knowledge_chunk.content`，不是完整知识切片。完整知识切片由触发场景、参考回复和使用边界共同组成；详情页主展示必须先展示这个整体，字段明细再拆开显示，便于审核和导入正式知识库结构。
5. 强个案内容应在正文或备注中标出边界；会误导检索的具体人、具体电话路径、具体临时日期不应直接发布。
6. `content` 原始列严禁修改。`slice_question` 和 `slice_answer` 也属于来源合并字段：有客户原文时必须合并客户原文，有销售原文时必须合并销售原文；不得把它们改写成总结、润色话术或标准回复。
7. 标准化只能写入 `normalized_content` 等预览字段，并且只能做最低限度处理：脱敏姓名、邮箱、电话、账号，替换无法读取的附件或媒体占位，清理乱码。不得删除正常表情标签，不得把“你看看垃圾邮箱里有不”改成“您可以先看下垃圾邮箱”这类语气改写。

### 7.4.1 前 182 行新增边界

页面前 182 行验证后补充以下边界，后续批量预处理必须遵守：

1. 销售主动探需问题可以作为候选模板，但必须满足“问题本身可复用、触发场景明确、不会误导”。例如“礼品定制有没有机会合作”“市场推广有什么新形式”可进入待人工确认；若只是零散闲聊或客户画像采集，不进回复知识库。
2. 客户画像信息不等于知识库知识。客户行业、部门、是否有市场部、是否创业、渠道/零售情况、是否参展等，如果没有可复用销售回复，应标记为不入库或待进入 CRM，而不是写入 `knowledge_chunk.content`。
3. 视频号、案例视频、图片案例、附件介绍若没有可读正文或摘要，默认进入 `待人工确认`，要求补视频/图片/附件摘要；不能只把“请看视频号”当作正式能力知识。
4. 人员离职/工作交接通知属于 `基础跟进 / 人员离职/交接 / handover`，可作为正式候选，但必须替换具体新旧联系人姓名。
5. 客户未收到邮件、资料未送达、需要查垃圾邮件等，优先归为 `客情经营 / 紧急响应/确定性交付 / emergency_response`；若只剩客户单句“没有”且无后续承接，则不入库。
6. 线下面谈、拜访时间、地址、楼号、会议室这类信息可形成“面谈确认模板”，但具体地址、当天时间和入场细节强个案，通常只能待人工确认，不直接发布。
7. 外部供应商低价、图文店价格、客户内部采购成本属于强个案或客户情报；不能进入报价规则，也不能作为我方标准报价知识。
8. 行业共鸣/吐苦水类内容只有在销售回复具备可复用表达、且适用于熟客关系维护时才进候选；过于口语、消极或内部经营表达需人工确认。
9. 真实引用消息要单独判断引用后的新回复价值。引用后只有“够卷”等情绪化反应，不入库；引用后有新方案、新报价路径、新能力说明，才可形成新切片。

### 7.5 切片号聚合约束

第 2 步生成切片号后，第 3 步所有预处理字段必须以 `slice_id` 为聚合键生成：

1. 输入给 LLM 或人工预处理的对象是一个已冻结 `slice_id` 对应的完整切片，而不是切片中的某一行。
2. 输出字段必须一次性写回该切片内所有行，保持同值。
3. 质量分和处理结论按切片整体评估，不允许客户行低分、销售行高分这类单行结果。
4. 第 3 步不得改变 `slice_id`。若发现切片边界错误，必须停止该切片预处理，回到第 2 步重新生成或人工修正切片号。
5. 若发现同一个 `slice_id` 内字段不一致，视为预处理失败；先修正同切片字段一致性，再做入库判断。
6. 若发现本应不同切片的行使用了同一 `slice_id`，不能在第 3 步临时按语义拆开处理，必须先回第 2 步拆切片号。
7. 界面必须显示切片号，便于人工确认当前看到的是同一处理单元。
8. 质检脚本必须检查：同一 `slice_id` 字段一致、不同 `slice_id` 不共享同一知识标题/问答正文、无 `slice_id` 的行不得进入 `入库_切片` 或 `待人工确认`。

### 7.5 质量和发布规则

1. 质量分按切片评估，同一 `slice_id` 内多行必须一致。
2. `入库_切片` 表示可进入正式候选/发布链路；`待人工确认` 表示可复核但不直接发布；`不入库_*` 表示当前不进入知识库。
3. 纯推广、群发模板、节气问候、案例视频、能力介绍不自动不入库；只要可复用、可检索、不会误导，就可以作为模板或案例候选。
4. 无上下文价值、纯确认、乱码、无法标准化附件、强时效过期内容，不进入正式知识库。
5. 除必要脱敏、引用整理、去乱码外，不得随意改写原始内容。

## 8. 商业阶段、商业场景和智能回复检索方案

### 8.1 字段合并结论

“阶段、环节、场景/意图”三个字段表达接近，继续并列会导致人工标注、检索过滤和前端展示都产生歧义。

推荐合并为两层：

| 中文字段 | 来源 | 含义 | 是否保留 |
|---|---|---|---|
| 商业阶段 | `llm_business_scenario_standards.category` | 标准商业场景的上层分类 | 保留，作为粗过滤和展示字段 |
| 商业场景 | `llm_business_scenario_standards.scenario` / `scenario_full` | 具体销售场景 | 保留，作为核心检索字段 |
| 商业场景编码 | `llm_business_scenario_standards.scene_value` | 稳定内部编码 | 保留在数据库，不直接给业务用户展示 |
| 临时阶段 | `wecom_raw_import.stage` | 方案阶段和前 182 行试处理字段 | 不进入正式知识库底层 |
| 临时环节 | `wecom_raw_import.section` | 方案阶段和前 182 行试处理字段 | 不进入正式知识库底层 |
| 旧场景/旧意图 | `scenario_label` / `intent_label` | 旧启发式标签 | 仅兼容旧知识库和旧智能回复链路 |

前端统一中文表达为：

1. 商业阶段。
2. 商业场景。

1. 第 1 步：读取企微对话。
2. 第 2 步：快速规则和线程状态判断。
3. 第 3 步：LLM 摘要和当前推进点提取。
4. 第 4 步：CRM 画像和客户上下文补充。
5. 第 5 步：检索知识库，返回可引用证据和候选话术。
6. 第 6 步：LLM 结合摘要、线程状态、CRM、知识证据生成回复建议。
7. 第 7 步：评分、记录、人工反馈和后续治理。

1. `knowledge_type`：`faq`、`pricing`、`process`、`capability`。
2. `chunk_type`：`template`、`example`、`rule`、`faq`、`constraint`、`definition`。
3. `business_line`：翻译、印刷、展会等业务线。
4. `language_pair`：语种。
5. `service_scope`：法律、医药、技术、认证、通用等服务范围。
6. `customer_tier`：客户层级。
7. 关键词、向量、BM25、元数据匹配、质量分、可生成标记。
8. 旧 `scenario_label`、`intent_label` 主要进入线程事实和提示词，尚未作为新商业场景标准过滤。

LLM1 当前没有直接输出 22 个商业场景。当前 LLM1 主要输出 `topic`、`core_demand`、`key_facts`、`todo_items`、`risks`、`to_be_confirmed`、`status`。后续 `build_thread_business_fact` 会基于摘要、最新消息、CRM 和规则生成旧的 `business_state`、`stage_signals`、`scenario_label`、`intent_label`，但这套结果不是 `llm_business_scenario_standards` 的 22 个商业场景。

因此应新增“商业场景识别结果”，位置建议放在线程事实层：

1. 输入：LLM1 摘要、客户最后一句、销售最后一句、CRM 画像、现有 `stage_signals`、附件摘要。
2. 输出：`business_stage`、`business_scenario_code`、`business_scenario_name`、`business_scenario_confidence`、`scenario_match_reason`。
3. 实现方式：规则高置信命中 + LLM 语义判断兜底；输出必须对齐 `llm_business_scenario_standards` 当前启用的 22 个选项。
4. 不建议完全靠代码规则强筛选，因为企微历史上下文缺失、老客户新加企微、客户简单确认等情况会让硬规则误伤。
5. 检索时把商业场景作为新增检索维度和排序加权，不作为唯一硬过滤条件。

同时还会调用外部销售知识库 API：`POST /kb-units/qa-retrieve`。外部接口当前按问答检索返回候选销售回复，也没有看到与本地 22 个商业场景标准的强绑定。

### 8.4 最终兼容方案

最稳妥方案是“新增标准字段、保留旧字段兼容、逐步让检索使用新字段加权”。

入库时：

1. 每个企微切片必须判断 `商业阶段` 和 `商业场景`。
2. `商业阶段` 取 `llm_business_scenario_standards.category` 的中文值。
3. `商业场景` 取 `llm_business_scenario_standards.scenario` 的中文值。
4. `商业场景编码` 取 `llm_business_scenario_standards.scene_value`。
5. `命中依据` 记录为什么命中该商业场景，例如客户明确询价、客户已有供应商、销售做老客唤醒等。
6. `stage`、`section` 只存在于企微暂存/预处理层，不进入正式知识库；`scenario_label`、`intent_label` 只作为现有智能回复和旧知识库兼容信息。

正式知识库存储：

1. 正式知识库底层直接给 `knowledge_chunk` 增加独立列：
   - `business_stage`：商业阶段中文名，例如“基础跟进”。
   - `business_scenario_code`：商业场景编码，例如 `lead_activation`。
2. `business_scenario_code` 必须能在 `llm_business_scenario_standards.scene_value` 中找到启用记录。
3. `business_stage` 必须与该启用记录的 `category` 一致。
4. `knowledge_chunk.structured_tags` 不作为主查询字段，只保留辅助信息：
   - `business_scenario_name`：商业场景中文名。
   - `business_scenario_full`：商业场景完整名。
   - `business_scenario_display`：前端展示值，例如“基础跟进-新客激活”。
   - `scenario_match_reason`：命中依据。
   - `source_preprocess_stage` / `source_preprocess_section`：仅在来源为企微暂存预处理时保留原始参考值，可为空。
5. 最终检索展示时可以合成为“基础跟进-新客激活”，但底层必须以独立列和编码字段参与检索，避免只靠中文拼接字段。

智能回复检索：

1. LLM1 摘要之后、知识库检索之前，先判断当前会话的商业场景，写入线程事实层。
2. 第 5 步检索时，商业场景不应作为硬过滤唯一条件，避免判断错场景导致无命中。
3. 商业场景应作为加权因子：同一商业阶段加分、同一商业场景更高加分、不同商业阶段但语义强命中仍可保留。
4. 对价格、能力、流程、案例这类强业务知识，仍优先使用 `knowledge_type`、业务线、语种、服务范围过滤。
5. 对企微和邮件回复模板，优先使用商业阶段、商业场景、客户最后一句类型、是否待客户回复、是否老客唤醒等上下文字段。
6. 外部销售知识库 API 若暂不支持商业场景字段，则把商业阶段、商业场景中文名拼入检索 query 或 metadata；等外部接口支持后再改成结构化字段。

这能兼容现有商业化企微回复和邮件回复：

1. 企微回复更依赖短话术、客户最后一句、当前线程状态，因此商业场景用于挑选“怎么回”的模板和案例。
2. 邮件回复更依赖邮件主题、正文目的、正式程度，因此商业场景用于筛选邮件模板和跟进策略。
3. 报价、能力、流程类知识不应被商业场景完全替代；商业场景只是提高检索准确率的一层上下文。
4. 知识库第 5 步只提供证据和候选，不负责完整判断客户关系、CRM 历史、是否该发、最终话术风格。这些仍由线程事实、CRM、LLM 生成和人工反馈共同完成。

## 14. 2026-05-18 195-1000 行切片口径纠偏与重处理

### 14.1 本轮人工确认的关键纠偏

本轮人工确认后，修正此前对底色和切片关系的误解：

1. 红/绿底色才表示“有客户和销售往返的对话切片”。同一红/绿切片内可以包含多行客户问和多行销售答，预处理时按同一个 `slice_id` 聚合。
2. 黄色底色表示“仅客户消息”，本身不是对话。黄色一行就是一个切片/跳过单元，不把连续黄色行合并成一个切片，也不存在用两种黄色交替区分。
3. 紫色底色表示“仅销售消息”，本身不是对话。紫色一行就是一个切片/候选单元，不把连续紫色行合并成一个“紫色块”整体处理，也不存在用两种紫色交替区分。
4. 因此“对整个紫色块做整体预处理”的口径是错误的。正确做法是：紫色每一行独立处理；它可以作为销售主动触达模板候选，但粒度必须是一行一个切片。
5. 第 3 步预处理仍然不能改切片边界。若发现当前 `slice_id` 把多行黄色或多行紫色合并了，说明第 2 步切片号本身错误，应先修正为一行一个 `slice_id`，再按新切片做预处理。

### 14.2 当前最新切片定义

当前企微历史记录切片定义以如下规则为准：

1. 红/绿对话切片：同一底色块内同时包含客户和销售消息，作为一个问答切片。客户方向多行合并为 `slice_question`，销售方向多行合并为 `slice_answer`。
2. 黄色单行切片：只有客户消息，不形成问答。每一行独立生成 `slice_id`，通常判断为 `不入库_无答`、`不入库_简单确认` 或 `不入库_无上下文价值`。
3. 紫色单行切片：只有销售消息，不形成问答。每一行独立生成 `slice_id`，可根据复用价值判断为 `入库_切片`、`待人工确认` 或不入库。
4. 底色是切片边界判断的关键依据，但颜色本身不是入库结论。紫色不是自动不入库，黄色也不是自动可丢弃；仍需按单行切片做处理结论。
5. 商业阶段和商业场景只在切片边界确认后判断，不能用商业场景反推合并或拆分 `slice_id`。

示例：
