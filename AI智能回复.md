# AI智能回复

## 目标
- 专门记录“AI 生成回复文字效果”的每次调整。
- 每次都保留：触发场景、误差原因、改动边界、验证方式、回滚点。
- 后续新增案例时，按同样格式追加，不能覆盖旧案例。

## 通用边界
- CRM 历史信息只能作为背景，不等于当前会话阶段。
- “客户未回复”场景下，AI 不能只是把上一条我方消息换同义词再说一遍。
- 如果线程状态是“我方已发出，客户未回”，下一条应优先：
  - 降低回复门槛。
  - 换一个新角度推进。
  - 给客户一个容易作答的短问题或二选一。
- 禁止编造：
  - 历史合作年限。
  - 当前客户职务。
  - 当前明确项目进度。
  - 未命中的价格、能力或交付承诺。
- “资料/项目资料”这类泛词不能直接推断成“正在等客户补附件”。
- CRM 里存在历史报价，不代表当前线程已经进入报价阶段。

## 本次改动范围
- 前端：
  - 当前选中会话标题显示完整 `external_userid`，便于人工核对和数据库追踪。
- 线程事实层：
  - 新增“待客户回复 / 连续我方发送 / 初次触达后二次跟进”等信号。
  - 修正“CRM 历史报价误判当前报价阶段”。
  - 收紧“等待客户补资料”的识别条件，避免被“项目资料”这类泛描述误伤。
  - 补充最近两条我方消息 `recent_sales_messages`，供提示词和输出校验共同使用，而不只看最后一条。
- 检索与生成：
  - 未回复跟进场景下，检索问题自动补入“客户未回复后的跟进方式 / 避免重复上一条话术 / 换个更容易回复的角度”等策略信号。
  - 第二轮修正中，把“上一条我方原文”从检索 query 中移除，改成抽象策略信号，避免拿旧外联原话去召回旧模板。
  - 未回复跟进场景的知识过滤从 `email_template + process` 放宽为 `email_template + process + faq`，避免只困在模板类知识里。
  - 第二阶段提示词显式约束：不能重复上一条我方原话，必须换角度推进。
- 输出校验与兜底：
  - 新增“未回复跟进复述旧外联”的阻断校验。
  - 如果第二阶段首轮结果仍复述旧话术，自动追加一次纠偏重写。
  - 如果纠偏后仍失败，落到低门槛安全兜底话术，不再把明显错误结果直接展示出来。

## 方案与当前落地对应
- 目标方案：
  - 不是只防“重复上一句”，而是把企微跟进回复拆成可复用的策略路线。
  - 对“客户未回复”这类线程，下一条回复应在以下方向中择优：
    - 低门槛确认。
    - 业务介绍。
    - 案例吸引。
    - 转介绍 / 找对接人。
  - 第 6 步理想状态不应只有“生成 + 校验”，还应有更明确的策略分流与优先级控制，而不是让模型自行漂移。
  - 原始设想中的“打分”应至少区分两层：
    - 检索证据打分：决定送什么知识给第 6 步。
    - 回复结果打分：决定哪种回复策略更适合当前线程。
- 当前已落地：
  - 已落地的是“检索证据打分 + 生成后校验 + 失败兜底”，还不是完整的“多策略回复内容打分选优”。
  - 当前仍保留的打分层：
    - 第 5 步 `知识库检索证据` 的 `score / 置信 / useful_score / effect_score / followup_strategy_score`，用于候选知识排序。
    - 知识资产与训练样本层的 `useful_score / effect_score / feedback_count / positive_feedback_count`，用于长期知识治理和样本回流。
  - 当前尚未落地的部分：
    - 第 6 步多策略生成后按“更适合当前线程”的数值化选优。
    - “业务介绍 / 案例吸引 / 低门槛确认 / 转介绍”四类回复路线的完整路由器。
- 本次修改实际完成的控制层：
  - 线程事实层：
    - 把“客户未回复 / 连续我方发送 / 二次跟进”从隐含现象变成显式信号。
  - 第 5 步检索层：
    - 把“上一条我方原文驱动召回”改成“跟进策略驱动召回”。
    - 把“负责人确认 / 转介绍”识别成更具体的 `followup_focus`。
    - 放开 `contact_discovery / contact_follow_up / sales_outreach` 的召回范围。
    - 把知识里的 `script` 归一到 `email_template`，避免负责人确认类知识被过滤掉。
  - 第 6 步生成层：
    - 仍由主模型 / 对比模型各自生成。
    - 对“重复旧外联 / 重复问负责人”增加运行时拦截。
    - 如果首轮重复，先纠偏重写；再失败则回退兜底短句。
  - 第 7 步校验层：
    - 现在能拦“复述旧外联”和“重复追问负责人”。
    - 校验通过后才视为当前结果可直接参考。
- 当前边界判断：
  - 这套逻辑已经是“同类场景通用规则”，不是只为 `案例1` 写死。
  - 但它仍偏“防错与保守推进”，还没达到“多种跟进策略自动择优”的完整方案状态。

## 案例1
- 日期：2026-04-29
- external_userid：`wmS8sICwAA5rmSiEFuJ4AYxhWnVH_6yA`
- session_id：`single_WangHuiYing_wmS8sICwAA5rmSiEFuJ4AYxhWnVH_6yA`
- 场景：
  - 2026-04-28 14:55:02 我方先发问候。
  - 2026-04-28 14:55:07 我方继续补充合作背景并询问对方是否仍负责相关工作。
  - 截止分析时，客户未回复。
- 原始问题：
  - 第二阶段生成的话术，基本重复了第一轮外联内容，没有体现“客户未回复后的二次跟进策略”。
- 排查结论：
  - 信息跑偏 1：线程事实把 CRM 历史商机摘要误当成当前报价阶段，导致 `business_state` 被错误打成 `formal_quote`。
  - 信息跑偏 2：会话里出现“整理项目资料”这类泛词，被误判成“等待客户补资料/附件确认”。
  - 信息缺失 3：系统没有明确识别“当前最后两条都是我方消息，客户尚未回复”。
  - 检索跑偏：知识检索问题只有“了解客户当前职责和合作意向”，过于抽象，召回了“提及历史合作并询问当前需求”的老模板，进一步把 LLM 带回原话复述。
  - 脚本边界不足：第二阶段提示词没有明确禁止“复述上一条我方外联”。
- 本次修正：
  - 新增 `awaiting_customer_reply` / `sales_only_conversation` / `followup_after_no_reply` 信号。
  - 把 CRM 历史报价改成单独背景信号，不再直接决定当前报价阶段。
  - 收紧“等待客户补资料”的规则，改为显式请求模式识别。
  - 未回复场景下自动增强检索 query。
  - 第二阶段提示词加入“禁止重复上一条我方原话、必须换角度推进”的硬约束。
- 第一轮修正后的残留问题：
  - 虽然线程事实和提示词都已补强，但“重新点击第二阶段”后，结果仍可能没有明显变化。
  - 根因不是按钮没触发，而是第一次记录漏了两条真正决定结果的链路：
    - 漏点 1：检索 query 仍把 `latest_sales_message` 原文直接拼进去了，等于用旧外联去召回旧模板。
    - 漏点 2：输出校验只拦截报价/付款/发货类越权，没有拦截“未回复跟进复述上一条外联”，所以模型即使违背意图，结果也会照常落库展示。
- 第二轮补充排查结论：
  - “重新点击后还是老味道”并不说明线程事实没生效，而是检索层还在用旧外联原句做召回锚点。
  - 只靠提示词约束不够，必须把“复述旧外联”升级成运行时校验规则，否则模型偶发违背约束时系统不会拦截。
  - 未回复跟进不应只检索模板，还应允许召回 FAQ / 流程类的“低门槛跟进策略”，否则知识面过窄。
- 第二轮补充修正：
  - `backend/knowledge_governance.py`
    - `build_thread_business_fact()` 新增 `recent_sales_messages`。
    - `validate_thread_state_consistency()` 新增“未回复跟进复述旧外联”的阻断规则。
  - `backend/intent_engine.py`
    - `sanitize_thread_fact_for_prompt()` 透传 `recent_sales_messages` 给第二阶段提示词。
    - `_followup_retrieval_hints()` 新增未回复跟进的抽象检索策略信号。
    - `build_retrieval_query()` 不再拼接 `上一条我方消息:...` 原文，改为拼接“降低门槛 / 二选一 / 避免重复自我介绍 / 避免重复合作背景 / 避免重复确认负责人 / 不要围绕历史资料寒暄”等策略词。
    - `_query_intent_profile()` 增补 follow-up 意图识别。
    - `infer_query_features()` 把未回复场景的 `knowledge_class` 调整为 `["email_template", "process", "faq"]`。
    - `generate_sales_assist_bundle()` 新增未回复复述场景的二次纠偏重写与安全兜底。
    - `build_followup_guard_response()` 提供系统兜底短句。
- 验证关注点：
  - 线程事实应显示为 `inquiry` 或 `followup` 语义，而不是误判 `formal_quote`。
  - 检索 query 里不应再出现 `上一条我方消息:` 加原文整句。
  - 生成结果不能再重复“之前合作过/您现在还负责吗”这一组原问题。
  - 如果模型首轮仍复述，校验层应返回阻断问题，而不是直接放行。
  - 优先生成更容易回复的一句话跟进，例如：
    - 给客户一个低门槛回应口。
    - 增加一个具体价值点。
    - 提供一个更短、更轻的确认问题。
- 本次文档补记说明：
  - 第一版记录只能解释“为什么理论上应该变好”，但不能解释“为什么实际重跑仍没明显变化”，因此不够可追溯。
  - 当前版本已补到“事实识别 -> 检索 query -> 提示词 -> 运行时校验 -> 兜底输出”五层，后续如果再出同类问题，应能直接从本文件定位到遗漏层级和对应代码点。
- 第三轮真实页面复测（仅用 8000，按页面按钮逐步点击）：
  - 第 3 步 `重新提取`：
    - 结果合理。
    - `线程业务事实层` 已稳定识别 `状态 inquiry / 场景 followup / 意图 followup`，且门禁明确提示“客户未回复，下一条必须换角度推进，不能重复上一条原话”。
  - 第 5 步 `重新检索` 首次复测：
    - 不合理。
    - 虽然 `检索问题` 已不再拼接上一条我方原文，但命中前排仍是“再次联系与背景铺垫 / 公司简介与服务范围 / 需求邀约与祝福”这类泛化老模板。
    - 根因 1：跟进场景下仍把默认 `service_scope=general` 作为硬过滤，导致 `contact_discovery / contact_follow_up / sales_outreach` 里的更贴近知识块进不来。
    - 根因 2：库内一批更合适的负责人确认类切片在 `structured_tags.knowledge_class` 中被标成 `script`，检索侧没有把它归一为 `email_template`，被显式知识分类过滤掉。
    - 根因 3：未回复跟进的重排只奖励“再次联系/跟进”这类宽泛词，没有对“负责人确认/转介绍”做更强提升，也没有继续压低长介绍模板。
  - 第 5 步修正：
    - `backend/intent_engine.py`
      - `_followup_retrieval_hints()` 增补“若对方已不负责，礼貌获取当前对接人”。
      - `_knowledge_class_for_chunk()` 将 `script` 归一为 `email_template`；`_knowledge_class_fields()` 同步兼容 `script`。
      - `infer_query_features()` 在“客户未回复”场景下不再把默认 `service_scope=general` 当作硬过滤，改为 `service_scope_allowlist = [general, contact_discovery, contact_follow_up, sales_outreach]`。
      - 同步新增 `followup_focus = owner_confirmation`，把“确认负责人/是否转给同事”识别成更具体的跟进意图。
      - `_followup_strategy_score()` 强化“负责人确认/转介绍”类切片加权，并继续压低“公司介绍/合作背景/长自我介绍”类老模板。
  - 第 5 步修正后复测：
    - 达到继续下一步的标准。
    - 前排已变成“当客户表示不负责相关业务领域时，如何礼貌地探询负责人的信息？”、“对医药公司挖掘注册部的需求和对口负责人”等负责人确认类证据。
    - 仍有个别医药泛模板残留在后排，但不再主导召回方向。
  - 第 6 步 `重新生成对比` 首次复测：
    - 仍不合理。
    - 主模型虽然不再自我介绍，也不再编造“最近项目多”，但仍在重复“现在还是您负责 / 还是其他同事对接”这一核心旧问题。
  - 第 7 步首次复测：
    - 不合理。
    - 旧校验未拦住“重复追问负责人”。
  - 第 6 / 7 步修正：
    - `backend/knowledge_governance.py`
      - `validate_thread_state_consistency()` 补充“重复追问负责人/对接人”模式识别：只要前一轮我方已问过“是否负责/谁负责/谁在对接”，新回复再次围绕同一问题发问，即判定为“未换角度推进”并阻断。
    - 这样第二阶段在生成阶段就会触发：
      - 先走一次纠偏重写；
      - 若仍重复，则退回 `build_followup_guard_response()` 的低门槛兜底短句。
  - 第 6 / 7 步修正后复测：
    - 主模型最终落库为：
      - `不多打扰，想确认下这块近期还有在推进吗？您回我“有/暂时没有”都行。`
    - 对比模型最终落库为：
      - `您那边目前有新的翻译需求吗？或者如果有同事需要帮忙的可以转给我。`
    - `输出校验与人工复核` 最终为 `ok / 可直接参考`，且不再触发“复述旧外联”阻断。
  - 复测补充说明：
    - 第 6 步后台已成功覆盖新结果后，前端短时可能还停留在旧 DOM；重新读取分析页后会显示最新落库结果。
    - 这属于前端轮询时机表现，不是本次回复逻辑仍未覆盖。
