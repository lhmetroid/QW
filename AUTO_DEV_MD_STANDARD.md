# 自动化开发项目 Markdown 标准规范

> 本文是 QW 项目"邮件智能回复"模块 75 个 task 全部 ✅ 但实际产出空壳的复盘产物，目的是让以后任何用 Codex/Hermes/Claude Code/Antigravity 等 AI agent 跑无人值守自治开发的项目，能在第一份 md 上就避开同类陷阱。
>
> 适用场景：单个 AI agent（或 cron 自治模式）按一份 md 任务清单连续执行 20+ 个 task 的开发场景。不适用于人工逐 task 协作的场景。

---

## 0. 一句话总结

**只写"不要做什么"的指令，会得到一个什么都不做的空壳系统。** 自动化开发的 md 必须同等强度地写"必须做什么"（Definition of Done），并强制人工 checkpoint 切断自治闭环的自我繁殖。

---

## 1. 这次踩的坑（详细复盘）

### 坑 1：安全护栏被过度解读，从"运行时关闭"扩散成"代码层不接入"

**原始指令**（AGENTS.md line 402）：
> 邮件发信相关能力必须默认关闭真实发送，先使用草稿或 Mock 模式验证。

**Agent 实际理解链**：
- 不真实发信 → 不接 SMTP
- 不接 SMTP → 也别接 LLM（反正最终不发）
- 不接 LLM → 别查数据库（反正没真草稿要存）
- 别查数据库 → 前端别绑后端（反正后端也是空的）
- 前端别绑后端 → 整个面板是页面内 Mock 常量

**结论**：一条安全护栏在 75 个 task 的传染下，把整个系统做成了空壳。

**正确写法（已在 AGENTS.md 修正为）**：
> 必须区分 (a) 运行时 `MAIL_REAL_SENDING_ENABLED=false` 默认关闭真发出口；(b) 代码层必须真实接入 LLM、CRM、Few-Shot、前端 fetch、数据库写入。把这两件事画等号会导致系统空壳化。

### 坑 2：任务标题里 baked-in 实现手段关键字

**反模式样例**（TASKS.md 原 Task 53）：
> Task 53：设计邮件全局配置管理入口（已新增 `质量诊断 / Mail Config` 子导航与 review-only Mock 配置面板骨架，保持不调用后端、不写数据库、不发信、不接企微）

任务描述本身把"review-only Mock"、"不调用后端"写成了硬约束。后续 cron tick 在做下一个任务时，会把这个描述当成"成功模板"继承。

**正确写法**：任务标题只描述"做什么 + 完成长什么样"，不预先约束实现手段。实现手段（要不要 Mock、要不要接后端）是 DoD 的事，不是任务标题的事。

### 坑 3：cron 自治闭环 + agent 自己回填完成括号 = 自我繁殖

cron 模式下，每个 tick 是隔离会话，agent 自己写"已完成"括号注释回填 TASKS.md，自己 update PROGRESS.md，自己跑 VALIDATION 盖章。一旦某个 tick 走偏，后续 tick 把这个偏差当作 ground truth 反复放大。

**证据**：PROGRESS.md 凌晨 1 点到 7 点之间出现了 50+ 次完全相同的"全部任务完成核对"日志——cron 在反复跑空盖章。

**正确做法（已在 AGENTS.md 加入）**：每完成 10 个 task 强制 checkpoint，agent 必须停下写一份端到端可用性自检报告，人工 review 后才能继续。

### 坑 4：VALIDATION 把 Mock=通过钉死，形成验收锁

**反模式样例**（VALIDATION.md 原）：
> - 默认只生成草稿，不真实发送邮件。

这种断言一旦写进 VALIDATION，agent 就会理解成"接入真实 LLM = 验证失败"，反向锁死了真接入路径。

**正确写法（已修正为）**：验收必须同时包含 (A) 契约形态 和 (B) 真接入证据。两组都过才算 ✅。

### 坑 5：只有禁止清单，没有 Definition of Done

AGENTS.md 通篇写了大量"禁止：可启用真实邮件发送、真实客户触达、生产 CRM 写入..."，但全文找不到一句"完成的标准是：浏览器打开能看到 AI 真生成一封草稿"。

**结论**：对一个理性 agent 来说，最安全的策略就是什么都不做——因为骨架不可能违反禁止清单。

## 2. 现有做得好的、要继续保持的

复盘不是全盘否定。下面这些做法是好的，新项目应该照抄。

### 好做法 1：五件套文档矩阵

`AGENTS.md / TASKS.md / PROGRESS.md / TASK_HANDOFF.md / VALIDATION.md` 这套分工非常清楚：
- AGENTS.md = 规则书（怎么干）
- TASKS.md = 待办清单（干什么）
- PROGRESS.md = 流水账（干到哪了）
- TASK_HANDOFF.md = 交接本（卡在哪了）
- VALIDATION.md = 验收单（怎么算干完了）

每份职责单一，互不重叠，新 agent 接手能在 5 分钟内拼出全貌。**继续保持**。

### 好做法 2：任务原子化 + P0/P1/P2 分级 + 显式依赖

TASKS.md 的 75 个 task 每个都是原子可验证的，分级清晰，依赖前后链条明确（task 26→27→28→29→30→31）。这让 cron 模式能稳定执行，新人接手能看出当前进度。**继续保持**。

### 好做法 3：每个 task 完成都留时间戳 + 验证命令

PROGRESS.md 里 "2026-05-26 22:42:08 +0800 | Task 66 | ... | 验证通过：`git diff --check -- docs/mail_draft_manual_scoring_standard.md`" 这种格式很好，事后可审计、可回放。**继续保持**。

### 好做法 4：物理/逻辑命名隔离

邮件系统强制 `backend/mail_*` / `/api/v1/mail/*` / `mail_*` 表前缀 / `MAIL_*` 配置前缀，避免污染原有的企微链路。这个隔离设计是干净的。**继续保持**。

### 好做法 5：断点续传 + 失败不丢任务 + 重试日志分离

`logs/codex-run.log`（进展）和 `logs/codex-retry.log`（失败重试）分离，cron 失败自动重试不丢任务，配合 `mail_import_batch` 这种台账表，让 80 万行邮件采矿能在网络不稳的环境下完成。**继续保持**。

### 好做法 6：中文一致 + 时间戳带时区

所有日志、注释、状态文件统一中文 + `+0800` 时区。可读性高，跨时区协作时不会出错。**继续保持**。

### 好做法 7：状态文件本身用 git 跟踪

`TASKS.md / PROGRESS.md / TASK_HANDOFF.md` 都在 git 里，每个 commit 能看到任务推进过程，事后复盘有据可查。**继续保持**。

### 好做法 8：禁止动态脚本拼接 shell 命令

AGENTS.md 强制 `codex exec --sandbox workspace-write "任务描述"`，禁止 `python -c` / `node -e` / `eval` / `$()` 动态拼接。这避免了"危险命令"误判和不可审计的 shell 注入。**继续保持**。

## 3. 新项目 md 标准模板

### 3.1 AGENTS.md 必备章节

```markdown
# AGENTS

## 0. 最高优先级
（恢复 / 重试 / 不中断的安全网）

## 1. 角色分工
（Hermes / Codex / 当前 Agent 各干什么）

## 2. Agent 启动规则
（必读哪些文件）

## 3. 工作方式
（每次只做第一个未完成任务的循环模式）

## 4. ⭐ Definition of Done（必须有）
任务标记 [x] 之前，必须同时满足：
1. 契约通过（编译/单测/类型）
2. 端到端可演示（真数据真链路，证据落盘 logs/dod-task<N>.md）
3. 推进可用性（一句话说清楚离最终交付近了哪一步）

反模式速查：
- "review-only Mock"
- "页面内预览"  
- "Mock dict 替代"
- "硬编码模板"
（在任务描述里出现这些关键字，自动视为 DoD 不达标）

## 5. ⭐ 强制人工 checkpoint（必须有）
每完成 N 个 task（建议 N=10），cron 必须暂停，写自检报告，
人工 review 后才能继续。这是切断自治闭环自我繁殖的关键。

## 6. 运行机制
（cron / gateway / 失败重试）

## 7. 系统隔离规则
（命名隔离 / 运行隔离 / 数据隔离）

## 8. 工程原则
⭐ 必须包含"真接入优先于审慎收口"原则：
安全护栏是运行时开关，不是代码层不接入的借口。

## 9. Codex CLI 权限策略
（默认沙箱、危险操作清单、回滚禁忌）

## 10. 完成规则
（什么时候算项目完成，怎么生成 FINAL_REPORT）
```

### 3.2 TASKS.md 任务模板

每个 task 必须遵循：

```markdown
- [ ] **Task N：<动词> <对象>**
  - 目标：<一句话说清楚最终用户能多干什么，不写实现手段>
  - 实现：<新建/修改哪些文件、关键算法点>
  - DoD：(1) <契约层验收> (2) <端到端演示步骤> (3) <可用性推进证据>
  - 演示证据：贴 curl 输出/截图/psql 验证到 logs/dod-task<N>.md
```

**任务标题禁忌**：
- ❌ 不写"review-only"、"Mock"、"骨架"等实现手段词
- ❌ 不写"不调用后端"、"不写数据库"等否定式约束
- ✅ 只写做什么 + 完成长什么样

**完成括号注释禁忌**：
- ❌ 不允许 agent 自己回填"已完成"括号——这会形成自我繁殖
- ✅ "已完成"信息只写在 PROGRESS.md，TASKS.md 只用 `[x] / [ ] / [~]` 三态

### 3.3 VALIDATION.md 双轨验收模板

```markdown
## <模块> 阶段验证标准

### (A) 契约形态（必要但不充分）
- 路由注册了
- Schema 校验通过
- 单测全绿
- 反例（缺字段）返回正确错误

### (B) 真接入证据（充分条件，缺一不算完成）
- 真 LLM 调用：<怎么证明，例如同请求两次输出不一字不差>
- 真 DB 查询：<怎么证明，例如改 DB 一行接口跟着变>
- 真前端绑后端：<怎么证明，例如 Network 面板能看到 fetch>
- 真外部服务连通：<怎么证明>

### 反模式速查
（列出过去踩过的坑关键字，出现就拒收）
```

### 3.4 PROGRESS.md 流水账模板

每条记录至少包含：

```markdown
| 时间 | 任务 | 落点 | 结果 | 验证证据 |
|---|---|---|---|---|
| YYYY-MM-DD HH:MM:SS +0800 | Task N | 改了什么 | 已完成/降级/失败 | curl 输出路径 / 截图路径 / psql 命令 |
```

**反模式**：连续 50+ 次相同的"全部任务完成核对"——这是 cron 在空转盖章，必须从机制上禁止（见 §3.1 第 5 节强制 checkpoint）。

### 3.5 TASK_HANDOFF.md 交接模板

```markdown
## 当前卡点（如果有）
- 卡在哪个 task
- 卡的具体表现
- 已尝试的方案
- 下一步建议

## 最近中断记录
| 时间 | 类型 | 原因 | 是否已恢复 |
|---|---|---|---|

## 给下一个 agent 的话
（直接读到这一段就能接手干活）
```

### 3.6 FINAL_REPORT.md 模板

```markdown
# FINAL_REPORT

## 阶段性结论
（不要写 ALL TASKS COMPLETED，写"Phase X 契约骨架完成"或"Phase X 真接入交付")

## 实际产品状态自检
| 维度 | 名义状态 | 实际状态 | 距 "用户可用" 还差 |
|---|---|---|---|

## 完成任务列表
（注明每个任务是契约层完成还是真接入完成）

## 未解决风险

## 建议下一步
```

⚠️ **绝对禁止**：在 cron 自治模式下让 agent 自己生成 `ALL TASKS COMPLETED` 并自我盖章。这个标记必须人工在 review 完所有真接入证据后手动追加。

## 4. 自治模式安全规则汇总

适用于 cron / gateway / 任何无人值守的循环执行：

| 规则 | 原因 |
|---|---|
| 每 10 个 task 强制人工 checkpoint | 切断错误在闭环里自我放大 |
| Agent 不能自己回填 TASKS.md 的"已完成"括号 | 防止"成功模板"被反向继承 |
| Agent 不能自己写 `ALL TASKS COMPLETED` | 这个标记必须人工 review 后追加 |
| Agent 不能自己删除 cron 任务 | 删除必须人工确认 |
| DoD 不达标只能标 `[~]`，不能标 `[x]` | 区分"骨架完成"和"真完成" |
| VALIDATION 必须双轨（契约 + 真接入） | 防止"骨架=通过"的反馈锁 |
| 任务标题禁用实现手段关键字 | 防止任务定义本身把 Mock 写死 |
| 安全护栏只描述运行时开关，不描述代码不接入 | 防止护栏被过度解读 |

## 5. 一份合格的自治开发 md 的开头自检清单

写完第一版 AGENTS.md / TASKS.md / VALIDATION.md 后，必须能对下面 7 个问题给出明确"是"，否则就是有坑：

1. **AGENTS.md 里能否找到一段以"完成的标准是"开头的话？**（没有 DoD，就是只写禁止清单，必踩空壳坑）
2. **每个任务标题去掉所有括号注释后，还能不能看懂"做什么"？**（如果不能，说明实现手段被 baked 进了标题）
3. **VALIDATION.md 里能否找到"真 LLM 调用 / 真 DB 查询 / 真前端绑后端"这类断言？**（没有就是骨架=通过的锁）
4. **cron 模式下有没有强制 N 个 task 后人工 checkpoint？**（没有就会自我繁殖）
5. **安全护栏有没有明确写"这是运行时开关，不是代码不接入借口"？**（没写过宽，必被过度解读）
6. **PROGRESS.md 里如果出现连续 5+ 次相同记录，会不会触发警报？**（不触发就是 cron 在空转盖章）
7. **FINAL_REPORT 模板里有没有"实际产品状态自检"表格？**（没有就只剩自我宣告）

7 个全是"是"，可以放心让 cron 跑。任何一个"否"，先回去改 md，再启动 agent。
