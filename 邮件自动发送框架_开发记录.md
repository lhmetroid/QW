# 邮件自动发送框架 开发记录

创建日期: 2026-07-06

## 本轮已完成(纯前端框架, 不影响现有流程)

位置: `frontend/index.html` #mail-quality 页顶部导航。

1. 顶部导航按钮 "自动发送" 改名为 "套装排期"(`mail-subtab-autosend`, 功能/面板不变, 仍是原自动排期工作台 iframe)。
2. 旁边新增按钮 "自动发送"(`mail-subtab-autosendrules`), 切换到新面板 `mail-autosendrules-panel`。
3. 新面板内搭好两种模式的整体框架, 所有内容仅在本面板内, 未改动其他页面:
   - 模式① 按统一规则批量排期(对应需求 15)
     - 选择范围: A/B/C 类联系人多选 + 套装范围多选(占位)
     - LLM 分析建议: 生成按钮 + 结果表(联系人 x 分类 x 适合套装按优先级 x 排除原因 x 采用)
     - 排期规则: 1)邮箱选择(仅个人/仅公共/个人优先当日不足用公共) 2)发送顺序(靠前联系人发完所有套装 / 轮询逐套装) 3)间隔(套装内每封间隔 + 套装之间间隔)
     - 预排预览按钮(占位)
   - 模式② 销售动作触发提示(对应需求 14)
     - 触发规则配置表: 跟进/丢单 启用开关 + 触发套装映射(占位, 预留"更多动作")
     - 触发计算与建议: 计算按钮 + 结果表(联系人 x 触发动作 x 建议套装 x 排除原因 x 确认发送)
     - 确认并后台生成按钮(占位)
4. 接线: `switchMailWorkspaceTab` 增加 `autosendrules`; 新增 `autosendRulesInit / asrSwitchMode` 及各占位动作函数(`asrGenerateSuggestion / asrPreviewSchedule / asrTriggerCompute / asrTriggerConfirm`), 当前点击仅提示"框架占位, 待接后端"。

## 待接后端(占位动作, 后续逐项实现)

- 需求15: 联系人 A/B/C 分类定义与筛选接口; 套装范围列表加载; LLM 分析给出联系人 x 套装优先级(结合套装适用范围); 排除已发套装/已有排期; 按规则生成排期预览(入待发清单)。

## 开放问题(待用户确认)

1. A/B/C 类联系人的判定维度是什么? 是否复用现有联系人筛选条件? 按 ContactId 个人口径(见既有约定)还是公司口径?

4. 需求14 中"跟进/丢单"等销售动作数据从哪个来源取(CRM? 企微?), 动作 -> 套装映射由谁维护?

6. 排期是否直接写入现有"套装排期/待发清单/CRM 待发队列", 还是独立队列? 本轮未接, 需明确以免冲突。
7. 间隔单位默认按"天", 是否需要支持小时? 默认值(每封2天/套装间7天)是否合适?
