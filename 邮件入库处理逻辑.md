# 开发邮件找回·清洗·入黄金库 处理逻辑 (SOP)

> 版本 v1 · 2026-06-12 · 已逐项验证可行。本文件只记录"最终明确可行"的方案；中途证伪的做法不收录。
> 适用：把销售实发的开发/营销邮件，逐年找回正文 → 清洗脱敏 → 去重 → 归入三大场景 → 入库 → 人工评审(认可/不认可/需修改)。

---

## 0. 目标与三大场景

把所有"销售实发"的开发/营销邮件，识别归入以下 **三个场景之一**，入库后由人工判断是否认可：

| 场景 key | 场景名 | 含义 | 典型信号 |
|---|---|---|---|
| `re_activation` | 老客户激活 | 唤醒/问候已合作或久未联系的老客户 | 久未联系、好久不联系、再次联系、合作回顾、节日/新年问候、感谢一路相伴 |
| `new_business_promotion` | 老客户其他业务介绍 | 向老客户介绍新增/其他业务与成功案例 | 成功案例、新服务上线、还能为您做、业务拓展、案例分享 |
| `new_contact_intro` | 新客户业务介绍 | 向新客户/新接触人做公司与服务介绍 | 我是…来自事必达、初次/第一次联系、公司简介、诚挚问候+服务简介 |

**不属于这三类的一律拒绝入库**（见 §3 垃圾过滤），如：报价单、发票、交稿/发货、合同记录、纯回复线程、退信、操作类通知。

---

## 1. 数据来源与取数口径

- **正文权威源 = CRM 发件表 `spSendInfo{销售编号}`**（按销售 staff_id 分表，约 890 张非空分表）。**不要用跟进表 `usrCustomerFollowUpRecord`**（其正文字段为空，只有标签/活动日志）。
- **开发邮件只取"发送"，不取"收件"**：`WHERE MessageType='Email'`，按 `PlanSendTime` 取。
- **逐年递推**：以 `PlanSendTime` 按年（或自定义时间段）分批处理，便于增量与回溯。
- 关键字段：`SendId, PlanSendTime, Subject, EmailContent, EmlFtpDir`。
- 涉及哪些销售分表，可由跟进表 `RemarkClass LIKE '邮件宣传-%'` 的 `FollowUpStaff` 定位（也可全量遍历非空分表）。

## 2. 正文获取

1. `EmailContent`（ntext）**有值 → 直接用**（`Content` 列恒空，忽略）。
2. `EmailContent` 为空 → 按 `EmlFtpDir` 走 FTP 下载 `.eml` 解析：
   - `EmlFtpDir` = `年\月\日\uuid`，取前 3 段为日期。
   - FTP 凭据按日期查 `usrFtpTree`（`FtpType='文件'` + `StartDate/EndDate` 区间）得 host/user，再查 `usrFtpBackUp`（按 UserName）得 port/password；查不到回退全局 FTPHost。**不同时间段对应不同 FTP 服务器**。
   - 解析 `.eml` **只取当前这封正文**，剥离 `Re:` 之前的引用历史（切断标记：`发件人:`/`From:`/`-----Original Message-----`/`On … wrote:`/`在…写道:`/连续 `>` 引用块）。

## 3. 垃圾过滤（不入库）

正文命中任一即判为非营销，丢弃：
- 活动/文件日志：`Admin:`、`跟进记录`、`File[PHONE]`、`□File`、`.txt/.docx/.xlsx`、`||[时间]…`、`用途(` 开头、`couldn't be delivered`(退信)、邮件头 `x-sender`/`Message-ID`。
- 非营销主题：`口译合同编号`、`报价单`、`发票`、`订单`、`对账`、纯 `Re:/Fw:/回复/转发` 线程、`11`/`邮件` 等无意义短主题。
- 过短：清洗后正文 `< 120` 字。

## 4. 占位符规范化（去尖括号）

CRM 合并字段是尖括号占位符（如 `<ContactName>`），未转义会被前端当 HTML 标签吞掉、也是检索脏数据。统一替换为无尖括号标记：

| 原占位符(大小写/含空格忽略) | 替换 |
|---|---|
| ContactName / ContactEnglishName | `[客户称呼]` |
| StaffName / SenderName / SenderEnglishName | `[发件人]` |
| SenderMobile | `[手机]` ｜ SenderEmailAddress → `[邮箱]` |
| SenderExtendNo→`[分机]` ｜ SenderDivision→`[部门]` ｜ SenderCode→`[工号]` | |
| ContractId→`[合同号]` ｜ InputTime/PlanSendTime→`[日期]` | |
| `<[EMAIL]>` / `<mailto:[EMAIL]>` | `[EMAIL]` |
| 空 `<>`、16+位十六进制(Message-ID) | 删除 |
| 其他 CamelCase 合并字段 `<Xxx>` | `[字段]` |
| 装饰性 `< 文本 >` | 去尖括号保留文本 |

最后把残留的单个裸 `<`/`>` 中和（`<`→`‹`、`>`→`›`），保证产物 0 尖括号。

## 5. 脱敏

复用 `backend/export_mail_gold_candidates.py` 的正则，对正文+主题处理：
- `EMAIL_RE`→`[EMAIL]`、`URL_RE`→`[URL]`、`PHONE_RE`→`[PHONE]`、`MONEY_RE`→`[AMOUNT]`。
- **不要用 `LONG_ID_RE`**（会误伤英文长词）。
- **保留换行**（不要做整体空白折叠）。
- 人名/泄露的具体客户或公司名（如收件人称呼、签名实名、对方公司名）按需替换为 `[客户称呼]`/`[发件人]`/`贵司`。

## 6. 格式清洗

- **乱码**：删 `�`、`??`/`？？`、`?C`、`\xa0`→空格；明显错符如 `10000O`→`10000㎡`。
- **大段空行**：`\r\n`→`\n`；连续空行折叠为单空行；删纯空格行。
- **杂散符号行**：丢弃整行只有 `‹ › s • ·` 的行。
- **错误断行（一行被拆成多行）**：仅当"碎片化"时才激进重连——非空行中"≤4 字符短行占比 > 25%" 或 "平均行长 < 12"；重连规则：上一行不以句末标点(`。！？!?；;`)结尾、且当前行不是列表项时合并（中文直接接、英文加空格）。**本就段落正常的只清乱码/空行，保留原换行，不要乱并。**
- 标点前空格、数字间空格（如 `gifts .`、`路 1065 号`）归一。

## 7. 去重（关键：按正文相似度，不是主题）

- 规范化正文用于比较：去 `[占位符]`、去称呼行(Dear/Hi/您好/亲爱的…)、去全部空白，取前 800 字。
- 用 `difflib.SequenceMatcher.ratio()`，**相似度 ≥ 0.90 视为同一封**（只差称呼/主题也算同一封）；并查集分组，每组保留正文最长/最完整 1 条。
- 同时与库内"已人工认可"的种子比对：与认可项重复的，删找回项、**认可项不动**。

## 8. 场景分类（归入三大场景或拒绝）

1. 有 `用途`(跟进表 `RemarkClass`) 时优先映射：
   - `邮件宣传-我们公司介绍` → `new_contact_intro`
   - `邮件宣传-节日问候`/`邮件宣传-双方合作历史` → `re_activation`
   - `邮件宣传-成功案例-*`/`本公司成功案例` → `new_business_promotion`
2. 无用途或需复核时，用正文/主题信号兜底（见 §0 信号词）。
3. 三类都不命中 → 不入库。

## 9. 入库（双写 + 待人工评审）

- **同时写入**：
  - `email_fragment_asset`（黄金库）：`source_type='mail_gold_seed'`、`function_fragment='full_email'`、`scenario_label/intent_label`=场景、`library_type='gold_seed'`、`useful_score=0.80`、`desensitized_status='desensitized'`、`source_snapshot` 记来源(`crm:spSendInfo`)/场景/批次标记。
  - `mail_raw_unified`（邮件原始数据）：`source_type='spsend_promo'`、`direction='outbound'`、`mail_uid` 用主题哈希去重。
- **编号**：`source_ref` 续接现有最大 `mail_gold_v2_NNN`。
- **状态**：入库即"未评审"；`allowed_for_generation/usable_for_reply/publishable=false`（脱敏完成但暂不放开 RAG 召回，待人工认可后再开）。
- **幂等**：按 `source_snapshot.email_stage` 批次标记 / `mail_uid` 防重复。

## 10. 人工评审与修改回路

人工在黄金库页面对每条打：**认可 / 不认可 / 需修改**（写入 `mail_gold_seed_review`，状态列显示，悬停看点评）。
- **需修改**：按 `review_comment` **仅改正文**（`content`），**状态及其他字段不变**。常见点评与处理：
  - "去掉时间信息"：删新年/马年/季节问候与 2025/2026 年份框架，保留营销主体。
  - "未脱敏"：补脱敏人名/客户公司名。
  - "格式乱/未分行"：按 §6 重排、服务点分行成项目列表。
  - "有乱字符"：按 §6 清乱码、改项目符号。
  - "只取当前段"：按 §2.2 剥离 Re 引用历史。
- 复核通过后再由人工/批量改为"认可"，并可放开 `allowed_for_generation` 进入 RAG。

## 11. 人工提出的问题 → 最终方案（速查）

| 人工问题 | 最终方案 |
|---|---|
| 正文找不到/跟进表是空的 | 正文在 `spSendInfo{staff}.EmailContent`；空则 FTP 拉 .eml（§1/§2） |
| 开发邮件该找哪条 | 只取发送(MessageType=Email)，不取收件；逐年 |
| FTP 不同时间段在哪 | 按 EmlFtpDir 日期查 `usrFtpTree/usrFtpBackUp`（§2.2） |
| .eml 含 Re 历史 | 只取当前段，剥离引用历史（§2.2） |
| 尖括号占位符导致显示空白/检索脏 | 占位符规范化为 `[…]`、中和裸尖括号（§4） |
| 去重不对（主题去重留了重复） | 改按正文相似度 ≥90% 去重（§7） |
| 大段空行 / 乱码 / 一行错误断成多行 | 格式清洗规则（§6） |
| 混入非营销（合同号/报价/Re 线程/退信） | 垃圾过滤丢弃（§3） |
| 需修改的怎么处理 | 按点评仅改正文，状态不变（§10） |
| 已认可的不要动 | 去重/清洗一律保护已认可项 |

## 12. 下一步（按本逻辑跑全量）

1. 逐年遍历 `spSendInfo` 全部非空分表，取 MessageType=Email 的发送邮件。
2. 依次执行 §2→§3→§4→§5→§6→§7→§8。
3. §9 双写入库（未评审）。
4. 交人工按 §10 评审；需修改回到 §10 处理；认可后放开召回。
