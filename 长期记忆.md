## 一、feedback 类（工作方式偏好）

### 1. 回复语言：中文 · `feedback_language.md`
所有文字回复必须用中文（说明/分析/总结/状态更新），代码本身（变量名、注释风格）不受限。

### 2. 每轮修改后更新进展 + 推送 GitHub · `feedback_each_round_update.md`
每轮代码改完必须：(1) 在 `项目进展.md` 末尾按顶部 SOP 追加版本节点；(2) `git add` **仅本轮改动文件**并 commit（格式 `chore: release v1.x.x - 一句话总结`）后 `git push`。工作区常有大量无关改动，勿裹挟。

### 3. 严禁中文弯引号 · `feedback_no_smart_quotes.md`
代码（HTML/JS/Python）一律用 ASCII 直引号 `"` `'`，禁用中文弯引号 `“” ‘’`。曾因弯引号使 HTML class/style 属性整段失效、排查多轮。

### 4. 回答先给核心结论 · `feedback_answer_core_first.md`
结构固定"先结论后展开"：开头一句话给明确是/否或核心判断（可加粗），再补原因/例外/依据（file:line）。不要先铺垫。

### 5.（新增）不要假设，要实证查证 · `feedback_no_assumptions_verify.md`
下结论前先跑查询/读代码/看日志，用真实字段值、真实记录、file:line 佐证，并举实际例子。拿不准就明说"需查证"并去查，不要用"应该/理论上/大概"糊弄。
来源：多轮反复出现"不要假设/不要拆不要假设/请查证告知/以实际查证举例"。

### 6.（新增）禁止不切实际的时效用语 · `feedback_no_unrealistic_timing.md`
生成给客户的话术/脚本/邮件/企微文案，禁止出现"几分钟/马上/立刻/立即"等现实做不到的时效承诺，改用"尽快/会为您跟进"等留余地表达；浮动提示文字要口语化、全中文、易懂。
来源：用户明确"上次就备注过，不要说几分钟类似的字眼，立刻马上的现实中做不到"。

## 二、project 类（本项目业务/技术事实）

### 7. 训练AI 第二回复途径 · `project_train_ai_pathway.md`
2026-05-27 接入的第二条 AI 回复途径，走平台 model-chat 接口（llm.cycleforce.cc），与主流程**并行**触发、10s 超时（`TRAIN_AI_TIMEOUT_SECONDS`）。
- "当前流程 / ai"：原 LLM-2(deepseek-chat)，列显示 `ai：`
- "训练ai"：model-chat（默认模型 unsloth-qwen2.5-task-45），列显示 `训练ai：`
- 配置在 `TRAIN_AI_*`（config.py 默认 + 服务器 .env，.env 不入库）。字段 `training_ai{reply/status/latency_ms/model/error}`。训练AI 独立打分暂显 `-`。

### 8. 测后端 DB 别 import main · `project_db_test_no_import_main.md`
写一次性 DB 验证脚本**不要 `import main`**：会跑 `auto_patch_db` + 启动 mail_scheduler 等非守护线程，进程不退且占满远端 PG 连接池，导致后续 `engine.connect()` 挂起。做法：只 `from database import SessionLocal, engine, <模型>`，用完 `db.close()`。卡死时 `taskkill //F //IM python.exe`（后端实跑在远端 api.speedasia.net，本机 python 基本都是测试进程）。

### 9.（新增）生产库默认只读，不改结构/字段 · `project_db_readonly_no_schema_change.md`
未经明确授权，对(远端 Postgres)生产数据**默认只读**：不修改数据、不改表结构、不加字段。差选/统计/找回脚本一律 SELECT；要写入先说明并获确认。新生成产物（如 .eml）要**对齐已有格式**（曾自造 .eml 导致发送程序无法加载、中文乱码）。已有明确人工结论的数据不擅改。
来源：多次"只读取数据不要修改/仅只读不要修改添加字段/往已有格式上靠拢"。
